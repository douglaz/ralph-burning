# Implementation Notes

## Decisions Made
- Introduced `FlowSemantics` plus `stage_plan_for_flow(...)` so the shared engine can drive `standard`, `docs_change`, and `ci_improvement` without stage-ID branching.
- Added generic engine entrypoints `execute_run` / `resume_run` and kept the legacy standard-only names as thin wrappers to preserve existing tests and callers.
- Kept `quick_dev` blocked at the CLI while still defining flow semantics for it, so the engine surface is ready for a later loop without exposing unfinished behavior now.
- Extended both engine-level and CLI-level tests for docs/CI flows, and corrected the conformance feature text that previously claimed only `standard` was executable.

## Spec Deviations
- Non-late `ConditionallyApproved` outcomes still do not persist follow-ups in `run.json`. The current canonical model only has the durable amendment queue, and the spec also states docs/CI flows should not queue durable amendments, so I preserved the existing non-blocking behavior instead of inventing a new snapshot field in this loop.

## Testing
- Verified with `cargo test` using explicit Nix toolchain paths plus `CARGO_TARGET_X86_64_UNKNOWN_LINUX_GNU_LINKER` because the default shell PATH did not expose `cargo` or `cc`.
