# Implementation Response (Iteration 1)

## Changes Made
1. **Stale CLI lease reconciliation** — Added a second pass to `LeaseService::reconcile()` (`lease_service.rs`) that scans `list_lease_records()` for `LeaseRecord::CliWriter` variants, checks staleness (using `ttl_override_seconds` or natural TTL), and cleans stale CLI leases by removing the lease record and releasing the writer lock. Strict cleanup accounting: `AlreadyAbsent` on either sub-step produces a `LeaseCleanupFailure` with `task_id: None`, and only positive removal of both counts as `released_leases`. No task is marked failed, no worktree is removed, and `failed_tasks` is never incremented for CLI leases. Added four unit tests (`reconcile_stale_cli_lease_cleans_lease_and_writer_lock`, `reconcile_stale_cli_lease_missing_writer_lock_reports_cleanup_failure`, `reconcile_stale_cli_lease_missing_lease_file_reports_cleanup_failure`, `reconcile_non_stale_cli_lease_is_not_cleaned`) and two CLI integration tests (`cli_daemon_reconcile_cleans_stale_cli_lease` — end-to-end: inject stale CLI lease + lock → reconcile cleans them → `run start` succeeds; `cli_daemon_reconcile_reports_failure_for_stale_cli_lease_missing_lock` — partial cleanup reports `writer_lock_absent`).

2. **Heartbeat shutdown ordering** — Redesigned `CliWriterLeaseGuard` to use a `closed: Arc<AtomicBool>` flag and a `tick_lock: Arc<std::sync::Mutex<()>>` so that no heartbeat tick can race with drop cleanup. The heartbeat task checks `closed` before and after acquiring `tick_lock`; drop sets `closed`, acquires `tick_lock` (waiting for any in-flight tick to finish), then aborts the handle before performing lease/lock cleanup. This guarantees deterministic shutdown: once drop begins, no heartbeat file I/O can occur.
