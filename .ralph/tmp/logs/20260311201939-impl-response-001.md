# Implementation Response (Iteration 1)

## Changes Made
1. **Canonical run snapshot schema** ‚Äî Expanded `RunSnapshot` in `model.rs` with `cycle_history: Vec<CycleHistoryEntry>`, `completion_rounds: u32`, `rollback_point_meta: RollbackPointMeta`, and `amendment_queue: AmendmentQueueState`. Added corresponding `CycleHistoryEntry`, `RollbackPointMeta`, and `AmendmentQueueState` types with `#[derive(Default)]`. Updated `RunSnapshot::initial()` to populate all fields with empty/zero defaults. The serialized `run.json` now contains every canonical field the spec requires.

2. **Durable history completeness** ‚Äî Updated `validate_history_consistency` in `queries.rs` to check both directions: orphaned artifacts (artifact without payload) AND orphaned payloads (payload without artifact). Updated `handle_tail` in `cli/run.rs` to render payload and artifact records in the durable section, matching what `run history` displays. `run tail` now shows the same durable history as `run history`.

3. **Fail-fast canonical reads** ‚Äî Changed `FsJournalStore::read_journal` and `FsRunSnapshotStore::read_run_snapshot` in `fs.rs` to return `CorruptRecord` errors when the canonical file is missing (instead of silently returning empty state). Missing `journal.ndjson` now produces `"canonical file is missing"` instead of an empty event list. Missing `run.json` now produces the same error instead of a synthesized `RunSnapshot::initial()`.

4. **Delete failure semantics** ‚Äî Replaced `fs::remove_dir_all` in `FsProjectStore::delete_project` with a two-phase transactional approach: (1) atomic rename to a trash path (`.{id}.deleting.{pid}`), (2) remove the trash directory. If removal fails, the project is renamed back to its original location so it remains addressable. The active-project pointer is only cleared in `service::delete_project` after the store delete succeeds (unchanged).
