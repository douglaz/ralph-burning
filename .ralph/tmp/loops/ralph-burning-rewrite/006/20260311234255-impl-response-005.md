# Implementation Response (Iteration 5)

## Changes Made
1. **Atomic stage-commit via staging + rename** (`fs.rs`): Rewrote `write_payload_artifact_pair` to use a staging approach. Both payload and artifact are first written to `runtime/temp/` as staging files, then renamed into their canonical `history/payloads/` and `history/artifacts/` locations. Files only appear in canonical history locations when both staging writes succeed. If any step fails, cleanup targets staging files (which aren't in canonical locations), eliminating the orphaned-payload leak path entirely.

2. **Engine defense-in-depth cleanup** (`engine.rs`): After `write_payload_artifact_pair` fails, the engine now explicitly calls `remove_payload_artifact_pair` before invoking `fail_run`. This provides a belt-and-suspenders cleanup guarantee: even if a non-standard `PayloadArtifactWritePort` implementation leaks canonical files, the engine removes them before marking the run as failed.

3. **`LeakingPayloadArtifactWriteStore` test seam** (`workflow_engine_test.rs`): Added a new test port that deliberately writes a payload to the canonical location before returning an error (simulating "payload written, artifact write failed, payload cleanup failed"). The `leaked_payload_cleanup_on_write_failure` test verifies the engine's defense-in-depth cleanup removes the orphaned canonical payload, leaving zero payload/artifact files and no `stage_completed` events.
