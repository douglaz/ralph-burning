# Implementation Response (Iteration 6)

## Changes Made
1. **`project delete` transactional semantics** — Reordered `delete_project` in `service.rs:221-237` to perform the filesystem delete *before* touching the active-project pointer. `clear_active_project` is now called only after `store.delete_project()` succeeds. If the delete fails, the pointer is never touched — no clear, no restore needed. If the delete succeeds but `clear_active_project` fails, the error propagates to the caller so the stale pointer is visible.

2. **Regression tests for pointer failure paths** — Replaced the old `delete_project_restores_active_pointer_on_delete_failure` test with two new tests in `project_run_record_test.rs`:
   - `delete_project_does_not_touch_pointer_on_delete_failure`: verifies that neither `clear_active_project` nor `write_active_project` is called when the delete itself fails, using a `TrackingActiveStore` that records both operations.
   - `delete_project_propagates_clear_pointer_failure_after_successful_delete`: verifies that when delete succeeds but `clear_active_project` returns an I/O error, the error propagates to the caller.

3. **Conformance scenarios** — Updated SC-PROJ-022 to reflect the new semantics (pointer is never touched on delete failure, no restore path). Added SC-PROJ-023 covering the case where delete succeeds but pointer clear fails.
