# Implementation Response (Iteration 3)

## Changes Made
1. **Active-project canonical validation in `validate_project_exists`** — Updated `workspace_governance::validate_project_exists` to read and parse `project.toml` content as TOML, not just check file existence. If the file is missing, a `CorruptRecord` error is returned (instead of `ProjectNotFound`). If the file contains malformed TOML, a `CorruptRecord` error with parse details is returned. This catches corruption at the `resolve_active_project` level, affecting all commands that resolve the active project.

2. **Canonical project record validation in run query CLI handlers** — Added `FsProjectStore::read_project_record()` calls in `handle_status`, `handle_history`, and `handle_tail` in `run.rs`. After resolving the active project ID, the canonical `project.toml` is fully deserialized into `ProjectRecord`, catching both TOML syntax errors (already caught by governance layer) and structural issues (wrong fields/types). This provides defense-in-depth validation before proceeding with run queries.

3. **CLI test coverage for active-project corruption paths** — Added 5 new CLI integration tests:
   - `run_status_fails_fast_when_active_project_toml_is_corrupt`

   - `run_tail_fails_fast_when_active_project_toml_is_corrupt`
   - `project_show_no_id_fails_fast_when_active_project_toml_is_corrupt`
   - `run_status_fails_fast_when_active_project_toml_is_missing`

4. **Conformance scenario coverage** — Added 5 new Gherkin scenarios (SC-RUN-019 through SC-RUN-023) covering malformed `project.toml` on `run status`, `run history`, `run tail`, `project show` (no ID), and missing `project.toml` on `run status`.
