# Implementation Response (Iteration 9)

## Changes Made
1. **RC1 — Remove legacy single-repo CLI branches**: The `Status`, `Abort`, `Retry`, and `Reconcile` daemon subcommands now require `--data-dir` (changed from `Option<String>` to `String`). `Abort` and `Retry` also require `--repo` (changed from `Option<String>` to `String`). The four legacy handler functions (`handle_status_legacy`, `handle_abort_legacy`, `handle_retry_legacy`, `handle_reconcile_legacy`) have been deleted entirely. The `handle()` match arms now unconditionally call the multi-repo handlers without any fallback branching. All 12 affected CLI tests and 10+ conformance scenarios were converted to use the data-dir layout with `--data-dir` and `--repo` arguments, with new helper functions (`write_datadir_daemon_task`, `write_datadir_worktree_lease`, `write_repo_registration`) for setting up data-dir test fixtures. Abort/retry tests now use issue-number identifiers with `issue_number` and `repo_slug` set on task fixtures. The test-only `daemon start` path (gated behind `RALPH_BURNING_TEST_LEGACY_DAEMON=1`) remains unchanged.

2. **RC2 — Per-repo failure isolation for label sync**: `process_cycle_multi_repo` in `daemon_loop.rs` now quarantines a repo when any resumed-task label sync fails. Previously the label sync error was logged and the loop continued to phase 3 (processing pending tasks) for the same repo. Now, on the first `sync_label_for_task` failure, the loop breaks out of the label-sync phase and `continue`s to the next repo, consistent with the quarantine behavior already applied to GitHub poll failures (line 288) and task-processing failures (line 370).

3. **RI1 — URL-encode refs in `is_branch_ahead()`**: The `base` and `head` parameters in `GithubClient::is_branch_ahead()` are now percent-encoded before being interpolated into the `/repos/{owner}/{repo}/compare/{base}...{head}` URL path. This prevents malformed URLs when branch names contain slashes (e.g. `rb/42-project-id`).
