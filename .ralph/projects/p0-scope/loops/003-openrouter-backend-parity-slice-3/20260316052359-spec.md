# Feature: OpenRouter Backend Parity (Slice 3)

## Description
Implement a dedicated `OpenRouterBackendAdapter` that communicates with the OpenRouter Chat Completions API over HTTP, integrate it as a first-class `BackendAdapter` variant, and remove the guard that currently blocks OpenRouter from panel resolution and process-backend execution. After this slice, OpenRouter can serve any workflow role or panel position and is fully wired into the config, policy, builder, and agent-execution layers.

## Acceptance Criteria
- [ ] New file `src/adapters/openrouter_backend.rs` implements `AgentExecutionPort` for `OpenRouterBackendAdapter`
- [ ] `check_availability` verifies the `OPENROUTER_API_KEY` env var is set and performs a lightweight HTTP probe (e.g. `GET /api/v1/models`) returning `BackendUnavailable` on failure
- [ ] `check_capability` accepts all `InvocationContract` variants (Stage and Requirements) for OpenRouter, as both require only structured JSON output
- [ ] `invoke` sends a POST to `https://openrouter.ai/api/v1/chat/completions` with: the configured model injected into the request body, the prompt/context serialized as a system+user message pair, a `response_format: { type: "json_object" }` directive, and the contract JSON schema as a `json_schema` field in `response_format` when supported
- [ ] Model injection: the model field in the API request body is always set to `request.resolved_target.model.model_id` — never inferred or defaulted by OpenRouter
- [ ] Token counts are parsed from the API response `usage` object and mapped to `TokenCounts { prompt_tokens, completion_tokens, total_tokens }`
- [ ] Timeout support: the adapter respects `InvocationRequest.timeout` by applying it as the HTTP client timeout; if the request times out, a `FailureClass::Timeout` error is returned
- [ ] Cancellation support: the adapter checks `CancellationToken` before sending the request and uses `tokio::select!` to race the HTTP future against `cancellation_token.cancelled()`; on cancellation the in-flight HTTP request is dropped (aborting the connection)
- [ ] Session reuse is explicitly NOT supported: the adapter ignores `prior_session` and never emits a `session_id` in the envelope metadata; `BackendSpec::from_family(OpenRouter)` already sets `supports_session_reuse: false`
- [ ] HTTP error mapping: 401/403 → `BackendUnavailable`; 429 → `FailureClass::TransportFailure` with rate-limit details; 5xx → `FailureClass::TransportFailure`; non-JSON or malformed response body → `FailureClass::SchemaValidationFailure`
- [ ] `BackendAdapter` enum in `src/adapters/mod.rs` gains an `OpenRouter(OpenRouterBackendAdapter)` variant with delegated `AgentExecutionPort` implementation
- [ ] `build_backend_adapter()` in `src/composition/agent_execution_builder.rs` recognises `RALPH_BURNING_BACKEND=openrouter` and constructs the OpenRouter variant; the default `process` path continues to use `ProcessBackendAdapter` for claude/codex only
- [ ] The runtime adapter selection logic gains a new code path: when the resolved `BackendFamily` is `OpenRouter`, the builder/dispatcher routes to `OpenRouterBackendAdapter` rather than `ProcessBackendAdapter`. This can be achieved either by (a) making `BackendAdapter` resolution family-aware in the builder, or (b) adding a composite adapter that dispatches by family at invocation time
- [ ] `panel_backend_resolvable` in `src/contexts/agent_execution/policy.rs` removes the `!matches!(family, BackendFamily::OpenRouter)` guard so that OpenRouter backends can participate in prompt-review, completion, and final-review panels when enabled
- [ ] `ProcessBackendAdapter::check_capability` and `check_availability` continue to reject `BackendFamily::OpenRouter` — the process adapter is not the execution path for OpenRouter
- [ ] `reqwest` (with `rustls-tls` feature) is added to `Cargo.toml` dependencies
- [ ] Conformance scenario `backend.openrouter.model_injection` passes: given an OpenRouter-resolved target with model `"anthropic/claude-3.5-sonnet"`, the API request body contains exactly `"model": "anthropic/claude-3.5-sonnet"`
- [ ] Conformance scenario `backend.openrouter.disabled_default_backend` passes: when `backends.openrouter.enabled = false`, policy resolution that would select OpenRouter fails with `BackendUnavailable`
- [ ] Conformance scenario `backend.openrouter.requirements_draft` passes: a requirements draft invocation targeting OpenRouter returns a valid structured JSON payload
