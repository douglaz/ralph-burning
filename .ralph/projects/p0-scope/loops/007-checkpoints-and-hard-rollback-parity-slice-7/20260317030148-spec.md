# Feature: Checkpoints and Hard Rollback Parity (Slice 7)

## Description
Extract checkpoint creation from the inline `persist_rollback_point` helper in `engine.rs` into a dedicated `VcsCheckpointPort` trait and `checkpoints.rs` module. Replace the current behavior of merely recording the HEAD SHA with creating explicit git checkpoint commits that carry structured metadata in the commit message. Update the `WorktreeAdapter` to implement the new port. Add the two required conformance scenarios.

Currently, `persist_rollback_point` (called 13 times across `engine.rs`) captures `current_head_sha()` but never creates a checkpoint commit. The spec requires that each checkpoint produces a real git commit with a deterministic message format so that checkpoints can be discovered by commit message and so that hard rollback targets a known, named commit rather than an ambient HEAD at the moment of stage completion.

## Acceptance Criteria
- [ ] New file `src/contexts/workflow_composition/checkpoints.rs` exists and is registered in `mod.rs`
- [ ] `VcsCheckpointPort` trait defined with methods: `create_checkpoint(repo_root, project_id, run_id, stage_id, cycle, completion_round) -> AppResult<String>` (returns SHA), `find_checkpoint(repo_root, project_id, stage_id, cycle, completion_round) -> AppResult<Option<String>>`, `reset_to_checkpoint(repo_root, sha) -> AppResult<()>`
- [ ] `create_checkpoint` stages all working-tree changes (if any), creates an empty or all-inclusive commit with the exact message format specified in the prompt, and returns the resulting SHA
- [ ] Checkpoint commit message format matches exactly:
  ```
  rb: checkpoint project=<project-id> stage=<stage-id> cycle=<n> round=<m>

  RB-Project: <project-id>
  RB-Run: <run-id>
  RB-Stage: <stage-id>
  RB-Cycle: <n>
  RB-Completion-Round: <m>
  ```
- [ ] `find_checkpoint` searches git log for commits matching the `rb: checkpoint` subject pattern for the given project/stage/cycle/round and returns the SHA if found
- [ ] `reset_to_checkpoint` delegates to `git reset --hard <sha>` (same as existing `RepositoryResetPort`)
- [ ] `WorktreeAdapter` implements `VcsCheckpointPort`
- [ ] `persist_rollback_point` in `engine.rs` is updated to call `VcsCheckpointPort::create_checkpoint` instead of `WorktreeAdapter::current_head_sha`, storing the checkpoint commit SHA in `RollbackPoint.git_sha`
- [ ] Checkpoints are created after each successful primary stage that advances run state: prompt review completion, implementation/apply-fixes/docs-update/ci-update, review approval, completion aggregate, acceptance QA pass, final review exit (all existing call sites in `engine.rs`)
- [ ] The `persist_rollback_point` function accepts a `&dyn VcsCheckpointPort` parameter instead of internally constructing `WorktreeAdapter`
- [ ] If `create_checkpoint` fails (e.g., not a git repo), the rollback point is still persisted with `git_sha: None` and a runtime warning is logged (matching existing error-tolerance behavior)
- [ ] `perform_rollback` with `hard: true` continues to use the recorded SHA for `git reset --hard` — the SHA now points to the checkpoint commit
- [ ] Conformance scenario `workflow.rollback.hard_uses_checkpoint` passes: verifies that hard rollback targets the SHA of a checkpoint commit (not an arbitrary HEAD)
- [ ] Conformance scenario `workflow.checkpoint.commit_metadata_stable` passes: verifies the checkpoint commit message contains all required trailer fields and the subject line matches the format
- [ ] All existing `run_rollback.feature` scenarios (SC-ROLLBACK-001 through SC-ROLLBACK-008) continue to pass
- [ ] `cargo test` broad regression suite passes
