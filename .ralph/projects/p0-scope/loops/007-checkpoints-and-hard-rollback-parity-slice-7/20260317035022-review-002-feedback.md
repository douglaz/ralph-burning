## Required Changes
1. **Checkpoint commits must exclude `.ralph-burning/**` from the commit tree, not just from newly staged changes**: [`create_checkpoint`](/root/new-ralph-burning/ralph-burning-rewrite/src/adapters/worktree.rs#L155) currently calls [`stage_checkpoint_changes`](/root/new-ralph-burning/ralph-burning-rewrite/src/adapters/worktree.rs#L19), which avoids staging new `.ralph-burning` changes but still creates a normal child commit of the current `HEAD`.
   - Current: if the parent commit already tracks `.ralph-burning`, the checkpoint commit still inherits those paths. I reproduced this with a plain git repo: after creating a checkpoint, `git ls-tree -r --name-only HEAD` still contained `.ralph-burning/projects/demo/run.json`, and `git show HEAD:.ralph-burning/projects/demo/run.json` returned the parent’s old runtime-state content. Because [`perform_rollback`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs#L575) writes the logical rollback snapshot before the hard reset at [`service.rs`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs#L607), `git reset --hard <checkpoint-sha>` can still overwrite `run.json` in that repo shape.
   - Expected: checkpoint creation must produce a commit whose tree has no `.ralph-burning/**` entries at all, even when those paths were already tracked in the parent commit. Otherwise hard rollback still violates the required “logical rollback first, then hard reset” behavior.
   - Reference: Slice 7 acceptance around hard rollback parity and checkpoint semantics, plus the master prompt’s rollback rule: logical rollback must remain durable before any hard reset.
