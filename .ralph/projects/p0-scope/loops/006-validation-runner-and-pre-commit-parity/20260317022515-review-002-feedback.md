# Review: SUGGESTIONS

## Required Changes
1. **Pre-commit remediation durability**: Persist pre-commit remediation feedback in a form that `run resume` can reconstruct.
   - Current: The review approval is durably committed before pre-commit runs at [engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L2977). When pre-commit fails, the feedback is only assigned to the in-memory `execution_context` at [engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L3294). Resume reconstruction only accepts a prior primary `ValidationPayload` whose outcome is `request_changes` at [engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L5067) and [engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L5136). After a pre-commit-triggered remediation, the prior primary payload is the review `approved` payload, so a later interruption/failure in that remediation cycle cannot resume with the required feedback.
   - Expected: Persist pre-commit remediation feedback durably, or teach resume reconstruction to derive it from the supporting pre-commit evidence, so implementation remediation remains resumable after pre-commit failure.
   - Reference: Slice 6 acceptance criteria for persisting supporting validation evidence and remediation feedback, and for routing pre-commit failure back to implementation remediation.

2. **Timed-out command evidence capture**: Preserve stdout/stderr that a command emitted before timing out.
   - Current: [validation_runner.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/adapters/validation_runner.rs#L164) wraps pipe reads inside the timeout, and the timeout branch returns `stdout: ""` with only a synthetic timeout message at [validation_runner.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/adapters/validation_runner.rs#L217). Any output produced before the timeout is discarded.
   - Expected: `ValidationCommandResult` should retain captured stdout/stderr for timed-out commands too, because the slice requires structured command results and durable evidence for the failing command.
   - Reference: Slice 6 acceptance criteria for capturing `stdout`, `stderr`, `exit_code`, `duration_ms`, and `passed`, and for preserving the failing command in order.

## Recommended Improvements
1. Add a regression that forces a pre-commit failure, advances into implementation remediation, then resumes from an interrupted run; the current suite covers the happy-path review context but not this durability path.
2. Exercise the real `pre_commit_fmt_auto_fix` and timeout-with-output paths in tests instead of only constructing synthetic `ValidationGroupResult` values.
