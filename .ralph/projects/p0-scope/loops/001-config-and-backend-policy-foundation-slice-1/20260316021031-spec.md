# Feature: Config and Backend-Policy Foundation (Slice 1)

## Description
Implement the expanded configuration model, project-level config, effective policy types, CLI backend overrides, backend policy service with role resolution and alternation logic, role-specific timeout resolution, and InvocationContract schema API. This is the foundational slice that all subsequent P0 slices depend on — it establishes the configuration surface, merge precedence, and backend policy resolution that workflow panels, completion, final review, and daemon features will consume.

## Acceptance Criteria
- [ ] `ProjectConfig` struct supports full TOML shape: `[settings]`, `[prompt_review]`, `[workflow]`, `[completion]`, `[final_review]`, `[validation]` sections with all fields from the spec
- [ ] `WorkflowSettings`, `CompletionSettings`, `FinalReviewSettings`, `ValidationSettings`, `BackendRuntimeSettings`, `BackendRoleModels`, `BackendRoleTimeouts` structs exist in `src/shared/domain.rs` and are serializable/deserializable
- [ ] `EffectiveRunPolicy`, `EffectivePromptReviewPolicy`, `EffectiveCompletionPolicy`, `EffectiveFinalReviewPolicy`, `EffectiveValidationPolicy`, `EffectiveBackendPolicy` exist and are computed from config merge
- [ ] Config merge follows exact precedence: CLI overrides > Project config > Workspace config > Code defaults
- [ ] Project config TOML is read from `.ralph-burning/projects/<project-id>/config.toml`
- [ ] Panel backend specs support `backend_name` (required) and `?backend_name` (optional) syntax with correct filtering: required-unavailable fails resolution, optional-unavailable silently skips, post-filter count must satisfy `min_reviewers`/`min_completers`
- [ ] Backend policy service at `src/contexts/agent_execution/policy.rs` implements: `resolve_role_target`, `resolve_completion_panel`, `resolve_prompt_review_panel`, `resolve_final_review_panel`, `timeout_for_role`, `opposite_family`, `planner_family_for_cycle`
- [ ] Role resolution order: CLI role override > Project role override > Workspace role override > Cycle-based default policy
- [ ] Default family alternation: odd cycles use base backend for planner; even cycles use opposite family; implementer opposite of planner; reviewer same as planner; QA same as implementer; completion planner same as planner; default completer opposite of completion planner
- [ ] `opposite_family` implements exact fallback chain: claude→codex (fallback openrouter), codex→claude (fallback openrouter), openrouter→claude (fallback codex); no-opposite-available fails resolution
- [ ] Role timeouts resolve from `backends.<name>.role_timeouts.<role>`, falling back to backend default timeout, falling back to process-backend default
- [ ] CLI overrides `--backend`, `--planner-backend`, `--implementer-backend`, `--reviewer-backend`, `--qa-backend` are wired into `run start` and `run resume`
- [ ] `InvocationContract` exposes a schema API consumed by both workflow composition contracts and requirements drafting contracts for structured-output requirements
- [ ] Conformance test `backend.role_overrides.per_role_override_beats_default` passes
- [ ] Conformance test `backend.role_timeouts.config_roundtrip` passes
- [ ] Unit tests cover config roundtrip (serialize → deserialize), merge precedence, panel filtering, opposite-family resolution, cycle-based alternation, and timeout fallback chain
