# Feature: Real Requirements Backend Path (Slice 2)

## Description
Loop 1 completed the config and backend-policy foundation, and the current loop artifacts plus branch history show no later approved slice. The next missing feature is Slice 2: move `requirements draft` / `requirements quick` and daemon requirements dispatch onto the real production backend path. The current baseline still constructs `StubBackendAdapter` directly in `src/cli/requirements.rs` and `src/contexts/automation_runtime/daemon_loop.rs`, and `ProcessBackendAdapter` still rejects `InvocationContract::Requirements`, so this parity slice is not yet implemented.

## Acceptance Criteria
- [ ] Add a shared builder in `src/composition/agent_execution_builder.rs` that owns runtime adapter selection for `AgentExecutionService`, with `process` as the default and the existing `stub` branch preserved only as a test seam.
- [ ] The shared builder centralizes current environment-driven adapter wiring now split across `run` and requirements code, including validation of `RALPH_BURNING_BACKEND` and the existing test override env vars.
- [ ] `requirements draft` and `requirements quick` use the shared builder and no longer directly instantiate `StubBackendAdapter` in production CLI code.
- [ ] Daemon `requirements_draft` and `requirements_quick` use the same shared builder path as the CLI requirements commands, so both surfaces resolve the same backend family/model defaults from `EffectiveConfig`.
- [ ] `ProcessBackendAdapter::check_capability` accepts `InvocationContract::Requirements` for `claude` and `codex` targets, while continuing to reject unsupported backend families.
- [ ] `ProcessBackendAdapter::invoke` runs requirements contracts through the existing structured-output path: schema injection, raw-output capture, JSON parsing, timeout enforcement, cancellation, and session recording must work the same way they already do for workflow stage contracts.
- [ ] Existing negative tests that assert requirements contracts are rejected by `ProcessBackendAdapter` are replaced with positive coverage for successful requirements invocation; unsupported-family rejection coverage remains in place.
- [ ] Failure invariant: if availability, capability, spawn, timeout, cancellation, or JSON-contract failure happens before a requirements stage commits its payload/artifact pair, that stage must leave no committed payload/artifact files or seed files behind, and the run must end in durable `failed` state.
- [ ] Failure invariant: if daemon requirements service construction or first backend invocation fails, the task must transition to the existing `requirements_draft_failed` or `requirements_quick_failed` path and `requirements_run_id` must remain unset.
- [ ] Failure invariant: if a requirements run completes but daemon task-link persistence fails, the requirements run and any generated seed files remain durable and addressable via `requirements show`; only the task transitions to `requirements_linking_failed`.
- [ ] No non-test production path directly calls `StubBackendAdapter::default()` for requirements execution after this slice; any remaining stub construction is isolated to builder test seams or explicit test helpers.
- [ ] Add conformance coverage for `backend.requirements.real_backend_path` that runs the CLI and daemon requirements paths with `RALPH_BURNING_BACKEND=process` and temporary fake `claude`/`codex` binaries on `PATH`, proving the real backend path is exercised rather than the stub path.
- [ ] Verification for the slice includes focused unit coverage for process-backend requirements support, focused daemon requirements builder coverage, and reruns of the requirements drafting conformance scenarios to ensure durability semantics did not regress.
