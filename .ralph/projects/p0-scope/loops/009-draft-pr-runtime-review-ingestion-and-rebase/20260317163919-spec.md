# Feature: Draft PR Runtime, Review Ingestion, and Rebase Parity (Slice 9)

## Description
Implement the final P0 parity slice: automated draft PR lifecycle management, PR review comment ingestion converting external feedback into amendments, and enhanced rebase parity with backend-assisted conflict resolution. This completes the daemon's GitHub integration by connecting workflow completion to PR creation, external reviewer feedback to the amendment pipeline, and robust rebase handling with proper journaling.

The GitHub adapter (`src/adapters/github.rs`) already exposes all needed PR API operations (`create_draft_pr`, `mark_pr_ready`, `close_pr`, `fetch_pr_reviews`, `fetch_pr_review_comments`, `is_branch_ahead`, `update_pr_body`, `fetch_pr_state`). The worktree adapter already implements `rebase_onto_default_branch`. Task metadata already carries `pr_url`, `last_seen_comment_id`, and `last_seen_review_id`. This slice wires these primitives into cohesive runtime services and adds the missing ingestion, deduplication, and reopening logic.

## Acceptance Criteria
- [ ] `pr_runtime.rs` service creates a draft PR when a task branch first moves ahead of base, persisting the PR URL on task metadata
- [ ] No duplicate PRs are created for the same task (checked via `pr_url` presence on task metadata)
- [ ] When a task completes with no diff from base, PR creation is skipped or existing PR is closed per policy
- [ ] PR runtime supports clean cancellation via the existing cancellation token
- [ ] Push to remote occurs before draft PR creation (`push_before_create` invariant)
- [ ] `pr_review.rs` service ingests inline review comments, top-level PR comments, and review summary comments
- [ ] Review ingestion filters by a configurable whitelist of allowed GitHub usernames
- [ ] Deduplication uses composite keys: `pull_comment:<id>`, `issue_comment:<id>`, `review:<id>` tracked via `last_seen_comment_id` and `last_seen_review_id`
- [ ] Accepted review text is converted into `QueuedAmendment` entries via the existing amendment pipeline
- [ ] Staged amendments are persisted before dispatch and survive restart and transient API failures
- [ ] If new review amendments arrive for a completed project, the project is reopened into the correct active preset state and remediation work is dispatched
- [ ] Rebase supports backend-assisted conflict resolution when enabled in config
- [ ] If rebase fails terminally, the task moves to failed status while preserving the worktree
- [ ] Rebase state and outcome are journaled as `DaemonJournalEvent` entries
- [ ] Conformance tests pass for all 11 required scenarios: `daemon.pr_runtime.*` (4), `daemon.pr_review.*` (4), `daemon.rebase.*` (3)
