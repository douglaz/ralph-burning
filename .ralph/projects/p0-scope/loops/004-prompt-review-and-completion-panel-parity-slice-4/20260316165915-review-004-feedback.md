1. **Completion panel transition invariant**: the `completion_panel` stage still makes the aggregate result durable before the post-panel transition is durably recorded, which violates the Slice 4 failure invariant and can resume from the wrong stage.
Current: [`engine.rs`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L1160) and [`engine.rs`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L1320) call `commit_completion_aggregate(...)` before the next-stage snapshot / `completion_round_advanced` write; [`engine.rs`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L1352) also bumps `snapshot.completion_rounds` before the round-advance append succeeds. Resume reconstruction then advances from any persisted `stage_completed` event at [`engine.rs`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L3452), which for the standard flow means moving past `completion_panel` toward `acceptance_qa` per [`mod.rs`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/mod.rs#L37). The new regression test explicitly blesses that behavior at [`workflow_engine_test.rs`](/root/new-ralph-burning/ralph-burning-rewrite/tests/unit/workflow_engine_test.rs#L4138).
Expected: if aggregate persistence or any post-panel transition write fails, no `StageAggregate` record or `stage_completed` event should remain durable, and resume should restart from `completion_panel`, not from the next stage.
Reference: Slice 4 acceptance criteria for `workflow.completion.panel_continue_verdict` and “Completion failure invariant”.

Expected: only make `stage_completed` durable once prompt replacement and prompt-hash update have succeeded, or roll back the primary record / event when prompt replacement fails, so the run cannot advance past `prompt_review` without the refined prompt being in place.
