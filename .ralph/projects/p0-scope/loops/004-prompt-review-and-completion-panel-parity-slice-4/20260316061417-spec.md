# Feature: Prompt Review and Completion Panel Parity (Slice 4)

## Description
Loop artifacts, approved review notes, and git history show Slice 1, Slice 2, and Slice 3 are complete. The next missing P0 feature is Slice 4: the current baseline still runs `prompt_review` and `completion_panel` as single generic stage contracts in `engine.rs`, persists only one payload/artifact pair per stage, and has no stage-resolution snapshot or resume-drift warning path. This slice should add true prompt-review and completion panels while preserving baseline flow ordering, the existing `prompt_review.enabled` stage toggle, and the existing cycle/completion-round machinery that is already present.

## Acceptance Criteria
- [ ] Add structured panel contracts for prompt-review refinement, prompt-review validation, prompt-review decision, completion votes, and completion aggregation; these contracts must not reuse the current generic planning/validation payloads where that would lose panel-specific fields such as refined prompt text, vote counts, or aggregate verdict metadata.
- [ ] `prompt_review` remains an optional stage in the standard flow only; existing `stage_plan_for_flow(...)` behavior stays unchanged when prompt review is disabled.
- [ ] At stage start, `ActiveRun` persists a `stage_resolution_snapshot` before any agent invocation for single-target stages and for the new prompt-review and completion panels; if snapshot persistence fails, the stage aborts with no agent side effects.
- [ ] `StageResolutionSnapshot` records exact resolved backend/model targets, including ordered panel members for prompt-review validators and completion completers.
- [ ] Prompt review resolves its refiner and validator panel through `BackendPolicyService::resolve_prompt_review_panel(...)`, invokes the refiner first, and persists the refinement result as `StageSupporting`.
- [ ] Prompt review invokes validators against the refined prompt, persists each validator result as `StageSupporting`, skips optional unavailable validators without counting them as executed reviewers, and fails early if any required validator is unavailable.
- [ ] Prompt review enforces `min_reviewers` after filtering and after execution; if executed validators are fewer than `min_reviewers`, the stage fails explicitly.
- [ ] Prompt review succeeds only when every executed validator accepts the refined prompt; on success it writes `prompt.original.md` with the pre-review prompt, replaces `prompt.md` with the refined prompt, recomputes `ProjectRecord.prompt_hash`, persists one canonical prompt-review `StagePrimary` record, and then advances to planning.
- [ ] Prompt review failure invariant: if the refiner or any validator fails or rejects after any supporting records were written, those supporting records remain durable evidence, but `prompt.md`, `prompt.original.md`, `project.toml` prompt metadata, `stage_completed`, and the stage cursor remain unchanged.
- [ ] Completion resolves its completer panel through `BackendPolicyService::resolve_completion_panel(...)`, invokes every resolved completer, and persists each completer response as `StageSupporting`.
- [ ] Completion computes its canonical aggregate from actual executed voters only; the verdict is `complete` only when `complete_votes >= min_completers` and `complete_votes / total_voters >= consensus_threshold`, otherwise the verdict is `continue_work`.
- [ ] Completion persists one canonical `StageAggregate` record containing the aggregate verdict, vote counts, threshold used, and executed voter list; this aggregate is the only completion result used by resume logic and downstream transitions.
- [ ] Completion `complete` advances to `acceptance_qa` without changing remediation cycle state; completion `continue_work` uses the existing completion-round restart path by advancing `completion_round`, returning to the flow’s planning stage, and not incrementing `cycle`.
- [ ] Completion failure invariant: if one or more completer supporting records were persisted but aggregate persistence or cursor transition fails, supporting records remain durable, but no aggregate record or `stage_completed` event is written and the run remains resumable from `completion_panel`.
- [ ] Resume drift detection compares the current stage or panel resolution against `ActiveRun.stage_resolution_snapshot` before resume preflight; if the resolution changed but still satisfies required-backend and minimum-count rules, the run emits one runtime `Warn` entry and one durable journal warning with old and new resolution details, updates the snapshot, and continues with the new resolution.
- [ ] Resume drift failure invariant: if re-resolution drops a required backend or leaves fewer than `min_reviewers` or `min_completers`, resume fails early with no invocations, no silent substitution, and no stage transition.
- [ ] `PayloadRecord` and `ArtifactRecord` gain `record_kind`, `producer`, and `completion_round`; existing single-agent stage success records are marked `StagePrimary`, prompt-review and completion per-agent records are `StageSupporting`, and completion aggregates are `StageAggregate`.
- [ ] `run history` and `run tail` remain usable with multi-record stages by surfacing record kind and producer metadata so canonical stage records stay distinguishable from supporting panel evidence.
- [ ] Add unit coverage for prompt-review accept/reject paths, `min_reviewers` enforcement, completion threshold math, supporting-versus-aggregate record serialization, prompt replacement atomicity, and resume drift comparison.
- [ ] Add conformance coverage for `workflow.prompt_review.panel_accept`, `workflow.prompt_review.panel_reject`, `workflow.prompt_review.min_reviewers_enforced`, `workflow.prompt_review.optional_validator_skip`, `workflow.prompt_review.prompt_replaced_and_original_preserved`, `workflow.completion.panel_two_completer_consensus_complete`, `workflow.completion.panel_continue_verdict`, `workflow.completion.optional_backend_skip`, `workflow.completion.required_backend_failure`, `workflow.completion.threshold_consensus`, `workflow.completion.insufficient_min_completers`, `backend.resume_drift.implementation_warns_and_reresolves`, `backend.resume_drift.qa_warns_and_reresolves`, `backend.resume_drift.review_warns_and_reresolves`, and `backend.resume_drift.completion_panel_warns_and_reresolves`.
