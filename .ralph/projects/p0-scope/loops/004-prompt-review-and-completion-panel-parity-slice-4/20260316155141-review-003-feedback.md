## Required Changes
1. **Completion failure invariant**: `completion_panel` still leaks post-panel state when aggregate commit fails, so the run is no longer resumable from `completion_panel` and a canonical aggregate can survive without a matching stage completion.
   - Current: [engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L1318) writes `completion_round_advanced` and [engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L1357) persists the planning cursor before aggregate commit; [engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L3978) writes the `StageAggregate` pair before the `stage_completed` append and [engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L3997) does not roll it back on append failure; [engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L3465) then reconstructs resume state from `completion_round_advanced`, and [queries.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/queries.rs#L177) always exposes `StageAggregate` records.
   - Expected: if aggregate persistence or the post-panel transition fails, no aggregate record or `stage_completed` event should remain durable, and resume must restart from `completion_panel`, not from the planning restart path.
   - Reference: Slice 4 acceptance criteria for `workflow.completion.panel_continue_verdict` and “Completion failure invariant”.

2. **Resume drift snapshot update**: the drift-warning path does not actually persist the new resolution snapshot during resume.
   - Current: [engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L755) detects drift while the failed/paused snapshot has no `active_run`; [engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L4224) only writes `active_run.stage_resolution_snapshot`, so nothing is updated; [engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L809) then creates a new `ActiveRun` with `stage_resolution_snapshot: None`, leaving the persisted snapshot stale.
   - Expected: the warning path must persist the newly resolved snapshot before continuing, either by updating `last_stage_resolution_snapshot` and/or seeding the resumed `ActiveRun.stage_resolution_snapshot` with the new resolution.
   - Reference: Slice 4 acceptance criteria for resume drift detection, warning emission, snapshot update, and continued execution with the newly resolved backend(s).
