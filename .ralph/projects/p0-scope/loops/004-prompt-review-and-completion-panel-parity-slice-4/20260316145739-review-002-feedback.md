## Required Changes
1. **Completion panel commit ordering**: `completion_panel` still commits its aggregate and `stage_completed` before the restart/advance cursor is durably recorded, which breaks the Slice 4 failure invariant.
   Current: [engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L3783), [engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L3818), [engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L1196), and [engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L1271) persist the aggregate and `stage_completed` inside `dispatch_completion_panel` before the caller durably writes the next-stage cursor or `completion_round_advanced`. Resume state is then derived from journal `StageCompleted` / `CompletionRoundAdvanced` ordering in [engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L3371), so a failure after `stage_completed` but before the restart commit can resume from the wrong stage.
   Expected: make the completion aggregate + `stage_completed` atomic with the post-panel transition, or roll them back on transition failure, so `continue_work` remains resumable from the planning restart path and no canonical completion result leaks when the transition did not commit.
   Reference: Slice 4 acceptance criteria for `workflow.completion.panel_continue_verdict`, “Completion continue_work uses the existing completion-round restart path…”, and “Completion failure invariant”.

2. **Resume drift ordering**: drift detection still runs after resume preflight instead of before it.
   Current: [engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L749) calls `preflight_check` before the drift comparison block at [engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L760). `preflight_check` itself only validates the static `StagePlan.target` values in [engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L205), so resume can fail or pass before the required drift re-resolution/warning path runs.
   Expected: re-resolve the current stage or panel, compare it to the persisted snapshot, and emit/fail through the resume-drift path before resume preflight proceeds.
   Reference: Slice 4 acceptance criterion “Resume drift detection compares the current stage or panel resolution … before resume preflight”.

3. **Per-member timeout resolution**: mixed-backend panels still reuse one timeout from the refiner/planner backend instead of resolving timeouts per invoked backend.
   Current: [engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L3529) and [engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L3714) compute a single `panel_timeout`, then [prompt_review.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/prompt_review.rs#L98), [prompt_review.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/prompt_review.rs#L150), and [completion.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/completion.rs#L121) apply that same timeout to every validator/completer invocation. A `claude` refiner with `codex` validators, or a mixed completion panel, therefore ignores per-backend role timeout policy for non-refiner/non-planner members.
   Expected: resolve `timeout_for_role(...)` per invocation from the actual member backend family being invoked.
   Reference: Shared Foundations, Backend Policy Rules: “Resolve role timeouts from `backends.<name>.role_timeouts.<role>`.”
