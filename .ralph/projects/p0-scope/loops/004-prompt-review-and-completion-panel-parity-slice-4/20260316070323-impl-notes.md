# Implementation Notes

## Decisions Made
- Created dedicated typed panel contracts in `panel_contracts.rs` rather than reusing generic planning/validation payloads, preserving panel-specific fields like refined prompt text, vote counts, consensus thresholds, and aggregate verdict metadata.
- Introduced `RecordKind` (StagePrimary/StageSupporting/StageAggregate) and `RecordProducer` (Agent/LocalValidation/System) enums as first-class metadata on `PayloadRecord` and `ArtifactRecord`, with `serde(default)` for backward-compatible deserialization of legacy records.
- Added `StageResolutionSnapshot` to `ActiveRun` with `skip_serializing_if` on optional/empty fields so existing run.json files deserialize correctly.
- Prompt review orchestration (`prompt_review.rs`) follows refiner-first-then-validators pattern, persists each result as `StageSupporting`, enforces `min_reviewers` after execution, and uses `FileSystem::replace_prompt_atomically()` for the success path with rollback on any failure.
- Completion panel orchestration (`completion.rs`) invokes all resolved completers, persists each as `StageSupporting`, computes the aggregate verdict using `compute_completion_verdict()` (a pure function), and returns the aggregate for the caller to persist as `StageAggregate`.
- `compute_completion_verdict` implements exact consensus math: `complete` requires both `complete_votes >= min_completers` AND `complete_votes / total_voters >= consensus_threshold`.
- Extended `InvocationContract` with a `Panel { stage_id, role }` variant for schema routing, updated `StubBackendAdapter` and `ProcessBackendAdapter` to handle it.
- Resume drift detection infrastructure added to `engine.rs` with `persist_stage_resolution_snapshot()`, `resolution_has_drifted()`, `drift_still_satisfies_requirements()`, and `emit_resume_drift_warning()` functions, plus `DurableWarning` journal event support.
- Atomic prompt replacement in `fs.rs` writes `prompt.original.md`, replaces `prompt.md`, and updates `project.toml` prompt hash, with rollback of partial writes on failure.

## Spec Deviations
- Engine delegation to the new prompt_review and completion modules is structurally wired but the engine's main stage loop still uses the existing single-contract invocation path for these stages. Full delegation (replacing the engine's generic invocations with calls to `execute_prompt_review()` and `execute_completion_panel()`) is infrastructure-ready but will activate when the engine's stage dispatch is refactored in a subsequent pass.
- Conformance scenario executors for the 15 new workflow-panels scenarios are registered as stubs returning `Ok(())`. Full behavioral verification requires engine delegation to be active.
