# Implementation Response (Iteration 5)

## Changes Made
1. **RC1 — Explicit GitHub command intake**: `poll_and_ingest_repo` now has two phases. Phase A polls `rb:in-progress` and `rb:failed` issues to discover `/rb retry` and `/rb abort` commands on non-ready issues. Phase B polls `rb:ready` for standard new-task intake. The new `handle_explicit_command` helper mutates durable task state and reconciles the GitHub label to match (retried → `rb:ready`, aborted → `rb:failed`). Label sync failures propagate via `?` to quarantine the repo for the cycle.

2. **RC2 — Abort cleanup uses correct repo root**: `cleanup_aborted_task` now accepts a separate `repo_root` parameter (distinct from `base_dir`). In multi-repo mode, `handle_abort_by_issue` passes `DataDirLayout::checkout_path()` as `repo_root` and the daemon shard as `base_dir`, so `LeaseService::release` runs Git worktree operations against the actual checkout. The legacy single-repo handler passes the same path for both (unchanged behavior).

3. **RC3 — Repo validation checks Git checkout**: `validate_repo_checkout` now verifies three things: (a) directory exists, (b) `.git` marker exists (directory or file), and (c) `git rev-parse --git-dir` succeeds. This catches corrupt clones, non-Git directories with `.ralph-burning`, and missing Git installations early at `daemon start` time.

4. **RC4 — File-based intake removed from production CLI**: `daemon start` without `--data-dir` and without `--single-iteration` returns a clear error requiring `--data-dir`. The `--single-iteration` flag without `--data-dir` routes to a test-only handler that uses `FileIssueWatcher` internally but is explicitly gated behind the test flag. The `handle_start_legacy` function that was previously reachable without any gating has been removed. Production daemon path (`--data-dir`) never instantiates `FileIssueWatcher`.
