# Review: SUGGESTIONS

## Required Changes
1. **Explicit GitHub command intake**: command polling still only sees `rb:ready` issues, so `/rb retry` on `rb:failed` issues and `/rb abort` on `rb:in-progress` issues are never observed, and the inline handler strips `rb:ready` instead of reconciling to the durable target label.
   - Current: [`github_intake.rs#L107`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/github_intake.rs#L107) polls only `rb:ready`; [`github_intake.rs#L140`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/github_intake.rs#L140) handles `/rb retry` and `/rb abort` only inside that ready-only loop; [`github_intake.rs#L167`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/github_intake.rs#L167) removes `rb:ready` after mutating task state and never syncs `rb:failed` or `rb:ready`.
   - Expected: explicit commands must be routable regardless of the issue’s current truthful status label, and label reconciliation must follow the durable task state after `/rb retry` and `/rb abort`.
   - Reference: Slice 8 acceptance criteria for explicit GitHub commands, routing precedence, and status-label mapping.

2. **Abort cleanup uses the wrong repo root in multi-repo mode**: `daemon abort <issue-number>` does not release active/claimed task resources against the checkout repo.
   - Current: [`daemon.rs#L364`](/root/new-ralph-burning/ralph-burning-rewrite/src/cli/daemon.rs#L364) calls `cleanup_aborted_task` with the per-repo `daemon/` shard; [`daemon.rs#L736`](/root/new-ralph-burning/ralph-burning-rewrite/src/cli/daemon.rs#L736) passes that same path as both `base_dir` and `repo_root` to `LeaseService::release`; [`lease_service.rs#L262`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/lease_service.rs#L262) removes the worktree by running Git in `repo_root`. In multi-repo mode that means cleanup runs in `<data-dir>/.../daemon`, not `<data-dir>/.../repo`.
   - Expected: abort-by-issue must release worktrees and writer locks using the registered checkout root so active/claimed tasks do not strand lease/worktree state.
   - Reference: Slice 8 acceptance criteria for `daemon abort <issue-number> --data-dir --repo`, truthful durable state, and per-repo worktree handling.

3. **Repo validation is not strong enough for “fail early and explicitly”**: startup accepts unusable checkouts until later runtime operations fail.
   - Current: [`repo_registry.rs#L168`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/repo_registry.rs#L168) only checks “directory exists” plus `.ralph-burning`; [`daemon.rs#L195`](/root/new-ralph-burning/ralph-burning-rewrite/src/cli/daemon.rs#L195) treats that as complete upfront validation. A directory with `.ralph-burning` but no usable Git checkout passes startup and fails later on worktree/rebase operations.
   - Expected: `daemon start` should reject unusable checkouts up front by validating that the registered `repo/` path is a usable Git checkout, not just a directory with a workspace subdirectory.
   - Reference: Slice 8 acceptance criterion that invalid repo state or checkout bootstrap failure must fail early and explicitly.

4. **File-based intake is still reachable from the shipping CLI**: the legacy watcher path remains a normal runtime branch.
   - Current: [`daemon.rs#L106`](/root/new-ralph-burning/ralph-burning-rewrite/src/cli/daemon.rs#L106) still routes `daemon start` without `--data-dir` into legacy mode when `--single-iteration` is set; [`daemon.rs#L538`](/root/new-ralph-burning/ralph-burning-rewrite/src/cli/daemon.rs#L538) instantiates `FileIssueWatcher` in that path.
   - Expected: GitHub-backed `--data-dir` mode should be the only CLI-accessible daemon intake path; file-based issue watching should remain test-only infrastructure.
   - Reference: Slice 8 acceptance criterion “Production daemon intake no longer instantiates `FileIssueWatcher`”.
