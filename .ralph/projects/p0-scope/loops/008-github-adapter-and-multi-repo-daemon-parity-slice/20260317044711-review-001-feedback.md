# Review: SUGGESTIONS

## Required Changes
1. **GitHub-backed multi-repo daemon start path**: `daemon start --data-dir` still does not execute the slice 8 GitHub runtime.
   - Current: [`src/cli/daemon.rs#L167`](/root/new-ralph-burning/ralph-burning-rewrite/src/cli/daemon.rs#L167) validates repos, then runs a single daemon loop against `registrations[0].repo_root`; [`src/contexts/automation_runtime/daemon_loop.rs#L177`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/daemon_loop.rs#L177) only processes one `base_dir` and only polls `watcher`; the available GitHub intake helper at [`src/contexts/automation_runtime/github_intake.rs#L95`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/github_intake.rs#L95) is not wired in. Production also still instantiates `FileIssueWatcher` in [`src/cli/daemon.rs#L524`](/root/new-ralph-burning/ralph-burning-rewrite/src/cli/daemon.rs#L524).
   - Expected: `daemon start --data-dir --repo ...` should ensure labels, poll candidate issues from every registered repo, ingest tasks through GitHub, and isolate per-repo failures. File-based issue intake should be test-only, not a production path.
   - Reference: Slice 8 acceptance criteria for GitHub-backed production intake, label ensuring on startup, multi-repo failure isolation, and `FileIssueWatcher` deprecation.

2. **Daemon state and worktree layout**: durable daemon state is still stored in each checkout’s `.ralph-burning`, and branch/worktree naming still follows the old single-repo shape.
   - Current: [`src/adapters/fs.rs#L1244`](/root/new-ralph-burning/ralph-burning-rewrite/src/adapters/fs.rs#L1244) stores tasks, leases, and daemon journal under `base_dir/.ralph-burning/daemon/...`; multi-repo CLI paths keep passing checkout paths into that store in [`src/cli/daemon.rs#L326`](/root/new-ralph-burning/ralph-burning-rewrite/src/cli/daemon.rs#L326), [`src/cli/daemon.rs#L360`](/root/new-ralph-burning/ralph-burning-rewrite/src/cli/daemon.rs#L360), and [`src/cli/daemon.rs#L459`](/root/new-ralph-burning/ralph-burning-rewrite/src/cli/daemon.rs#L459). `WorktreePort` still only exposes `branch_name(task_id)` in [`src/contexts/automation_runtime/mod.rs#L103`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/mod.rs#L103), and [`src/adapters/worktree.rs#L341`](/root/new-ralph-burning/ralph-burning-rewrite/src/adapters/worktree.rs#L341) still uses `.ralph-burning/worktrees/<task-id>` with branch `rb/task/<task-id>`.
   - Expected: daemon runtime state must live under `<data-dir>/repos/<owner>/<repo>/daemon/...`, and task worktrees/branches must use `<data-dir>/repos/<owner>/<repo>/worktrees/<task-id>/` and `rb/<issue-number>-<project-id>`.
   - Reference: Slice 8 acceptance criteria for `--data-dir` layout, worktree isolation, and branch naming.

3. **Explicit GitHub commands**: `/rb run`, `/rb retry`, and `/rb abort` are not actually supported by the ingest path, and comment-based commands are ignored during polling.
   - Current: [`src/contexts/automation_runtime/routing.rs#L67`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/routing.rs#L67) treats those commands as non-routing errors; [`src/contexts/automation_runtime/task_service.rs#L519`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/task_service.rs#L519) only filters out requirements commands before flow resolution, so daemon commands still fail task creation instead of falling through to label/default routing. Polling also ignores issue comments by passing an empty comment list in [`src/contexts/automation_runtime/github_intake.rs#L119`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/github_intake.rs#L119).
   - Expected: explicit GitHub commands should work with the documented precedence, including commands authored in comments, without breaking ingestion.
   - Reference: Slice 8 acceptance criteria for command vocabulary, routing precedence, and issue-comment fetching.

4. **Durable repo registration**: repo registration is not persisted as registry state.
   - Current: [`src/contexts/automation_runtime/repo_registry.rs#L142`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/repo_registry.rs#L142) defines `RepoRegistryPort`, but there is no implementation; [`src/contexts/automation_runtime/repo_registry.rs#L190`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/repo_registry.rs#L190) only creates directories and returns a `RepoRegistration`; `status --data-dir` rebuilds repo membership by directory scan in [`src/cli/daemon.rs#L271`](/root/new-ralph-burning/ralph-burning-rewrite/src/cli/daemon.rs#L271).
   - Expected: each registered repo should have durable metadata including `repo_slug`, `repo_root`, and `workspace_root`, and daemon commands should read that registry rather than infer it from folder shape.
   - Reference: Slice 8 acceptance criteria for durable `RepoRegistration` and repo/task metadata.
