# Feature: GitHub Adapter and Multi-Repo Daemon Parity (Slice 8)

## Description
Loops 1 through 7 are already complete and approved, and git history currently stops at `ralph(p0-scope): loop 7 committing -> planning`, so the next missing P0 slice is the GitHub-backed daemon layer. The rewrite repo already has single-repo daemon routing, task/lease lifecycle, requirements handoff, checkpoint-aware worktree handling, and baseline rebase support, but production intake still depends on the test-only `FileIssueWatcher`, current-directory storage, and task-id based daemon commands. This slice adds the missing GitHub adapter, repo registry, and `--data-dir` multi-repo runtime without re-planning the already-implemented workflow engine, routing core, lease cleanup, or requirements execution paths.

## Acceptance Criteria
- [ ] `ralph-burning daemon start --data-dir <dir> --repo <owner/repo>... [--poll-seconds N] [--single-iteration] [--verbose]`, `status --data-dir [--repo <owner/repo>...]`, `abort <issue-number> --data-dir --repo <owner/repo>`, `retry <issue-number> --data-dir --repo <owner/repo>`, and `reconcile --data-dir <dir>` are implemented exactly.
- [ ] A durable `RepoRegistration` model exists for each registered repo and records at least `repo_slug`, `repo_root`, and `workspace_root`.
- [ ] The daemon uses the required layout under `<data-dir>/repos/<owner>/<repo>/repo`, `worktrees`, and `daemon/{tasks,leases,journal.ndjson}`; project and run state remain inside each checkout’s `.ralph-burning` workspace.
- [ ] `daemon start` validates `--data-dir` and every requested repo before polling. Invalid repo slug syntax, missing/unusable checkout state, or checkout bootstrap failure fail early and explicitly instead of silently skipping.
- [ ] Production daemon intake no longer instantiates `FileIssueWatcher`; GitHub-backed intake is the only production path, and file-based issue sources remain test-only.
- [ ] `src/adapters/github.rs` provides the P0 GitHub operations required by slices 8 and 9: ensure labels, poll candidate issues by label, read/add/remove/replace labels, fetch issue comments, fetch PR review comments, fetch PR review summaries, post idempotent comments, create draft PRs, mark PRs ready, close PRs, fetch PR URL/state, update PR body, and detect whether a branch is ahead of base.
- [ ] Startup ensures the label vocabulary `rb:ready`, `rb:in-progress`, `rb:failed`, `rb:completed`, `rb:flow:standard`, `rb:flow:quick_dev`, `rb:flow:docs_change`, `rb:flow:ci_improvement`, `rb:requirements`, and `rb:waiting-feedback` exists for every registered repo.
- [ ] GitHub intake polls candidate issues from registered repos and persists task metadata including `repo_slug`, `issue_number`, derived `issue_ref`, optional `pr_url`, and a dedup cursor or equivalent last-seen review/comment state.
- [ ] Routing precedence remains exact: explicit command, then flow label, then repo default routing policy.
- [ ] Explicit GitHub commands support `/rb flow <preset>`, `/rb requirements`, `/rb run`, `/rb retry`, and `/rb abort`, and they reuse the existing workflow and requirements execution paths instead of creating a second orchestration stack.
- [ ] Task identity and duplicate prevention are keyed by `repo_slug + issue_number`; `daemon abort` and `daemon retry` resolve tasks by repo plus issue number and fail clearly when no matching task exists or the requested transition is invalid.
- [ ] Status-label mapping is driven from durable task state: pending uses `rb:ready`, claimed/active uses `rb:in-progress`, waiting-for-requirements uses `rb:waiting-feedback`, completed uses `rb:completed`, and failed/aborted uses `rb:failed`.
- [ ] Partial label-write failures preserve truthful durable state. The task record must exist before `rb:in-progress` becomes authoritative; if add/remove label operations only partially succeed, the daemon must still prevent duplicate task creation, keep lease/worktree ownership accurate, and surface the mismatch for later reconcile/repair.
- [ ] Multi-repo failure isolation holds: a GitHub polling, label, or comment failure for one repo does not block other repos in the same daemon cycle, and no further task/lease/worktree mutation occurs for the failing repo after the failing operation.
- [ ] Task worktrees move under `<data-dir>/repos/<owner>/<repo>/worktrees/<task-id>/`, and task branches use `rb/<issue-number>-<project-id>`.
- [ ] Existing daemon routing, requirements handoff, lease reconciliation, and default-branch rebase behavior continue to work after GitHub intake claims a task.
- [ ] Conformance coverage is added and passing for `daemon.github.start_validates_repos_and_data_dir`, `daemon.github.multi_repo_status`, `daemon.routing.command_beats_label`, `daemon.routing.label_used_when_no_command`, `daemon.labels.ensure_on_startup`, `daemon.tasks.abort_by_issue_number`, `daemon.tasks.retry_failed_issue`, `daemon.tasks.reconcile_stale_leases`, and `daemon.tasks.worktree_isolation`.
