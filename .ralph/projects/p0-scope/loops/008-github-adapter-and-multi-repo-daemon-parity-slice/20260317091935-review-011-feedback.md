# Review: SUGGESTIONS

## Required Changes
1. **Per-repo label-failure isolation**: dirty-label repair failures still allow further mutation in the same repo cycle.  
Current: [daemon_loop.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/daemon_loop.rs#L274) treats Phase 0 `label_dirty` repair as best-effort; if `sync_label_for_task()` fails there, the loop still proceeds into GitHub polling at [daemon_loop.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/daemon_loop.rs#L288) and pending-task processing at [daemon_loop.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/daemon_loop.rs#L342).  
Expected: a GitHub label failure during repair must quarantine that repo for the rest of the cycle, just like the later resumed-task label-sync failure already does, so no more task/lease/worktree mutation happens after the failing operation.  
Reference: Slice 8 acceptance criteria for multi-repo failure isolation and repairable label mismatch handling.

2. **Direct abort/retry label drift**: `daemon abort` and `daemon retry` can change durable task state without surfacing an unsynced GitHub label.  
Current: [daemon.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/cli/daemon.rs#L346) and [daemon.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/cli/daemon.rs#L391) only attempt GitHub label reconciliation when `GithubClientConfig::from_env()` succeeds. If `GITHUB_TOKEN` is absent, the command still aborts or retries the task but never marks `label_dirty`, so the required status-label mismatch is neither explicit nor repairable later.  
Expected: if these commands cannot initialize GitHub sync, they should either fail explicitly or at minimum persist `label_dirty` and warn, so reconcile has durable evidence to repair.  
Reference: Slice 8 acceptance criteria for status-label mapping and the requirement to surface label-write mismatches for later reconcile/repair.
