# Implementation Response (Iteration 24)

## Changes Made
1. **RC1 — Repo slug validation: reject dot segments and unsafe components**: Tightened `parse_repo_slug()` in `repo_registry.rs` to explicitly reject `.` and `..` segments (in addition to empty segments and path separators). Values like `acme/.`, `acme/..`, `./repo`, and `../repo` now fail with a descriptive `InvalidConfigValue` error before any data-dir path is built. Added `parse_rejects_dot_segments` unit test covering all five dot-segment variants.

2. **RC2 — Explicit command precedence for `/rb run` vs `/rb requirements`**: In `poll_and_ingest_repo()` (`github_intake.rs`), the extracted explicit command is now checked for `/rb run` *before* it is filtered out as a daemon command. When `/rb run` is detected, `DispatchMode::Workflow` is forced directly, bypassing `resolve_dispatch_mode()` entirely. This prevents stale `/rb requirements` text in the issue body from overriding a newer `/rb run` comment.

3. **RI1 — Conformance coverage**:
   - `daemon.slug_validation.rejects_dot_segments`: verifies `acme/.`, `acme/..`, `./repo`, `../repo`, and `./..` are all rejected, and valid slugs still succeed.
   - `daemon.routing.run_overrides_stale_requirements`: verifies that when an issue body contains `/rb requirements draft` but a newer comment says `/rb run`, the dispatch mode is forced to `Workflow`. Also proves the fix is load-bearing by showing that without the override, body scanning would produce `RequirementsDraft`.
