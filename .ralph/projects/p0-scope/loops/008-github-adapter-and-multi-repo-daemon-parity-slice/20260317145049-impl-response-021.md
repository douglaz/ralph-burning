# Implementation Response (Iteration 21)

## Changes Made
1. **RC1 — Startup repo validation: parse and validate workspace config**: Strengthened `validate_repo_checkout()` in `repo_registry.rs` to call `load_workspace_config()` after verifying `workspace.toml` exists. A malformed workspace config (invalid TOML, missing required fields like `version`, wrong structure) is now caught at `daemon start` time instead of failing later during task processing when `EffectiveConfig::load()` is called. The error message identifies the specific parse/validation failure.

2. **RC2 — Startup label ensure: fail explicitly on partial success**: Changed `run_multi_repo()` in `daemon_loop.rs` to compare `active_registrations.len()` against `self.registrations.len()` after `ensure_labels_on_repos()`. If any requested repo was quarantined (label ensure failed), the daemon now returns an explicit startup error naming the failing repos, instead of silently excluding them and continuing with a subset.

3. **RI1 — Extended conformance coverage for startup paths**:
   - `daemon.github.start_validates_repos_and_data_dir`: Added two new sub-tests verifying that `validate_repo_checkout()` rejects checkouts with structurally invalid `workspace.toml` (valid TOML but wrong schema) and completely broken TOML syntax.
   - `daemon.labels.ensure_on_startup`: Extended to verify that `ensure_labels_on_repos()` correctly filters repos whose label ensure fails, using `InMemoryGithubClient` with a new `set_ensure_labels_failure()` test helper. Also verifies the startup contract (filtered count < requested count triggers the startup error path).
   - Added `ensure_labels_failure_repos` field and `set_ensure_labels_failure()` method to `InMemoryGithubClient`, with the `ensure_labels` port implementation checking the failure set and returning `BackendUnavailable` for configured repos.
