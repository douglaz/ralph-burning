# Review: SUGGESTIONS

## Required Changes
1. **Repo bootstrap/start path**: `daemon start` still cannot bring up a fresh repo shard from just `--data-dir` plus `--repo`.
Current: [`handle_start_multi_repo()`](/root/new-ralph-burning/ralph-burning-rewrite/src/cli/daemon.rs#L142) calls [`register_repo()`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/repo_registry.rs#L230), which only creates empty directories under `<data-dir>/repos/<owner>/<repo>/repo`, and then immediately calls [`validate_repo_checkout()`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/repo_registry.rs#L171), which requires both `.git` and `.ralph-burning` to already exist. A fresh `daemon start --data-dir ... --repo owner/repo` therefore fails unless the checkout was manually pre-populated out of band.
Expected: implement the missing checkout/workspace bootstrap path under the data-dir shard and then validate that result, so only real bootstrap failures fail start explicitly.
Reference: [20260317041753-spec.md](/root/new-ralph-burning/.ralph/projects/p0-scope/loops/008-github-adapter-and-multi-repo-daemon-parity-slice/20260317041753-spec.md#L24), [20260317041753-spec.md](/root/new-ralph-burning/.ralph/projects/p0-scope/loops/008-github-adapter-and-multi-repo-daemon-parity-slice/20260317041753-spec.md#L25), [20260317041753-spec.md](/root/new-ralph-burning/.ralph/projects/p0-scope/loops/008-github-adapter-and-multi-repo-daemon-parity-slice/20260317041753-spec.md#L43)

2. **Dedup cursor persistence**: the slice adds comment/review cursor fields, but the runtime never persists a real cursor value.
Current: [`build_github_meta()`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/github_intake.rs#L77) initializes `last_seen_comment_id` and `last_seen_review_id` to `None`, and [`poll_and_ingest_repo()`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/github_intake.rs#L229) only writes those `None` values onto the task. There is no later update, so tasks do not actually store a "last-seen review/comment state".
Expected: persist the latest seen issue-comment/review ids, or an equivalent durable cursor, during intake/command polling so later cycles and slice-9 review ingestion can deduplicate from task state.
