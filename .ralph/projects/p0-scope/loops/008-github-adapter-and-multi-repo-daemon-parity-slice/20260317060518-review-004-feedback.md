# Review: SUGGESTIONS

## Required Changes
1. **Repo workspace boundary**: the multi-repo daemon still roots workflow and requirements state in the daemon shard instead of the checkout workspace.
   - Current: multi-repo processing passes the per-repo `daemon/` directory into requirements, project creation, and workflow dispatch in [daemon_loop.rs#L359](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/daemon_loop.rs#L359), [daemon_loop.rs#L421](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/daemon_loop.rs#L421), [daemon_loop.rs#L463](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/daemon_loop.rs#L463), [daemon_loop.rs#L1275](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/daemon_loop.rs#L1275), and [daemon_loop.rs#L1376](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/daemon_loop.rs#L1376). Those stores derive `.ralph-burning` and requirements paths from whatever `base_dir` they receive in [fs.rs#L356](/root/new-ralph-burning/ralph-burning-rewrite/src/adapters/fs.rs#L356) and [fs.rs#L2378](/root/new-ralph-burning/ralph-burning-rewrite/src/adapters/fs.rs#L2378).
   - Expected: only daemon task/lease/journal state should live under `<data-dir>/repos/<owner>/<repo>/daemon`; project, run, artifact, and requirements state must live under `<data-dir>/repos/<owner>/<repo>/repo/.ralph-burning`.
   - Reference: Slice 8 acceptance criteria for the `<data-dir>` layout; Master Prompt non-negotiables “Daemon state lives under `--data-dir`” and “Project and run state live inside each repo checkout’s `.ralph-burning` workspace”.

2. **Production intake path**: the normal CLI still exposes the legacy file-watcher daemon mode.
   - Current: `daemon start` without `--data-dir` still falls back to `handle_start_legacy` in [daemon.rs#L98](/root/new-ralph-burning/ralph-burning-rewrite/src/cli/daemon.rs#L98), and that path still instantiates `FileIssueWatcher` in [daemon.rs#L509](/root/new-ralph-burning/ralph-burning-rewrite/src/cli/daemon.rs#L509).
   - Expected: GitHub-backed `--data-dir` mode should be the only production daemon path; file-based issue watching should remain test-only and not be reachable through the normal CLI surface.
   - Reference: Slice 8 acceptance criterion “Production daemon intake no longer instantiates `FileIssueWatcher`”; Master Prompt delivery rule 7.

3. **GitHub labels and repo-failure handling**: label state is still best-effort instead of being driven from durable task state.
   - Current: startup ignores label-ensure failures in [daemon_loop.rs#L201](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/daemon_loop.rs#L201); new tasks remove `rb:ready` while the durable task remains `Pending` in [github_intake.rs#L208](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/github_intake.rs#L208); steady-state sync swallows add/remove failures in [github_intake.rs#L251](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/github_intake.rs#L251); issue-based `abort` and `retry` mutate task state without any GitHub reconciliation in [daemon.rs#L357](/root/new-ralph-burning/ralph-burning-rewrite/src/cli/daemon.rs#L357) and [daemon.rs#L393](/root/new-ralph-burning/ralph-burning-rewrite/src/cli/daemon.rs#L393).
   - Expected: `Pending -> rb:ready`, `Claimed/Active -> rb:in-progress`, `WaitingForRequirements -> rb:waiting-feedback`, `Completed -> rb:completed`, `Failed/Aborted -> rb:failed`; label ensure/sync failures must stop or quarantine further mutation for that repo in the current cycle instead of being silently ignored.
   - Reference: Slice 8 acceptance criteria for startup label ensuring, status-label mapping, partial label-write failure behavior, and multi-repo failure isolation.
