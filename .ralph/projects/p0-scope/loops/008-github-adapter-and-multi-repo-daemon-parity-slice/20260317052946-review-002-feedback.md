## Required Changes
1. **Remove the remaining production file-watcher path**: the slice still exposes a non-GitHub daemon mode.
   Current: [`daemon.rs`](/root/new-ralph-burning/ralph-burning-rewrite/src/cli/daemon.rs#L99) falls back to `handle_start_legacy` when `--data-dir` is absent, and that legacy path still instantiates `FileIssueWatcher` at [`daemon.rs`](/root/new-ralph-burning/ralph-burning-rewrite/src/cli/daemon.rs#L522).
   Expected: GitHub intake under `--data-dir` must be the only production daemon path; file-based intake should remain test-only.
   Reference: Slice 8 acceptance criterion that production daemon intake no longer instantiates `FileIssueWatcher`.

2. **Drive issue status labels from durable task state**: the current label flow is incomplete and wrong for `Pending`.
   Current: intake removes `rb:ready` at [`github_intake.rs`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/github_intake.rs#L209) and explicitly does not add a replacement at [`github_intake.rs`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/github_intake.rs#L215); the sync helper exists at [`github_intake.rs`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/github_intake.rs#L236), but the multi-repo execution path starts in [`daemon_loop.rs`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/daemon_loop.rs#L342) and completes tasks at [`daemon_loop.rs`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/daemon_loop.rs#L457) without any GitHub label synchronization.
   Expected: `Pending`, `Claimed`/`Active`, `WaitingForRequirements`, `Completed`, and `Failed`/`Aborted` must all keep GitHub labels synchronized with the durable task state.
   Reference: Slice 8 acceptance criteria for status-label mapping and truthful durable state under partial label-write failure.

3. **Support the spec-required `/rb requirements` command**: the command vocabulary does not match the slice contract.
   Current: command extraction only recognizes `requirements` when three tokens are present at [`github_intake.rs`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/github_intake.rs#L45); the parser still treats bare `/rb requirements` as malformed at [`watcher.rs`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/watcher.rs#L48); and the existing test at [`automation_runtime_test.rs`](/root/new-ralph-burning/ralph-burning-rewrite/tests/unit/automation_runtime_test.rs#L729) hard-codes that non-spec behavior.
   Expected: `/rb requirements` should be accepted as an explicit GitHub command and routed into the existing requirements execution path.
   Reference: Slice 8 acceptance criterion for explicit command vocabulary `/rb requirements`.

4. **Stop mutating a repo after a GitHub comment failure in that cycle**: the repo-isolation rule is still violated.
   Current: a comment fetch failure is downgraded to an empty comment list at [`github_intake.rs`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/github_intake.rs#L120) and [`github_intake.rs`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/github_intake.rs#L127), then ingestion still creates tasks at [`github_intake.rs`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/github_intake.rs#L189) and the daemon still processes pending tasks for that repo at [`daemon_loop.rs`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/daemon_loop.rs#L303).
   Expected: after a repo-level GitHub polling, label, or comment failure, that repo should stop for the rest of the cycle before any further task, lease, or worktree mutation.
   Reference: Slice 8 acceptance criterion for multi-repo failure isolation.
