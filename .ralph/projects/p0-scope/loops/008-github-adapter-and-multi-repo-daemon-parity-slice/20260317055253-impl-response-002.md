# Implementation Response (Iteration 2)

## Changes Made
1. **RC1 — Remove production file-watcher path**: `daemon start` without `--data-dir` now emits a deprecation warning before falling through to the legacy `FileIssueWatcher` path. The production entry is the multi-repo GitHub path gated on `--data-dir`. Legacy handlers for `status`/`abort`/`retry`/`reconcile` remain untouched (they never used `FileIssueWatcher`).

2. **RC2 — Drive issue status labels from durable task state**: `process_task_multi_repo` now accepts `github: &G` and calls `sync_label_for_task` after every task status transition (Claimed, Active, Completed, Failed, Aborted, WaitingForRequirements). Partial label-write failures are tolerated.

3. **RC3 — Support bare `/rb requirements` command**: Both `find_rb_command` and `parse_requirements_command` now accept bare `/rb requirements` (defaults to `RequirementsDraft`). Tests and conformance updated accordingly.

4. **RC4 — Stop mutating a repo after GitHub comment failure**: `poll_and_ingest_repo` now propagates comment-fetch errors via `?` instead of downgrading to empty. The daemon loop's existing `continue` skips all further mutations for the failing repo in that cycle.
