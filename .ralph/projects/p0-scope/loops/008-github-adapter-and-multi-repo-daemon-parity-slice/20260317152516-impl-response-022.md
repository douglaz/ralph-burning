# Implementation Response (Iteration 22)

## Changes Made
1. **RC1 — Requirements-draft GitHub label lifecycle**: Restructured `handle_requirements_draft()` to accept a `github: &G` parameter and sync labels at every state transition:
   - **On claim/active**: `rb:in-progress` is synced immediately after the `Pending -> Claimed -> Active` transition (daemon_loop.rs:1325-1335), before the draft agent runs. Label-sync failure triggers `label_dirty` and quarantines the repo, matching the existing workflow-claim pattern.
   - **On failure exits**: Every `mark_failed` call within the draft path is followed by `sync_label_after_mutation()` to push `rb:failed` to the issue.
   - **On WaitingForRequirements**: `rb:waiting-feedback` is synced after `mark_waiting_for_requirements` succeeds.
   - **On empty-question completion (requeue to Pending)**: `rb:ready` is synced after the task transitions back to `Pending`.
   - Added `sync_label_after_mutation()` helper (daemon_loop.rs:805-821) that does best-effort label sync with `label_dirty` fallback, keeping the method DRY across all draft exit paths.
   - Updated the `process_task_multi_repo` call site to pass `github` directly instead of post-hoc label sync.
   - Single-repo `process_task` (test-only path) uses `InMemoryGithubClient::new()` as a no-op since file-watcher tasks have no `repo_slug` and `sync_label_for_task` short-circuits.

2. **RI1 — Conformance coverage for requirements-draft label lifecycle**: Added `daemon.labels.requirements_draft_lifecycle` scenario that exercises the full state-label chain: `Pending (rb:ready) -> Active (rb:in-progress) -> WaitingForRequirements (rb:waiting-feedback) -> Failed (rb:failed)`, verifying that stale status labels are removed at each transition.
