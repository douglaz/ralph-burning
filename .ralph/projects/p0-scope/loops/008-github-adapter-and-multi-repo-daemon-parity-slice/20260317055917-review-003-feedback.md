## Required Changes
1. **Repo workspace boundary**: The multi-repo daemon writes workflow and requirements state to the data-dir daemon shard instead of the repo checkout workspace.
   Current: `process_task_multi_repo` passes `store_dir` into requirements, project creation, and workflow dispatch in [daemon_loop.rs#L359](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/daemon_loop.rs#L359), [daemon_loop.rs#L421](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/daemon_loop.rs#L421), and [daemon_loop.rs#L1257](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/daemon_loop.rs#L1257); those stores derive `.ralph-burning` from `base_dir` in [fs.rs#L356](/root/new-ralph-burning/ralph-burning-rewrite/src/adapters/fs.rs#L356) and requirements runs from the same base in [fs.rs#L2378](/root/new-ralph-burning/ralph-burning-rewrite/src/adapters/fs.rs#L2378).
   Expected: Only daemon task/lease/journal state should live under `<data-dir>/repos/<owner>/<repo>/daemon`; project, run, artifact, and requirements state must be rooted in `<data-dir>/repos/<owner>/<repo>/repo/.ralph-burning`.
   Reference: Slice 8 acceptance criteria for the `<data-dir>` layout; Master Prompt non-negotiables “Daemon state lives under `--data-dir`” and “Project and run state live inside each repo checkout’s `.ralph-burning` workspace”.

2. **Production intake path**: The CLI still exposes the legacy file-watcher daemon mode.
   Current: `daemon start` without `--data-dir` falls back to `handle_start_legacy` in [daemon.rs#L98](/root/new-ralph-burning/ralph-burning-rewrite/src/cli/daemon.rs#L98), and that path still instantiates `FileIssueWatcher` in [daemon.rs#L509](/root/new-ralph-burning/ralph-burning-rewrite/src/cli/daemon.rs#L509).
   Expected: GitHub-backed `--data-dir` mode should be the only production daemon path; file-based issue intake should remain test-only and not be reachable through the normal CLI.
   Reference: Slice 8 acceptance criterion “Production daemon intake no longer instantiates `FileIssueWatcher`”; Master Prompt delivery rule 7.

3. **GitHub label authority and failure handling**: Label state is still best-effort instead of being driven from durable task state.
   Current: startup ignores label-ensure failures and continues in [daemon_loop.rs#L201](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/daemon_loop.rs#L201); ingestion removes `rb:ready` while leaving the task `Pending` in [github_intake.rs#L208](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/github_intake.rs#L208); `sync_label_for_task` swallows add/remove failures in [github_intake.rs#L251](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/github_intake.rs#L251); CLI `abort`/`retry` mutate durable task state without any GitHub sync in [daemon.rs#L324](/root/new-ralph-burning/ralph-burning-rewrite/src/cli/daemon.rs#L324) and [daemon.rs#L369](/root/new-ralph-burning/ralph-burning-rewrite/src/cli/daemon.rs#L369).
   Expected: startup should fail or quarantine a repo when required labels cannot be ensured; `Pending` tasks must remain `rb:ready`; every durable status transition, including CLI abort/retry, must reconcile labels or stop further repo mutation after the label failure.
   Reference: Slice 8 acceptance criteria for startup label ensure, status-label mapping, partial label-write failure behavior, and multi-repo failure isolation.
