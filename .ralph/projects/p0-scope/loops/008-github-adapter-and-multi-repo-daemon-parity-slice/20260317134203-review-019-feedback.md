# Review: SUGGESTIONS

## Required Changes
1. **Repo bootstrap credentials**: `bootstrap_repo_checkout()` still leaks the GitHub token into the cloned repo config.
   Current: [`bootstrap_repo_checkout()`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/repo_registry.rs#L256) appends `-c http.extraHeader=Authorization: Bearer ...` after `clone` in [`repo_registry.rs`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/repo_registry.rs#L303). With Git’s `clone -c` semantics, that setting is written into `.git/config`; I verified locally that this form leaves `[http] extraheader = Authorization: Bearer TESTTOKEN` in the cloned checkout.
   Expected: pass auth via a non-persisted mechanism such as `git -c http.extraHeader=... clone ...` or `GIT_CONFIG_COUNT`/`GIT_HTTP_EXTRA_HEADER`, and add a regression proving the checkout remote/config stays credential-free after bootstrap.
   Reference: Slice-8 startup/bootstrap contract in [20260317041753-spec.md](/root/new-ralph-burning/.ralph/projects/p0-scope/loops/008-github-adapter-and-multi-repo-daemon-parity-slice/20260317041753-spec.md#L25); review criterion 2 (`Code quality - secure`).

2. **Usable workspace validation/bootstrap**: `daemon start` can still accept a repo checkout that is not actually usable by the daemon.
   Current: [`handle_start_multi_repo()`](/root/new-ralph-burning/ralph-burning-rewrite/src/cli/daemon.rs#L160) treats [`validate_repo_checkout()`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/repo_registry.rs#L171) as the fail-early gate, but that validator only requires `.git` plus a `.ralph-burning/` directory at [`repo_registry.rs`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/repo_registry.rs#L217). `bootstrap_repo_checkout()` only creates that directory at [`repo_registry.rs`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/repo_registry.rs#L321), while the runtime later requires `.ralph-burning/workspace.toml` via [`load_workspace_config()`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workspace_governance/mod.rs#L59). That means a freshly bootstrapped repo can pass startup validation and then fail later on first config load instead of failing early as required.
   Expected: either initialize a real workspace during bootstrap or strengthen validation to require `workspace.toml` and the required workspace layout before accepting the repo.
   Reference: [20260317041753-spec.md](/root/new-ralph-burning/.ralph/projects/p0-scope/loops/008-github-adapter-and-multi-repo-daemon-parity-slice/20260317041753-spec.md#L24), [20260317041753-spec.md](/root/new-ralph-burning/.ralph/projects/p0-scope/loops/008-github-adapter-and-multi-repo-daemon-parity-slice/20260317041753-spec.md#L25)
