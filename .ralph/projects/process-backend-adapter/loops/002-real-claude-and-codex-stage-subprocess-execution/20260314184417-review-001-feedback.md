# Review: SUGGESTIONS

## Required Changes
1. **Cancellation and timeout handling**: the child lifecycle is not cancel-safe as implemented.
Current: [process_backend.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/adapters/process_backend.rs#L159) awaits `child.wait()` while holding `active_children`, and [process_backend.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/adapters/process_backend.rs#L533) awaits `kill` while holding the same mutex. Because [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/agent_execution/service.rs#L196) calls `cancel()` on cancellation and timeout, a long-running subprocess cannot be interrupted until it exits on its own.
Expected: drop the mutex guard before any `await` in both wait and cancel paths, so `cancel()` can send SIGTERM to a live child and the map is cleaned up on every return path.
Reference: Feature spec Acceptance Criteria "State invariant on process lifecycle" and "State invariant on cancellation"; Master Prompt Required Changes 2 and 6.

2. **Invoke-path capability enforcement**: `invoke()` does not enforce the same stage-only/supported-family rules as preflight.
Current: [process_backend.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/adapters/process_backend.rs#L520) dispatches only on backend family. That means `InvocationContract::Requirements` with Claude/Codex will still reach subprocess execution, and unsupported families return `InvocationFailed { failure_class: TransportFailure }` instead of the required `AppError::CapabilityMismatch` with the explicit `default_backend=claude|codex` guidance.
Expected: `invoke()` must reject non-`Stage` contracts and unsupported backend families before spawning, using the same clear `CapabilityMismatch` messages required by the spec.
Reference: Feature spec Acceptance Criteria first bullet; Master Prompt Required Changes 3 and 6.
