# Feature: Real Claude and Codex Stage Subprocess Execution

## Description
Loop 1 already completed the foundation: `working_dir`, `BackendAdapter` dispatch, `RALPH_BURNING_BACKEND` selection, stub-preserving harness wiring, and the 3600-second workflow timeout. This feature replaces the placeholder `ProcessBackendAdapter::invoke()` path with real stage-only subprocess execution for Claude and Codex, including stdin prompt assembly, child tracking, resume/session handling, output parsing, and transport/schema error mapping. It should not revisit CLI wiring or the later `engine.rs` prompt-enrichment work; the workflow engine can keep emitting the placeholder `Execute stage: ...` prompt in this loop.

## Acceptance Criteria
- [ ] `ProcessBackendAdapter` enforces stage-only subprocess execution in both preflight and invoke paths: `InvocationContract::Stage` with backend family `claude` or `codex` may execute, while `Requirements`, `openrouter`, and `stub` return `AppError::CapabilityMismatch` without spawning a process. The unsupported-family detail must explicitly state that `ProcessBackendAdapter` currently supports only `claude` and `codex` and that self-hosted workflow runs require `default_backend=claude` or `default_backend=codex`.
- [ ] The adapter builds a single stdin payload from `request.payload.prompt` and `request.payload.context`. The rendered input must include the contract label, backend role, primary instructions, a pretty-printed `Context JSON` section when `context` is not `null`, and a final instruction to return only JSON matching the authoritative schema. `payload.context` must remain part of the subprocess input even though `engine.rs` still uses the placeholder stage prompt in this loop.
- [ ] Claude stage execution runs in `request.working_dir` via `tokio::process::Command` with `claude -p --output-format json --model <model_id> --permission-mode acceptEdits --allowedTools Bash,Edit,Write,Read,Glob,Grep --json-schema <schema_json>`, adding `--resume <session_id>` only when `SessionPolicy::ReuseIfAllowed` and `prior_session` are both present. The adapter writes the assembled input to stdin, parses stdout as the Claude outer envelope, parses the envelope `result` string as JSON, and returns `RawOutputReference::Inline(stdout_text)` plus the parsed result JSON.
- [ ] Codex stage execution runs in `request.working_dir` via `tokio::process::Command` with `codex exec --dangerously-bypass-approvals-and-sandbox --skip-git-repo-check --model <model_id> --output-schema <schema_path> --output-last-message <message_path> -` for new sessions, or `codex exec resume --dangerously-bypass-approvals-and-sandbox --skip-git-repo-check --model <model_id> --output-schema <schema_path> --output-last-message <message_path> <session_id> -` when resuming. The adapter writes the schema to `<project_root>/runtime/temp/<invocation_id>.schema.json`, reads `<project_root>/runtime/temp/<invocation_id>.last-message.json` after a successful exit, parses it as JSON, and returns `RawOutputReference::Inline(last_message_text)` plus the parsed JSON payload.
- [ ] Successful Claude and Codex envelopes set `metadata.invocation_id`, `TokenCounts::default()`, cloned backend/model, `attempt_number`, and `Duration::from_millis(0)` inside the adapter. Session metadata must follow backend rules: Claude prefers the returned `session_id` and falls back to the reusable prior session when resuming; Codex sets `session_id` only when `codex exec resume` was used; `session_reused` is `true` only when a resume command actually ran.
- [ ] State invariant on process lifecycle: the adapter must not add a second timeout wrapper, must register each spawned child in `active_children` before awaiting it, and must remove that entry on every return path after spawn, including success, non-zero exit, parse failure, and cancellation races. If spawn fails before registration, the map remains unchanged.
- [ ] State invariant on error mapping: spawn failures and non-zero exits return `AppError::InvocationFailed { failure_class: FailureClass::TransportFailure, ... }`; invalid Claude envelope JSON, invalid Claude `result` JSON, and invalid Codex last-message JSON return `FailureClass::SchemaValidationFailure`. Transport failures must include stderr text when present. Cleanup errors must never overwrite the primary invocation error.
- [ ] State invariant on temp artifacts: Codex schema and last-message files are created only under `<project_root>/runtime/temp`, are deleted on a best-effort basis after successful reads and after failures that occur after file creation, and deletion failures do not change the success/error result already determined for the invocation.
- [ ] State invariant on cancellation: `cancel()` sends SIGTERM only to a currently registered child, treats “already exited” races as `Ok(())`, and leaves no stale `active_children` entry behind for that `invocation_id`. Session persistence remains unchanged on cancelled or failed invocations because only successful envelopes are recorded by `AgentExecutionService`.
- [ ] `tests/unit/process_backend_test.rs` uses fake `claude` and `codex` executables earlier on `PATH` and verifies at minimum: Claude command construction, Claude double-parse behavior, Claude `--resume` flag behavior, Codex command construction and temp-file contents, openrouter capability-mismatch detail text, cancellation delivering SIGTERM to a long-running fake child, and `active_children` cleanup after success and failure. Existing missing-binary coverage should remain green without calling real external services.
