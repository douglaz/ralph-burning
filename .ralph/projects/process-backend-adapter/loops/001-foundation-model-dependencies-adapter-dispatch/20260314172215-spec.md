# Feature: Foundation — Model, Dependencies, Adapter Dispatch, CLI Wiring, and Test Harness

## Description
Establish the foundational plumbing that all subsequent features depend on: add `working_dir` to `InvocationRequest`, enable the Tokio `process` feature, create the `BackendAdapter` dispatch enum, make `build_agent_execution_service()` configurable via `RALPH_BURNING_BACKEND`, create a skeleton `ProcessBackendAdapter` (with stub trait implementations that return `unimplemented!()` for `invoke`), update the test harness to inject `RALPH_BURNING_BACKEND=stub` by default, and increase the workflow stage timeout from 300s to 3600s. This feature produces a compiling, all-tests-green codebase with the dispatch infrastructure in place, so later features can fill in Claude execution, Codex execution, and prompt enrichment independently.

## Acceptance Criteria
- [ ] `InvocationRequest` has a new `working_dir: PathBuf` field
- [ ] All existing construction sites for `InvocationRequest` compile: `engine.rs` populates `working_dir` with `base_dir` (the workspace root passed into `execute_run`/`resume_run`); `requirements_drafting/service.rs` populates `working_dir` with `run_root` so code compiles without changing requirements behavior
- [ ] `Cargo.toml` includes `"process"` in the Tokio features list
- [ ] `src/adapters/process_backend.rs` exists with a `ProcessBackendAdapter` struct that:
  - has `ProcessBackendAdapter::new()` constructor
  - has an `active_children: Arc<tokio::sync::Mutex<HashMap<String, tokio::process::Child>>>` field for future cancellation support
  - implements `AgentExecutionPort` with:
    - `check_capability`: succeeds for `BackendFamily::Claude` and `BackendFamily::Codex` with `InvocationContract::Stage`; returns `AppError::CapabilityMismatch` for `OpenRouter`, `Stub`, or `InvocationContract::Requirements`
    - `check_availability`: verifies `claude`/`codex` binary exists on PATH via `tokio::process::Command::new("which").arg(binary_name)`; returns `AppError::BackendUnavailable` on missing binary
    - `invoke`: returns `AppError::InvocationFailed` with `FailureClass::TransportFailure` and message "ProcessBackendAdapter invoke not yet implemented" (placeholder for next features)
    - `cancel`: looks up child in active map, sends SIGTERM if found, returns `Ok(())` if not found
- [ ] `src/adapters/mod.rs` exports `pub mod process_backend;` and defines `pub enum BackendAdapter { Stub(StubBackendAdapter), Process(ProcessBackendAdapter) }` with `AgentExecutionPort` delegating to the inner variant
- [ ] `build_agent_execution_service()` in `src/cli/run.rs` returns `AppResult<AgentExecutionService<BackendAdapter, FsRawOutputStore, FsSessionStore>>`
- [ ] `build_agent_execution_service()` reads `RALPH_BURNING_BACKEND` env var: `"stub"` preserves all existing `RALPH_BURNING_TEST_*` seams; `"process"` constructs `BackendAdapter::Process(ProcessBackendAdapter::new())`; default is `"process"`; any other value returns a clear `AppError`
- [ ] All call sites in `src/cli/run.rs` (`handle_start` line 166, `handle_resume` line 263) and `src/cli/daemon.rs` (line 67) updated to use `?` on the now-fallible builder
- [ ] Workflow stage timeout in `engine.rs` changed from `Duration::from_secs(300)` to `Duration::from_secs(3600)`
- [ ] `tests/cli.rs`: every `Command::new(binary())` call that exercises `run start`, `run resume`, or `daemon start` has `.env("RALPH_BURNING_BACKEND", "stub")` added so existing tests keep using the stub adapter. - [ ] `nix develop -c cargo build` passes
