# Review: SUGGESTIONS

## Required Changes
1. **Zero-prefix append failure recovery**: Preserve a resumable planning state when the first `amendment_queued` append fails in a late-stage batch.
   - Current: In [engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L995), cleanup deletes the failed-and-later amendment files, but the failed snapshot is only updated when `last_journaled_amendment_index` is set at [engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L1015). If the first append fails, the snapshot keeps no pending amendments and no advanced completion round, so resume only rewinds to planning when `snapshot.amendment_queue.pending` is non-empty at [engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L2580). That means resume will advance past the conditional-approval stage, and after `final_review` it can even report “nothing to resume.”
   - Expected: A first-append failure must still leave enough durable state to resume into the next planning round without duplicating amendments. Cleaning disk to match the durable journal prefix is fine, but the failed snapshot still has to preserve the conditional-approval outcome so resume does not skip required amendments or terminate the run incorrectly.
   - Reference: Master Prompt `Required Outcome` (“Public CLI behavior must not change”) and `Acceptable Fix Strategies` / `Preferred Invariant`.

## Recommended Improvements

The new coverage in [workflow_engine_test.rs](/root/new-ralph-burning/ralph-burning-rewrite/tests/unit/workflow_engine_test.rs#L3866) only exercises the “first append succeeds, second append fails” case (`fail_after = 19`), so this zero-prefix failure path can currently regress unnoticed.
