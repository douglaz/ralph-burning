# Feature: Fix amendment journal orphan on mid-batch append failure

## Description
Fix the amendment persistence bug where a journal append failure during a batch of `amendment_queued` events leaves orphaned amendment files on disk, causing duplicate amendments after resume. Implements both Strategy A (cleanup unjournaled files on failure) and Strategy B (make reconciliation idempotent against the journal) for defense-in-depth correctness. Adds a journal append failpoint and deterministic tests for mid-batch failure, resume-after-failure, and full-batch-success scenarios.

## Acceptance Criteria
- [ ] **Strategy A — Cleanup on failure**: When journal append fails at amendment N in a batch, delete amendment files for the failed amendment and all later amendments that lack a successfully appended journal event. Do not delete files for already-journaled amendments.
- [ ] **Strategy B — Idempotent reconciliation**: `reconcile_amendments_from_disk` consults the durable journal to skip amendments that already have an `amendment_queued` event, preventing duplicate additions to the snapshot.
- [ ] **Journal append failpoint**: Add a deterministic failure injection mechanism for `FsJournalStore::append_event` (env var `RALPH_BURNING_TEST_JOURNAL_APPEND_FAIL_AFTER`, same pattern as the existing `RALPH_BURNING_TEST_AMENDMENT_WRITE_FAIL_AFTER` failpoint in `fs.rs:884`).
- [ ] **Test: mid-batch journal append failure**: Simulate a 3-amendment batch where the first `amendment_queued` append succeeds and the second fails. Verify that amendment files for the failed and subsequent amendments are cleaned up (Strategy A), and that only the first amendment's journal event exists.
- [ ] **Test: resume after partial failure**: From the post-failure state of the previous test, run the resume/reconciliation path. Verify no duplicate amendments in the recovered snapshot — each amendment appears at most once.
- [ ] **Test: full batch success (regression)**: Verify the normal success path persists all amendment files and journal events correctly. Assert amendment count and persisted state match expectations with no regressions.
- [ ] **Invariant**: After any append failure, the combination of on-disk amendment files plus journal replay represents each amendment zero or one times, never more.
- [ ] **Journal remains append-only**: No journal rewrites, compaction, or non-append mutations introduced.
- [ ] **Public CLI behavior unchanged**: No changes to public interfaces.
- [ ] **`nix develop -c cargo build` succeeds**.
- [ ] **`nix develop -c cargo test` succeeds** (all existing + new tests pass).
