# Implementation Notes

## Decisions Made
- Modified `build_requirements_service()` in `daemon_loop.rs` to accept `&EffectiveConfig` and return `AppResult<...>`, building `BackendSelectionConfig::from_effective_config()` and calling `.with_workspace_defaults()` — the exact same construction pattern the CLI uses in `cli/requirements.rs`.
- Updated both `handle_requirements_quick` and `handle_requirements_draft` signatures to accept `&EffectiveConfig`, threaded from the already-loaded config in `process_task` (line 368).
- If `build_requirements_service` (specifically `BackendSelectionConfig::from_effective_config`) fails before a requirements run is created, the task is marked failed with the path-specific failure class (`requirements_quick_failed` or `requirements_draft_failed`) and no run directory is created, satisfying the pre-run failure invariant.
- Added a recording seam (`RecordedInvocation` + `recorded_invocations` field) to `StubBackendAdapter` in `src/adapters/stub_backend.rs` so tests can observe the actual `ResolvedBackendTarget` passed into each invocation at runtime, rather than relying on source-string assertions.
- Regression tests build the `RequirementsService` using the same daemon construction pattern (load config → derive `BackendSelectionConfig` → apply via `.with_workspace_defaults()`), then run `quick()` / `draft()` and assert the recorded invocations match the expected backend family and model ID.
- Added coverage for full defaults, partial defaults (backend-only, model-only), and unset defaults (role fallback behavior).

## Spec Deviations
- Added `src/adapters/stub_backend.rs` as an additional modified file (not listed in the spec's "Files to Modify/Create") because the spec requires "a narrow recording seam or equivalent observable" for test assertions, and the stub adapter is the natural location for that seam.
