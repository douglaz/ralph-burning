# Feature: Daemon requirements dispatch honors workspace backend/model defaults

## Description
Loops 1, 2, 3, and 5 completed the prompt-scoped integrity work: overflow-safe `StageCursor` transitions, round-aware workflow history IDs, round-aware requirements history IDs, and completion-round resume integrity after a failed `completion_round_advanced` append. The git checkpoints on the current branch stop after loop 5, and the remaining uncovered final-review item is still present in `src/contexts/automation_runtime/daemon_loop.rs`: both daemon requirements entry points call a zero-argument `build_requirements_service()`, so daemon-driven `requirements quick` and `requirements draft` ignore workspace `default_backend` / `default_model` settings that the direct CLI requirements path already applies with `.with_workspace_defaults(...)`. The next feature is to make daemon requirements dispatch resolve backend/model targets exactly like the CLI path, without changing CLI surface area, task linkage semantics, or on-disk requirements layout.

## Acceptance Criteria
- [ ] `handle_requirements_quick` and `handle_requirements_draft` no longer construct a requirements service without workspace defaults; both paths use defaults derived from the daemon’s loaded `EffectiveConfig`.
- [ ] The daemon derives those defaults through `BackendSelectionConfig::from_effective_config(...)` or an equivalent shared helper, so daemon and CLI use the same backend/model precedence rules.
- [ ] The daemon requirements service calls `.with_workspace_defaults(...)` before invoking `quick()` or `draft()`.
- [ ] Full defaults behave identically to the CLI path: if `default_backend` and `default_model` are both set, daemon requirements invocations use that exact backend family and model ID.
- [ ] Partial defaults behave identically to the CLI path: `default_backend` alone selects that backend’s default model, and `default_model` alone overrides only the model on the role’s default backend.
- [ ] When workspace defaults are unset, daemon requirements behavior remains unchanged and still falls back to existing role defaults.
- [ ] Existing daemon behavior is otherwise unchanged: same stub adapter, same filesystem raw-output/session/requirements stores, same `RALPH_BURNING_TEST_LABEL_OVERRIDES` seam, and no CLI, schema, or storage-layout changes.
- [ ] Failure invariant: if workspace-default resolution or requirements-service construction fails before a requirements run is created, no requirements run directory/history is created, no `requirements_run_id` is linked, and the task is marked failed with the existing path-specific failure class (`requirements_quick_failed` or `requirements_draft_failed`).
- [ ] Failure invariant: if the requirements run succeeds and a later task-link or metadata write fails, the existing `requirements_linking_failed` behavior remains unchanged; the run stays addressable and only the task transitions to the existing failure state.
- [ ] Add automation-runtime regression coverage proving daemon-driven `requirements quick` resolves the same backend family and model ID as the direct CLI requirements path for a workspace with explicit defaults.
- [ ] Add automation-runtime regression coverage proving daemon-driven `requirements draft` resolves the same backend family and model ID as the direct CLI requirements path for a workspace with explicit defaults.
- [ ] The new tests verify the resolved target actually passed into requirements-stage invocation, using a narrow recording seam or equivalent observable, rather than a source-string assertion.
- [ ] `nix develop -c cargo build`, `nix develop -c cargo test`, and `nix develop -c ralph-burning conformance run` succeed after the change.
