# Implementation Notes

## Decisions Made
- Added `-cr{completion_round}` to the base ID format in `history_record_base_id()` by inserting `cursor.completion_round` into the format string, placed after `-a{attempt}` and before any `-rb{rollback_count}` suffix. This matches the convention already used by amendment IDs (`-cr{source_completion_round}-amd{idx}`) and follow-up IDs elsewhere in the engine.
- Updated the two rollback-test assertions that filtered on `-implementation-c1-a1` to `-implementation-c1-a1-cr1` using exact-match strengthening rather than weakening to a substring that omits the round.
- The new regression test targets the `planning` stage for round coexistence assertions because planning is the first stage re-entered on a completion-round restart, making `-planning-c1-a1-cr1` vs `-planning-c1-a1-cr2` the clearest signal.
- The regression test also validates that every payload and artifact filename in the history directories contains a `-cr` substring, ensuring no legacy-format IDs leak through.

## Spec Deviations
- None. All acceptance criteria are implemented exactly as specified.

## Testing
- `nix develop -c cargo build` — compiles cleanly.
- `nix develop -c cargo test` — all 413 tests pass (including the new `completion_round_restart_creates_distinct_round_aware_payload_artifact_files` test and the updated rollback assertions).
- `nix develop -c cargo run -- conformance run` — all 218 conformance scenarios pass.
- The new regression test drives a full 2-round standard run (CompletionPanel returns `conditionally_approved` then `approved`), then asserts:
  - Exactly one `-planning-c1-a1-cr1` payload and artifact exist (round 1).
  - Exactly one `-planning-c1-a1-cr2` payload and artifact exist (round 2).
  - The round-1 and round-2 filenames are distinct (no overwrite).
  - Every file in `history/payloads/` and `history/artifacts/` contains `-cr`.
