The StageCursor overflow hardening is the foundational change — it alters method signatures from `Self` to `AppResult<Self>`, which propagates to all call sites in the engine. This must come first before the history ID changes.

# Feature: StageCursor overflow-safe transitions

## Description
Harden `StageCursor::retry()`, `StageCursor::advance_cycle()`, and `StageCursor::advance_completion_round()` so that u32 counter increments use `checked_add(1)` and return `AppResult<Self>` instead of bare `Self`. Add a dedicated `AppError::StageCursorOverflow` variant for the error path. Update all call sites in the workflow composition engine (and anywhere else these methods are called) to propagate the new `AppResult` with `?`. Also fix two bare `+ 1` increments in the engine's resume and remediation paths that bypass `StageCursor` methods entirely.

## Acceptance Criteria
- [ ] `StageCursor::retry()` returns `AppResult<Self>` and uses `checked_add(1)` on `self.attempt`

- [ ] `StageCursor::advance_completion_round()` returns `AppResult<Self>` and uses `checked_add(1)` on `self.completion_round`
- [ ] All three transition methods route through `StageCursor::new(...)` after computing checked values, so the existing `> 0` validation remains centralized
