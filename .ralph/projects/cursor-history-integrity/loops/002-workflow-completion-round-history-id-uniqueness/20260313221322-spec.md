# Feature: Workflow completion-round history ID uniqueness

## Description
Loop 1 completed the `StageCursor` overflow hardening, and the checkpoint history shows no later loop implementing either history-ID fix. The current workflow engine still builds history base IDs as `{run_id}-{stage_id}-c{cycle}-a{attempt}` in `src/contexts/workflow_composition/engine.rs`, so a second completion round for the same stage/cycle/attempt can overwrite round-1 payload and artifact files. The next logical feature is to make workflow history IDs round-aware by adding `-cr{completion_round}` before any rollback suffix, while preserving the existing rollback branching behavior, artifact derivation, CLI surface, and storage layout. Requirements round-aware IDs remain a separate follow-up feature.

## Acceptance Criteria
- [ ] `history_record_base_id(...)` uses this exact base format before any rollback suffix: `{run_id}-{stage_id}-c{cycle}-a{attempt}-cr{completion_round}`.
- [ ] When `rollback_count == 0`, the payload ID is exactly the round-aware base ID with no extra suffix.
- [ ] When `rollback_count > 0`, the payload ID appends `-rb{rollback_count}` after `-cr{completion_round}`.
- [ ] Artifact IDs remain derived exactly as `{payload_id}-artifact`.
- [ ] No directory names, file extensions, or write locations change under `history/payloads/` or `history/artifacts/`.
- [ ] A completion-round restart of the same workflow stage/cycle/attempt creates a new round-2 payload/artifact pair instead of overwriting round 1.
- [ ] Regression coverage proves both round-1 and round-2 files exist simultaneously on disk and their IDs include `-cr1` and `-cr2`.
- [ ] Existing rollback branching still works on top of the new round-aware base, so rollback-resumed files remain distinct with `-rb{rollback_count}` appended after `-cr{completion_round}`.
- [ ] Failure invariant: if writing a new round-aware payload/artifact pair fails, previously committed round-1 files remain untouched and no partial round-2 pair survives.
- [ ] Failure invariant: if payload/artifact persistence succeeds but the subsequent `stage_completed` journal append fails, cleanup removes only the newly written round’s pair and leaves prior-round files intact.
- [ ] Existing workflow-engine assertions that currently match `-implementation-c1-a1` are updated to the explicit `-implementation-c1-a1-cr1` shape instead of weakened matching.
- [ ] `nix develop -c cargo build`, `nix develop -c cargo test`, and `nix develop -c ralph-burning conformance run` succeed after the change.

## Files to Modify/Create
- `src/contexts/workflow_composition/engine.rs` - update `history_record_base_id(...)` to include `cursor.completion_round` before rollback suffixes and keep artifact ID derivation unchanged.
- `tests/unit/workflow_engine_test.rs` - add a completion-round restart regression covering coexistence of round-1 and round-2 payload/artifact files, and update existing rollback/history filename assertions to the new `-cr{completion_round}` format.
