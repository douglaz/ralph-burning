# Implementation Notes

## Decisions Made
- Added `create_project_from_seed()` in the project service and extended `SeedHandoff` with `requirements_run_id` so seed-originated `ProjectCreated` events can durably record `source = "requirements"` and the originating run ID.
- Kept seed ingestion fully canonical: project `prompt.md` is always written from `SeedHandoff.prompt_body`, `prompt_reference` remains `"prompt.md"`, and prompt hashing is computed from the seed body at creation time.
- Implemented `project create --from-requirements` on the existing `project create` path, plus `project bootstrap` with `--idea` / `--from-file`, optional `--flow`, and optional `--start`.
- Reused the normal writer-lease + `engine::execute_run()` start path for bootstrap, but wrapped the pre-start failure case so a created project remains selected with `not_started` state and a retry message pointing to `ralph-burning run start`.

## Spec Deviations
- Implemented `project create --from-requirements <run-id>` as a flag on the existing `project create` subcommand instead of adding a separate `create-from-requirements` subcommand variant, so the user-facing CLI matches the acceptance criteria exactly.
- Left `bootstrap --flow` as an optional override instead of defaulting it to `"standard"`, because the slice invariants require flow override behavior only when the flag is explicitly provided; omitting `--flow` now correctly preserves the seed flow.
