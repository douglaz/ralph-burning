# Feature: Slice 2 — Bootstrap and Auto Parity

## Description
Loops 1 and 2 completed Slice 0 (P0 hardening) and Slice 1 (full requirements and PRD parity). The baseline now has staged full-mode requirements pipeline, quick-mode revision loop, versioned `ProjectSeed` (v2), `extract_seed_handoff()`, and atomic `create_project()`. What is missing is the glue: there is no CLI path to create a project directly from a completed requirements run, and no convenience bootstrap that chains quick requirements into project creation and optional run start. Users must manually copy seed fields into `project create --id --name --prompt --flow` arguments. This slice closes that gap by adding `project create --from-requirements`, `project bootstrap`, and transactional failure semantics that prevent partial project state.

## Acceptance Criteria
- [ ] `ralph-burning project create --from-requirements <run-id>` extracts the seed from a completed requirements run via `extract_seed_handoff()`, creates the project using seed-derived fields (project_id, project_name, flow, prompt_body), selects it as the active project, and prints the created project summary.
- [ ] `ralph-burning project bootstrap --idea "..." --flow <preset>` runs quick requirements inline, extracts the seed, creates the project, and selects it as active. The `--flow` flag overrides the seed's `recommended_flow` when explicitly provided.
- [ ] `ralph-burning project bootstrap --from-file <requirements-file> --flow <preset>` reads the requirements file as the idea input to quick requirements, then follows the same seed → project creation path as `--idea`.
- [ ] `ralph-burning project bootstrap --start` (combinable with `--idea` or `--from-file`) additionally starts a run on the newly created project after creation, equivalent to `ralph-burning run start` on the active project.
- [ ] Failure before project creation completes (requirements failure, seed extraction failure, seed version validation failure, `create_project_atomic` failure) leaves no project directory under `.ralph-burning/projects/` and no active-project pointer change.
- [ ] Failure after project creation but during `--start` (engine initialization failure, preflight failure) leaves a valid project with `not_started` run status and the project selected as active; the user sees a clear error indicating the project was created but the run failed to start, and can retry with `ralph-burning run start`.
- [ ] `create --from-requirements` validates that the referenced run exists, is completed, has a supported seed version, and has not already been consumed by another project with the same `project_id`; each validation failure produces a distinct, actionable error message.
- [ ] Seed-derived prompt is written to the project's canonical `prompt.md` from the seed's `prompt_body` field, not copied from the seed filesystem path; the `prompt_path` stored in `project.toml` is always `"prompt.md"` (the canonical in-project reference).
- [ ] Journal event for seed-originated projects records `source: "requirements"` and `requirements_run_id: "<run-id>"` in the `ProjectCreated` event details, distinguishing them from manually-created projects.
- [ ] Bootstrap with `quick_dev` flow produces a valid `quick_dev` project identical in structure to manual `project create --flow quick_dev`.
- [ ] Conformance scenarios with `parity_slice2_*` naming cover: create from requirements, bootstrap standard, bootstrap quick_dev, bootstrap with `--start`, bootstrap from file, failure before project creation, failure after creation but before run start, and duplicate project_id from same seed.
- [ ] A Slice 2 report records the exact legacy `auto` and `quick-dev-auto` references consulted, contract changes, tests run, results, and remaining gaps.

## State Invariants

**Seed ingestion atomicity:**
- `create_project_from_seed()` delegates to `create_project()` which uses `create_project_atomic()`. If any file within the atomic batch fails to write, the entire project directory is removed. No partial project.toml, run.json, or journal.ndjson can survive a failed creation.

**Active project pointer on failure:**
- If `create_project_from_seed()` fails, `set_active_project()` must not have been called yet. The handler must call `set_active_project()` only after `create_project_from_seed()` succeeds. This means: on creation failure, the active project pointer is unchanged.
- If `set_active_project()` itself fails after a successful project creation, the project exists but is not selected. The error message must indicate the project was created and the user should manually select it.

**Bootstrap --start failure semantics:**
- If `--start` is requested and project creation succeeds but `engine::execute_run()` fails (preflight, backend unavailability, writer lease conflict), the project remains in `not_started` status with a valid `run.json`. The active project pointer has already been set to this project. The error message must clearly state: "Project '<id>' created successfully but run failed to start: <reason>. Use `ralph-burning run start` to retry."
- The writer lease acquired for `--start` must be released on failure. If the lease guard's drop implementation handles this, no explicit cleanup is needed; otherwise, ensure the guard is dropped before the error is returned.

**Flow override precedence:**
- If `--flow` is explicitly provided by the user, it overrides the seed's `flow` field and the seed's `recommended_flow`. If `--flow` is not provided, the seed's `flow` field is used (this is the flow selected during requirements drafting). `recommended_flow` is informational and logged but never silently applied as an override.

**Prompt provenance:**
- The project's `prompt.md` is always written from `SeedHandoff.prompt_body` (the canonical prompt text from the seed payload), never by copying the file at `SeedHandoff.prompt_path`. This ensures the prompt is self-contained within the project directory and does not depend on the requirements run's filesystem state after creation.
