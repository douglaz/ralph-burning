# Feature: Prompt and Template Override Parity

## Description
Approved loop artifacts show Slices 0-6 are already complete: P0 hardening, full requirements/PRD parity, bootstrap/auto parity, manual amendments, operator UX, backend diagnostics, and `tmux`/streaming. The remaining missing slice is prompt/template override parity: the current repo still hardcodes prompt assembly in workflow and requirements code, and there is no shared catalog, override resolution, or Slice 7 implementation in the checkpoint history. Implement a shared template catalog that resolves built-in defaults, workspace overrides, and project overrides for every workflow and requirements prompt surface, using validated placeholder manifests so operators can change wording/layout without changing CLI contracts, bypassing schema blocks, or corrupting canonical state.

## Acceptance Criteria
- [ ] Freeze template IDs for all prompt surfaces: workflow stage templates use existing `StageId::as_str()` names; panel templates use `prompt_review_refiner`, `prompt_review_validator`, `completion_panel_completer`, `final_review_reviewer`, `final_review_voter`, and `final_review_arbiter`; requirements templates use `requirements_draft`, `requirements_review`, `requirements_question_set`, `requirements_project_seed`, `requirements_ideation`, `requirements_research`, `requirements_synthesis`, `requirements_implementation_spec`, `requirements_gap_analysis`, and `requirements_validation`.
- [ ] Resolution order is deterministic and singular: `.ralph-burning/projects/<project-id>/templates/<template-id>.md` overrides `.ralph-burning/templates/<template-id>.md`, which overrides the built-in default template; layers are never merged.
- [ ] Workflow prompt generation uses the catalog for standard stage prompts and for prompt-review, completion, and final-review panel prompts, with no new CLI flags or command changes.
- [ ] Requirements prompt generation uses the same catalog for quick draft/review, quick revisions, full-mode stages, question rounds, and project-seed generation.
- [ ] Built-in defaults preserve current behavior when no override exists; generated prompts remain structurally equivalent to the current builders, including authoritative JSON schema blocks and existing context sections.
- [ ] Override files are plain UTF-8 markdown templates over pre-rendered blocks only; unknown placeholders, missing required placeholders, unreadable files, and non-UTF-8 contents are all rejected as malformed overrides.
- [ ] Failure invariants: if the selected override is malformed, the command fails before backend invocation for that contract, no new payload/artifact/journal entry is written for that contract, no prompt file is mutated, and prior durable history remains unchanged; if a project override exists but is malformed, the system must not silently fall back to workspace or built-in defaults.
- [ ] Conformance covers workspace override, project override, project-over-workspace precedence, malformed override rejection on a workflow prompt, and malformed override rejection on a requirements prompt.
