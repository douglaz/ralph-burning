# Feature: Slice 0 ‚Äî P0 Hardening and Sign-off

## Description
Close correctness and hardening gaps in the current P0 implementation before adding new product surface. Three concrete defects exist in the baseline code, plus conformance coverage is missing for all Slice 0 acceptance criteria. This slice fixes the defects, adds the missing conformance scenarios, and verifies that existing P0 conformance remains green.

## Acceptance Criteria
- [ ] Backend availability checks require executable permission, not just path existence (`process_backend.rs:596-599` currently uses `metadata.is_file()` without checking the executable bit)
- [ ] Cancel and timeout handling uses in-process signal APIs (POSIX `libc::kill`) instead of shelling out to the `kill` binary (`process_backend.rs:674-689` currently calls `std::process::Command::new("kill")`)
- [ ] Resolution snapshots and backend-drift detection include the final-review planner target (currently `StageResolutionSnapshot` has no `final_review_planner` field; `FinalReviewPanelResolution` has no `planner` field; `build_final_review_snapshot()` does not accept or record a planner target)
- [ ] Panel preflight validates the final-review planner alongside reviewers and arbiter (engine.rs line ~6041 already checks planner availability at invocation time, but this must be part of the snapshot-based drift detection path)
- [ ] Permission-check failure surfaces a clear `BackendUnavailable` error with actionable detail
- [ ] Cancel or timeout leaves no orphan child processes (existing SIGTERM ‚Üí grace ‚Üí SIGKILL ‚Üí background-reap logic is correct; only the signal delivery mechanism changes)
- [ ] Final-review planner drift is reported on resume when the planner's resolved backend changes between suspend and resume
- [ ] Compare URL generation works for refs with reserved characters (already implemented at `github.rs:762-768` ‚Äî add conformance coverage to lock this in)
- [ ] Conformance scenarios exist for each acceptance criterion above, using `parity_slice0_*` naming
- [ ] Existing P0 conformance suite remains green after all changes

## State Invariants

**Executable permission check failure:**
- If a binary is found on PATH but lacks executable permission, `check_availability` must return `Err(AppError::BackendUnavailable)` with details including the path and the fact it is not executable. No partial state is modified.

**Signal delivery failure (ESRCH):**
- If `libc::kill()` returns `ESRCH` (process already exited), treat as success ‚Äî the child is already gone. This matches the existing `"No such process"` handling.
- If `libc::kill()` returns any other error, propagate as `std::io::Error`. The background reaper path (`spawn_background_reap`) remains unchanged: it escalates to SIGKILL after the grace period.

**Planner drift detection:**
- If only the final-review planner's backend has drifted but reviewers and arbiter are unchanged, drift must still be reported. Planner drift and reviewer/arbiter drift are independent failure paths and must be evaluated independently.
- If drift detection itself fails (e.g., corrupt snapshot), the resume must abort with a clear error rather than silently proceeding with a potentially stale backend.

**Snapshot backwards compatibility:**
- The new `final_review_planner` field uses `#[serde(default, skip_serializing_if = "Option::is_none")]`, so existing persisted snapshots without this field will deserialize with `None` and will not trigger false drift reports. Drift detection must treat `None` (no prior planner snapshot) as "no baseline to compare" and skip planner drift checking for that resume.
