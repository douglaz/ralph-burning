# Feature: Slice 4 — Operator UX Parity

## Description
Restore daily operator inspection and rollback ergonomics by adding JSON output modes, verbose history, tail pagination and follow, payload/artifact inspection by ID, rollback target discovery, and stage-aware filtering to the existing `run` CLI surface. All new commands and flags are read-only queries over canonical state and durable history — no mutations.

## Acceptance Criteria
- [ ] `run status --json` emits a stable, documented JSON object with project_id, status, stage, cycle, completion_round, summary, and amendment_queue_depth fields
- [ ] `run history --verbose` prints full event details (JSON details field), full payload metadata, and artifact content previews (truncated to first 120 chars with ellipsis)
- [ ] `run history --json` emits a single JSON object with events, payloads, and artifacts arrays; combinable with `--verbose` to include full payload JSON and artifact content
- [ ] `run history --stage <stage>` filters events, payloads, and artifacts to a single stage
- [ ] `run tail --last <n>` limits output to the most recent N journal events and their associated payloads/artifacts
- [ ] `run tail --follow` polls canonical state and runtime logs at 2-second intervals, printing new events as they appear, until interrupted with Ctrl-C
- [ ] `run tail --follow --logs` includes new runtime log entries in follow output
- [ ] `run show-payload <payload-id>` prints full payload JSON content; fails with clear error for unknown IDs
- [ ] `run show-artifact <artifact-id>` prints full artifact content (rendered markdown); fails with clear error for unknown IDs
- [ ] `run rollback --list` enumerates visible rollback targets with rollback_id, stage, cycle, created_at, and optional git_sha
- [ ] All JSON outputs are stable, script-friendly, and documented in CLI help text
- [ ] Conformance scenarios cover each new flag and subcommand
- [ ] Existing run_queries conformance (SC-RUN-001 through SC-RUN-028) remains green

## Implementation Notes

### JSON Output Schema — `run status --json`
```json
{
  "project_id": "string",
  "status": "not_started | running | paused | completed | failed",
  "stage": "string | null",
  "cycle": "u32 | null",
  "completion_round": "u32 | null",
  "summary": "string",
  "amendment_queue_depth": "usize"
}
```

### JSON Output Schema — `run history --json`
```json
{
  "project_id": "string",
  "events": [{ "sequence": "u64", "timestamp": "ISO8601", "event_type": "string", "details": {} }],
  "payloads": [{ "payload_id": "string", "stage_id": "string", "cycle": "u32", "attempt": "u32", "created_at": "ISO8601", "record_kind": "string", "producer": "string | null", "completion_round": "u32" }],
  "artifacts": [{ "artifact_id": "string", "payload_id": "string", "stage_id": "string", "created_at": "ISO8601", "record_kind": "string", "producer": "string | null", "completion_round": "u32" }]
}
```
When `--verbose` is combined with `--json`: payload objects include the full `"payload": {}` field and artifact objects include the full `"content": "string"` field. Without `--verbose`, these heavy fields are omitted from JSON output to keep responses compact.

### JSON Output Schema — `run rollback --list`
```json
[
  {
    "rollback_id": "string",
    "stage_id": "string",
    "cycle": "u32",
    "created_at": "ISO8601",
    "git_sha": "string | null"
  }
]
```

### Stage-Aware Filtering
`--stage <stage>` accepts the same stage names used elsewhere in the CLI (e.g., `planning`, `implementation`, `review`, `qa`, `completion_panel`, `final_review`). Filtering is applied after rollback-boundary resolution and history-record filtering. Events are filtered by matching `stage_id` in their `details` JSON field (for `stage_entered`, `stage_completed`, `stage_failed` events). Payloads and artifacts are filtered by their `stage_id` field. Journal events without a stage association (e.g., `project_created`, `run_started`, `cycle_advanced`) are excluded by stage filters.

### Tail Follow Mode
`--follow` uses a polling loop with 2-second sleep intervals. Each iteration:
1. Re-reads the journal and compares against the last-seen sequence number
2. Prints any new events since last poll
3. If `--logs` is active, also polls runtime logs for new entries
4. Exits cleanly on SIGINT (Ctrl-C) via `tokio::signal::ctrl_c()`

The follow loop does NOT hold a file lock; it re-reads on each poll. This is safe because journals are append-only and runtime logs are best-effort.

### Tail Last-N
`--last <n>` takes the last N visible journal events (after rollback-boundary filtering) and includes only the payloads and artifacts associated with those events. If fewer than N events exist, all events are shown. `--last` and `--follow` are mutually exclusive (clap conflicts_with).

### Payload/Artifact Inspection
`show-payload` and `show-artifact` resolve through the full visibility pipeline: read journal, apply rollback boundaries, filter records, then find by ID. This ensures that rolled-back records are not inspectable, maintaining consistency with `run history`. The payload content is printed as pretty-printed JSON. The artifact content is printed as-is (rendered markdown).

### Rollback List
`rollback --list` is a new flag on the existing `Rollback` subcommand. When `--list` is present, the `--to` argument is not required. The listing uses the existing `list_rollback_points()` service function which already filters by visible journal events. Text output shows a table; `--list` does not need `--json` in this slice (it can be added later).

### State Invariants
- All new commands are read-only: no mutations to run.json, journal, or any durable state
- Follow mode must not interfere with concurrent writes by the engine or daemon
- Rolled-back payloads and artifacts must never be visible through show-payload/show-artifact
- JSON output must be parseable by standard tools (jq) — no trailing commas, no comments
