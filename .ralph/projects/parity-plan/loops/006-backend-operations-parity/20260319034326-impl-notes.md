# Implementation Notes

## Decisions Made
- The diagnostics service (`BackendDiagnosticsService`) is constructed directly from `&EffectiveConfig` rather than going through the composition layer, since it is purely read-only and does not need backend adapters, session stores, or output stores.
- `backend check` validates config-level enablement and panel resolution only ‚Äî no live transport-level availability checks, preserving the strictly read-only requirement.
- Panel probe supports three synthetic targets (`completion_panel`, `final_review_panel`, `prompt_review_panel`) plus all 10 singular `BackendPolicyRole` values.
- CLI override flags reuse the same `CliBackendOverrides`/`BackendSelection::from_backend_name` parsing as `run start`/`run resume`.
- Added `BackendPolicyService::backend_enabled_public()` as a thin public wrapper rather than exposing the private method.

## Spec Deviations
- `workspace_governance/config.rs` did not need new helpers ‚Äî existing `ConfigEntry`/`ConfigValueSource` with `EffectiveConfig::get()` was sufficient for source-precedence reporting.
- `composition/agent_execution_builder.rs` was not modified ‚Äî the diagnostics service is constructed directly from `EffectiveConfig` at the CLI handler level.
- `backend check` accepts but does not vary by flow preset since backend enablement is flow-independent.
