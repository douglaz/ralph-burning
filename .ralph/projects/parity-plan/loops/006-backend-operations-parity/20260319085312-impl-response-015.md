# Implementation Response (Iteration 15)

## Changes Made
1. **`backend check` now uses the runtime completion-panel resolution path when `completion.backends` is implicit**: Fixed `check_completion_panel_config()` in `diagnostics.rs` to check `config.completion_backends_are_explicit()` first. When implicit (no explicit `completion.backends` in workspace or project config), the check validates via `resolve_role_target(BackendPolicyRole::Completer, 1)` тАФ the same path that `default_completion_targets()` uses at runtime тАФ instead of iterating the built-in default backend list `[claude(required), codex(required)]`. This prevents false failures on default-list backends that runtime would never use. Before: with `default_backend=claude`, claude disabled, all roles overridden to codex, and openrouter enabled, `backend check` failed on `completion_panel.member[0] / claude` from `completion.backends`. After: check passes because the Completer role resolves to openrouter (opposite of codex, falling through from disabled claude) тАФ matching probe and runtime behavior.

2. **`backend check` now validates final review based on flow stage plan, not `final_review.enabled`**: Removed the `self.config.final_review_policy().enabled` gate from both `check_backends()` and `check_backends_with_availability()` in `diagnostics.rs`. The engine's `stage_plan_for_flow()` does NOT filter `FinalReview` based on `final_review.enabled` тАФ it always includes FinalReview when the flow definition contains it. Diagnostics now matches this behavior. Before: with `final_review.enabled=false` and broken final-review backends on a standard flow, `backend check` returned `{ "passed": true }`. After: check correctly reports the final-review failures because the engine would still execute that stage.

3. **Four regression tests added** (72 total backend diagnostics unit tests):
   - `check_passes_with_implicit_completion_backends_when_runtime_targets_are_available` тАФ verifies check/probe alignment when implicit completion targets resolve via opposite-family fallback
   - `check_fails_with_implicit_completion_backends_when_completer_role_unavailable` тАФ verifies check correctly fails when the Completer role's implicit resolution target is disabled
   - `check_validates_final_review_even_when_final_review_enabled_is_false` тАФ verifies check validates final review on standard flows regardless of the enabled flag
   - `check_still_skips_final_review_for_flows_without_final_review_stage` тАФ verifies flows like docs_change that lack FinalReview in their stage definition still skip final review validation
