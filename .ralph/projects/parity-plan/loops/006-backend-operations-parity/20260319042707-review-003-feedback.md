# Review: SUGGESTIONS

## Required Changes
1. **Adapter readiness is being masked instead of diagnosed**: `backend check` can return a clean pass even when the production adapter cannot be constructed, so it does not reliably validate the same readiness path that execution uses.
   - Current: [`handle_check`](/root/new-ralph-burning/ralph-burning-rewrite/src/cli/backend.rs#L184) falls back to config-only [`check_backends`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/agent_execution/diagnostics.rs#L179) whenever [`build_backend_adapter`](/root/new-ralph-burning/ralph-burning-rewrite/src/composition/agent_execution_builder.rs#L50) returns an error. Direct repro: `RALPH_BURNING_BACKEND=bogus ... backend check --json` returns `{"passed":true,"failures":[]}` and exits `0`.
   - Expected: If the diagnostics command cannot construct the same adapter stack that `run` would use, it should fail non-zero and surface that readiness error instead of silently skipping availability validation. The same issue exists in [`handle_probe`](/root/new-ralph-burning/ralph-burning-rewrite/src/cli/backend.rs#L224).
   - Reference: [backend.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/cli/backend.rs#L190), [agent_execution_builder.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/composition/agent_execution_builder.rs#L50), [20260319032842-spec.md](/root/new-ralph-burning/.ralph/projects/parity-plan/loops/006-backend-operations-parity/20260319032842-spec.md#L26), [prompt.md](/root/new-ralph-burning/.ralph/projects/parity-plan/prompt.md#L230)

2. **`backend probe` still treats required unavailable panel targets as success**: the availability-aware probe path only filters optional members and never fails on unavailable required members, planner/refiner targets, or the final-review arbiter.
   - Current: [`probe_with_availability`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/agent_execution/diagnostics.rs#L661) only iterates `panel.members`, explicitly keeps required unavailable members in the success payload, and does not check `result.target` or `panel.arbiter`. Direct repro: `PATH=/tmp ... backend probe --role final_review_panel --flow standard --json` exits `0` and returns a resolved panel even though runtime preflight checks planner, arbiter, and reviewers before execution.
   - Expected: Probe should use the same required-vs-optional availability semantics as execution: optional unavailable members move to `omitted`, but unavailable required members, the prompt-review refiner, the final-review planner, and the final-review arbiter must fail the probe with exact member identity and source field.
   - Reference: [diagnostics.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/agent_execution/diagnostics.rs#L661), [engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L6111), [engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L6296), [20260319032842-spec.md](/root/new-ralph-burning/.ralph/projects/parity-plan/loops/006-backend-operations-parity/20260319032842-spec.md#L29), [20260319032842-spec.md](/root/new-ralph-burning/.ralph/projects/parity-plan/loops/006-backend-operations-parity/20260319032842-spec.md#L31)

3. **Availability aggregation drops exact panel-member failures**: `backend check` deduplicates failures by backend family/model, so it does not actually report all blocking required panel members or preserve their identities.
   - Current: [`check_backends_with_availability`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/agent_execution/diagnostics.rs#L294) keys targets on `family:model` in [`add_target`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/agent_execution/diagnostics.rs#L307), keeping only the first alias for a shared target. With `PATH=/tmp`, `backend check --json` reports only `planner` and `implementer`, while required blockers such as completion members, prompt-review refiner/validators, and final-review reviewers/arbiter are omitted from the failure list.
   - Expected: Aggregate failures per blocking role/member, or preserve every role/member alias for a failing target, so `backend check` truly evaluates required panel members and surfaces the exact failing identity/config field the spec asks for.
   - Reference: [diagnostics.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/agent_execution/diagnostics.rs#L301), [diagnostics.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/agent_execution/diagnostics.rs#L335), [20260319032842-spec.md](/root/new-ralph-burning/.ralph/projects/parity-plan/loops/006-backend-operations-parity/20260319032842-spec.md#L26), [20260319032842-spec.md](/root/new-ralph-burning/.ralph/projects/parity-plan/loops/006-backend-operations-parity/20260319032842-spec.md#L31)
