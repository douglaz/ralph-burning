# Implementation Response (Iteration 16)

## Changes Made
1. **Opposite-family role attribution now reflects the runtime resolution path**: Rewrote `family_for_role()` in `diagnostics.rs` to mirror the resolution logic in `BackendPolicyService::resolve_role_target()`. Roles with explicit overrides still report the override's family. Roles that use planner-family resolution (Planner, Reviewer, PromptReviewer, PromptValidator, FinalReviewer, Arbiter) report `base_backend.family`. Opposite-family roles (Implementer, Qa, AcceptanceQa, Completer) now call `opposite_family(planner_family)` to determine the attempted family; when no opposite family is enabled, they report `opposite_of(<base>)` instead of incorrectly claiming the base backend. This fix affects `show-effective` (broken role `backend_family` field), `check` (failure `backend_family` field), and `probe` (error identity in `make_probe_target_error`). Added helper `role_uses_opposite_family()` to classify roles by their resolution path.

2. **Implicit completion-panel availability failures now report the correct config source**: Added `completion_panel_config_source()` helper that returns `"completion.backends"` when backends are explicitly configured, or `config_source_for_role(Completer)` (typically `"default_backend"`) when implicit. Changed `panel_members_config_source()` from a static method to an instance method so it can check config state. Updated three call sites: `check_backends_with_availability()` (line 336), `probe_completion_panel()` (line 846), and `probe_with_availability()` (line 1065). Previously, all three hard-coded `"completion.backends"` even when backends came from implicit Completer-role resolution.

3. **6 new regression tests** (78 total backend diagnostics unit tests):
   - `show_effective_opposite_family_roles_report_attempted_family_not_base` ŌĆö verifies implementer/qa/completer report the attempted opposite family (not base) when resolution fails
   - `show_effective_opposite_family_roles_report_resolved_family_when_available` ŌĆö verifies successful opposite-family resolution reports the correct family (codex, not claude)
   - `check_reports_opposite_family_not_base_for_failed_opposite_roles` ŌĆö verifies `backend check` failure entries use the opposite family
   - `probe_singular_opposite_role_reports_correct_family` ŌĆö verifies `backend probe --role implementer` error identifies opposite-family resolution, not the base backend
   - `check_with_availability_implicit_completion_reports_correct_source` ŌĆö verifies availability failures for implicit completion members don't report `completion.backends`
   - `probe_with_availability_implicit_completion_reports_correct_source` ŌĆö verifies probe availability failures for implicit completion don't reference `completion.backends`

4. **Docs and slice report updated**: `cli-reference.md` documents opposite-family role behavior in `show-effective` and implicit completion source behavior in `check`. `slice-5.md` test counts updated to 78 unit / 19 CLI tests with gap notes for both fixes.
