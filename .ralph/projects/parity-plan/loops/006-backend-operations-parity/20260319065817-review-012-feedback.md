# Review: SUGGESTIONS

## Required Changes
1. **Availability-time panel member identity is not exact after optional members are omitted**: required panel failures can still report the wrong `member[N]` when a disabled optional entry appears earlier in the configured panel list.
   Current: [`check_panel_availability()`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/agent_execution/diagnostics.rs#L436) assigns `member[N]` from `members.iter().enumerate()`, and [`probe_with_availability()`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/agent_execution/diagnostics.rs#L1017) assigns `member[N]` from `panel.members.drain(..).enumerate()`, but [`build_panel_member_views()`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/agent_execution/diagnostics.rs#L1189) has already dropped disabled optional specs. Direct repro on `HEAD`: with `prompt_review.validator_backends = ["?openrouter", "claude"]`, disabled `openrouter`, and no `claude` binary on `PATH`, `backend probe --role prompt_review_panel --flow standard` reports `prompt_review_panel.validator[0]`, even though the failing configured member is index `1`.
   Expected: carry the original configured-spec index through availability checks so `backend check` and `backend probe` always report the exact configured panel member identity, even after optional members are omitted.
   Reference: Slice 5 acceptance criteria requiring exact role/panel-member failure identity and exact required-member vs minimum-violation reporting.

2. **`backend list` does not report compile-time-only backends consistently with the current build**: `stub` is always marked `compile_only`, even in a build where the stub backend is actually available.
   Current: [`list_backends()`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/agent_execution/diagnostics.rs#L170) hard-codes `compile_only = Some(true)` for `BackendFamily::Stub` at [`diagnostics.rs`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/agent_execution/diagnostics.rs#L183). On this `test-stub` build, `RALPH_BURNING_BACKEND=stub` successfully runs `backend check --json`, so the emitted `compile_only: true` is not consistent with the current build.
   Expected: make `compile_only` build-sensitive, so `stub` is only marked compile-time-only when the current binary was built without stub support.
   Reference: Slice 5 acceptance criterion requiring `backend list` to report compile-time-only backends such as `stub` consistently with the current build.
