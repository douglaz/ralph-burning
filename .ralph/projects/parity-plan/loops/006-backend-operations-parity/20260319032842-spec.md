# Feature: Backend Operations Parity

## Description
Loop artifacts and slice reports show Slices 0 through 4 are complete: P0 hardening, staged requirements/PRD parity, project bootstrap/create-from-requirements, manual amendments, and operator run inspection. Git checkpoints through `5dfeac8` only land Slice 4 `run` query work, and the current CLI still has no `backend` top-level command, so the next missing slice is Slice 5.

This feature should add a read-only backend diagnostics surface that reuses the existing backend primitives already in baseline code (`EffectiveConfig`, `BackendResolver`, `BackendPolicyService`, process/OpenRouter adapters) instead of introducing parallel resolution logic. The new surface is `ralph-burning backend list`, `backend check`, `backend show-effective`, and `backend probe --role <role> --flow <flow> [--cycle <n>]`, with stable `--json` DTOs for script use. `backend check` must aggregate readiness failures without mutating project/run state. `backend show-effective` must expose resolved backend, model, timeout, session policy, and source precedence. `backend probe` must support both single-role targets and synthetic panel targets needed to inspect completion and final-review behavior, and it must distinguish optional-member omission from required-member failure.

## Acceptance Criteria
- [ ] Add a new top-level `backend` CLI command wired into the existing CLI tree with subcommands `list`, `check`, `show-effective`, and `probe`.
- [ ] `backend list` shows each supported backend family, whether it is enabled in the effective config, and the transport/readiness mechanism that will be used; compile-time-only backends such as `stub` must be reported consistently with the current build.
- [ ] `backend check` evaluates only effectively required backends and panel members for the active project/workspace scope, aggregates all blocking failures in one run, and exits non-zero if any required backend or panel minimum cannot be satisfied.
- [ ] `backend check` is strictly read-only: it must not create or modify run snapshots, project state, journals, payloads, artifacts, sessions, runtime logs, or active-run markers, even when checks fail partway through.
- [ ] `backend show-effective` exposes stable text and `--json` views for base backend selection, per-role overrides, resolved model IDs, timeout seconds, session reuse policy, and per-field source precedence across default, workspace, project, and CLI override layers.
- [ ] `backend probe --role <role> --flow <flow> [--cycle <n>]` uses the same resolution paths as run execution and supports both singular policy roles and synthetic panel targets at minimum for `completion_panel` and `final_review_panel`.
- [ ] Optional panel members that are disabled or unavailable are reported as omitted and do not fail the command unless their omission causes the configured panel minimum to be unsatisfied; that failure must be reported as the exact minimum-violation or required-member violation, not as a generic fallback error.
- [ ] Required panel failures identify the exact role/panel member, backend family, and effective config source field that caused the probe or check to fail.
- [ ] CLI override simulation for backend selection must be available on diagnostics commands that resolve targets, using the same override flags already accepted by `run start` and `run resume`, so source precedence is inspectable outside a live run.
- [ ] Add targeted unit tests for precedence/source reporting, aggregate readiness evaluation, and required-vs-optional panel semantics.
- [ ] Add CLI tests for each subcommand’s human-readable output, `--json` contract, and non-zero exit behavior.
- [ ] Add conformance scenarios `parity_slice5_backend_list`, `parity_slice5_backend_check`, `parity_slice5_backend_show_effective`, `parity_slice5_backend_probe_completion_panel`, and `parity_slice5_backend_probe_final_review_panel`.
- [ ] Update operator docs and add a Slice 5 report capturing legacy references, contracts changed, tests run, and remaining known gaps.
