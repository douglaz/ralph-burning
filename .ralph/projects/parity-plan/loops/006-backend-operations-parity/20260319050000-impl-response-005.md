# Implementation Response (Iteration 5)

## Changes Made
1. **Effective-required scope in `backend check`**: Rewrote `check_backends()` and `check_backends_with_availability()` in `diagnostics.rs` to evaluate only effectively required backends:
   - Added `effectively_required_stage_roles()` which filters `roles_for_flow()` to exclude `Completer` and `FinalReviewer` — these panel-specific roles are covered by dedicated panel checks below and should not be validated via the generic stage-derived path.
   - `default_backend` is now checked only when at least one effectively-required stage role lacks an explicit override (via `BackendPolicyService::has_explicit_override()`). When all roles in the active flow have explicit overrides, a disabled `default_backend` no longer produces a false failure.
   - Added `has_explicit_override()` public method to `BackendPolicyService` in `policy.rs` that exposes the existing `selection_for_role` check.
   - Direct repro fix: a `docs_change` project with all workflow roles explicitly overridden now passes `backend check` even when `default_backend` is disabled. A `standard` project with explicit `final_review.backends`/`arbiter_backend` and all stage roles overridden also passes.

2. **`probe_primary_config_source` for `prompt_review_panel`**: Fixed the source field mapping so `prompt_review_panel` maps to `BackendPolicyRole::PromptReviewer` (resolving to `prompt_review.refiner_backend`) instead of `BackendPolicyRole::Planner` (`workflow.planner_backend`). This matches the actual primary target exposed by `probe_prompt_review_panel`, which is the refiner.
