# Implementation Response (Iteration 3)

## Changes Made
1. **Adapter readiness is being masked instead of diagnosed**: `handle_check` now calls `check_backends_with_adapter_failure()` when `build_backend_adapter()` returns an error, surfacing the construction failure as an `availability_failure` entry instead of silently falling back to config-only checks. `handle_probe` now returns a `BackendUnavailable` error when the adapter can't be constructed, so `backend probe` exits non-zero instead of silently using config-only resolution.

2. **`backend probe` treats required unavailable panel targets as success**: Rewrote `probe_with_availability()` to: (a) check the primary/planner target and fail if unavailable; (b) check the arbiter target and fail if unavailable; (c) fail on required unavailable panel members with exact member identity; (d) move optional unavailable members to `omitted`; (e) check panel minimum satisfaction after filtering.

3. **Availability aggregation drops exact panel-member failures**: Removed the `family:model` deduplication in `check_backends_with_availability()`. Every role/member now gets its own availability check with distinct identity in the failure report (e.g., `final_review_panel.planner`, `final_review_panel.reviewer[0]`, `final_review_panel.arbiter`). Panel planners and arbiters are now included in the target list alongside reviewers/members.
