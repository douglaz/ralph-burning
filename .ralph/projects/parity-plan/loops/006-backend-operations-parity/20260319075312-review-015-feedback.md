## Required Changes
1. **`backend check` is not using the runtime completion-panel resolution path when `completion.backends` is implicit**: it can fail on backends the run would never use.
   - Current: [diagnostics.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/agent_execution/diagnostics.rs#L249) always routes completion validation through [check_completion_panel_config()](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/agent_execution/diagnostics.rs#L497), which iterates [effective `completion_policy().backends`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/agent_execution/diagnostics.rs#L501). Runtime panel resolution does not do that when the list is implicit: [policy.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/agent_execution/policy.rs#L88) switches to [default_completion_targets()](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/agent_execution/policy.rs#L94). Direct repro on `HEAD`: with `RALPH_BURNING_BACKEND=stub`, `default_backend=claude`, `claude` disabled, all stage roles/final review overridden to `codex`, and no explicit `completion.backends`, `backend probe --role completion_panel --flow standard --json` succeeds with two `codex` members, but `backend check --json` exits non-zero on `completion_panel.member[0]` / `claude` from `completion.backends`.
   - Expected: when `completion.backends` is not explicitly configured, `backend check` must validate the same resolved completer targets that execution/probe use, and report the real selecting source instead of treating the built-in default list as required panel members.
   - Reference: Slice 5 acceptance criteria for `backend check` evaluating only effectively required backends/panel members and reusing existing resolution primitives in the Feature Description ([20260319032842-spec.md](/root/new-ralph-burning/.ralph/projects/parity-plan/loops/006-backend-operations-parity/20260319032842-spec.md#L21), [20260319032842-spec.md](/root/new-ralph-burning/.ralph/projects/parity-plan/loops/006-backend-operations-parity/20260319032842-spec.md#L26)).

2. **`backend check` skips final review based on `final_review.enabled`, but the workflow stage plan still includes final review**: diagnostics can report a passing setup for a flow the engine will still try to execute.
   - Current: [diagnostics.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/agent_execution/diagnostics.rs#L253) and [diagnostics.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/agent_execution/diagnostics.rs#L352) gate final-review checks on `self.config.final_review_policy().enabled`. The actual stage plan still contains `FinalReview` for standard/quick-dev flows in [mod.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/mod.rs#L41) and [mod.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/mod.rs#L52), and [engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L837) builds the run plan without consulting `final_review.enabled`. Direct repro on `HEAD`: with a standard project, `final_review.enabled=false`, final-review arbiter/reviewers on disabled `claude`, and all other stages on `codex`, `backend check --json` returns `{ "passed": true }`, while `backend probe --role final_review_panel --flow standard` fails immediately on the required arbiter.
   - Expected: align `backend check` scoping with actual run stage planning. Either final review must be validated whenever the flow still contains that stage, or the run stage plan must start honoring `final_review.enabled` and diagnostics must use that same gate.
   - Reference: Slice 5 acceptance criterion that `backend check` covers the effectively required backends/panel members for the active flow/workspace scope ([20260319032842-spec.md](/root/new-ralph-burning/.ralph/projects/parity-plan/loops/006-backend-operations-parity/20260319032842-spec.md#L26)).
