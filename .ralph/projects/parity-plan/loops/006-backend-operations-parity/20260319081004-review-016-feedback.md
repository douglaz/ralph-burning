# Review: SUGGESTIONS

## Required Changes
1. **Opposite-family role attribution is wrong when implicit roles fail before resolution completes**: config-time failures and unresolved `show-effective` entries for roles that default to the opposite family are still labeled as the base backend.
Current: [diagnostics.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/agent_execution/diagnostics.rs#L237) and [diagnostics.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/agent_execution/diagnostics.rs#L797) use `family_for_role()`, whose fallback at [diagnostics.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/agent_execution/diagnostics.rs#L1274) returns `default_backend`’s family for `implementer`, `qa`, `acceptance_qa`, and `completer`, even though runtime resolution at [policy.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/agent_execution/policy.rs#L60) selects `opposite_family(planner_family)` for those roles. Direct repro on `HEAD`: with `default_backend=claude` and both opposite families disabled, `backend show-effective --json` reports `implementer/qa/completer.backend_family = "claude"` and `backend probe --role implementer --flow standard` errors as `backend 'claude' is unavailable`, while the actual failure is “no opposite backend family is enabled”.
Expected: unresolved opposite-family roles must reflect the runtime resolution path, not the base backend. `backend show-effective`, `backend check`, and singular-role `backend probe` should either surface the attempted opposite-family target/fallback semantics or mark the family unresolved, but they must not claim these roles resolve to `default_backend`.
Reference: Slice 5 acceptance criteria for `backend show-effective` exposing resolved backend selection and `backend probe --role <role> --flow <flow>` using the same resolution paths as run execution.

2. **Implicit completion-panel availability failures still report the wrong config source**: the config-time fix for implicit completion backends did not carry through to the availability-aware paths.
Current: [diagnostics.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/agent_execution/diagnostics.rs#L332) hard-codes `"completion.backends"` when `check_backends_with_availability()` validates completion members, and [diagnostics.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/agent_execution/diagnostics.rs#L1064) plus [diagnostics.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/agent_execution/diagnostics.rs#L1227) do the same for `probe_with_availability()`. Direct repro on `HEAD`: with implicit completion backends, a fake executable `claude` on `PATH`, and no `codex`, `backend probe --role completion_panel --flow standard` fails with `[source: completion.backends]`, and `backend check --json` emits `completion_panel.member[0/1].config_source = "completion.backends"`, even though those targets came from implicit Completer/default-backend resolution.
Expected: when `completion.backends` is implicit, availability-time failures must report the actual selecting source (`default_backend` via the Completer-role path), matching the runtime resolution path and the existing config-time behavior.
Reference: Slice 5 acceptance criteria that `backend check` and `backend probe` reuse execution resolution semantics and that required panel failures identify the effective config source field.
