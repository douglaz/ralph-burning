# Feature: Slice 1 — Full Requirements and PRD Parity

## Description
Loop 1 completed Slice 0 hardening, and the git checkpoint history after `master` shows only Slice 0 implementation plus parity-plan artifacts. The baseline already ships `requirements draft|quick|show|answer`, `extract_seed_handoff()`, and a simple `question_set -> requirements_draft -> requirements_review -> project_seed` flow. The next missing slice is full requirements parity: add the staged PRD pipeline for full mode, keep quick mode’s existing draft/review contracts but wrap them in a revision loop, and freeze a versioned `ProjectSeed` handoff for Slice 2.

## Acceptance Criteria
- [ ] `requirements draft` runs a staged full-mode pipeline `ideation -> research -> synthesis -> implementation_spec -> gap_analysis -> validation -> project_seed`, with `question_set` inserted only when validation finds blocking missing information, and every committed stage writes one canonical JSON payload plus one deterministic Markdown artifact.
- [ ] Existing quick-mode `requirements_draft` and `requirements_review` contracts are retained and reused for `requirements quick`; quick mode becomes a bounded writer/reviewer revision loop with structured revision feedback, approval-based termination, and at least one covered request-changes cycle before approval.
- [ ] Requirements run state becomes stage-aware without changing the existing high-level lifecycle semantics: canonical state records current stage, latest committed payload/artifact IDs per stage, question round count, quick revision count, and whether the last transition was fresh execution or cache reuse.
- [ ] Stage reuse is keyed by deterministic hashes of stage input plus upstream dependency outputs. On resume or after `requirements answer`, unchanged stages are reused without reinvocation, while any changed stage invalidates itself and all downstream dependents.
- [ ] If validation opens a question round, ideation and research history remain durable, synthesis and downstream stages are invalidated for rerun, the run transitions to `awaiting_answers`, and no `ProjectSeed` or completed state is written until answers are durably stored.
- [ ] `requirements answer <run-id>` applies answers against the latest committed question set, reruns only the invalidated full-mode stages, and preserves the invariant that no partial payload/artifact pair or partial `seed/` handoff survives a failed resume.
- [ ] Both full and quick modes emit the same versioned `ProjectSeed` shape. New seeds include an explicit version field and source metadata; `extract_seed_handoff()` accepts supported versions only and fails clearly on unsupported versions.
- [ ] If any stage write, journal append, or seed write fails, canonical state remains pinned to the last fully committed stage, previously committed stage history remains readable, and `seed/project.json` plus `seed/prompt.md` are rolled back together.
- [ ] `requirements show` remains artifact-scan-free and reports stage-aware progress, pending question state, and completed seed availability from canonical state and journal data.
- [ ] Conformance adds `parity_slice1_*` coverage for staged happy path, cache reuse on resume, conditional question round end to end, quick-mode revision loop, and versioned seed output; unit and CLI tests cover stage contract validation and failure invariants.
- [ ] A Slice 1 report records the exact legacy `prd` and `quick-prd` references consulted, contract changes, tests run, results, and remaining gaps.
