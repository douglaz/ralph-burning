# Review: SUGGESTIONS

## Required Changes
1. **Quick-mode revision-request rollback**: `quick_revision_count` is still persisted past the last durable boundary when `RevisionRequested` cannot be journaled.  
Current: [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L1435) increments `revision` and copies it into `run.quick_revision_count`, then [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L1457) appends `RevisionRequested`. If that append fails, [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L1461) goes straight to `fail_run` without restoring the prior count, so the failed `run.json` no longer reflects the last committed `ReviewCompleted` boundary.  
Expected: snapshot and restore `quick_revision_count` before attempting the `RevisionRequested` append so journal failure leaves canonical state pinned to the prior committed review state.  
Reference: Slice 1 acceptance criterion “If any stage write, journal append, or seed write fails, canonical state remains pinned to the last fully committed stage...”.

2. **Full-mode cached-resume state fidelity**: resumed full-mode runs can still report non-durable stage metadata after answer-boundary invalidation or stage journal failure.  
Current: after answers are accepted, [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L373) removes `synthesis` and downstream from `committed_stages` and [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L378) rewrites `run.json`, but it never recomputes `current_stage` from the surviving committed stages. Then the cached branches set `run.last_transition_cached = true` before the `StageReused` append ([service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L428), [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L435)), and fresh-stage rollback clears `last_transition_cached` at [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L1188) without restoring it on rollback at [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L1206). A failed resumed run can therefore advertise stale `current_stage` / transition metadata that was never durably committed.  
Expected: after post-answer invalidation, derive `current_stage` from the remaining committed stages (or clear it), and snapshot/restore `last_transition_cached` alongside the other stage-aware fields on `StageReused` and `StageCompleted` journal failures.  
Reference: Slice 1 acceptance criteria for stage-aware run state, cached resume, and journal-failure rollback.
