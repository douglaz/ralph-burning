# Implementation Notes

## Decisions Made
- Added `ExecutionMode` and `ExecutionSettings` to the shared config model and threaded them through `EffectiveConfig` with the same default/workspace/project/CLI precedence tracking used by existing backend settings.
- Refactored `ProcessBackendAdapter` around a reusable prepared-command path so `TmuxAdapter` can reuse backend-specific Claude/Codex argument construction and response decoding without maintaining a second copy of that logic.
- Implemented `TmuxAdapter` as an `AgentExecutionPort` wrapper that creates deterministic session names, captures pane output to runtime files, mirrors output into runtime logs, and performs session cleanup on cancel/timeout.
- Added `run attach` plus watcher-first `run tail --follow --logs` behavior when `execution.stream_output = true`, with polling retained as a fallback.
- Added focused unit coverage for execution-config precedence, tmux session lifecycle/equivalence, and tmux readiness diagnostics, plus a shared PATH test guard so tmux-related tests can run safely in parallel within the unit target.
- Added a Slice 6 conformance feature file with matching registry entries so scenario discovery stays in sync.

## Spec Deviations
- The daemon CLI path still constructs its long-lived agent service with the legacy direct builder, so project-specific `execution.mode = "tmux"` is fully wired for run/project/requirements paths but not yet for daemon-managed workflow execution.
- The new Slice 6 conformance feature is registered end-to-end, but only the highest-signal scenarios are implemented with concrete assertions today; several registry entries are placeholders to keep conformance discovery/registry checks aligned in this pass.
