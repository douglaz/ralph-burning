# Feature: Tmux and Streaming Parity (Slice 6)

## Description
Restore optional `tmux` execution mode and live streaming behavior so operators can attach to running backend invocations, observe live output, and retain full durable-history equivalence regardless of execution mode. This adds a `tmux` adapter that wraps `ProcessBackendAdapter` child-process spawning in tmux sessions, a configurable execution mode (`direct` vs `tmux`), a `run attach` CLI command for live session attachment, and enhanced `tail --follow --logs` streaming. Direct mode remains the default and must continue working unchanged when tmux is unavailable.

## Acceptance Criteria
- [ ] `[execution] mode = "direct" | "tmux"` is configurable at workspace and project level with standard source precedence (default ‚Üí workspace ‚Üí project ‚Üí CLI).
- [ ] `[execution] stream_output = true | false` is configurable and controls whether runtime log capture includes live incremental output or only final output.
- [ ] Starting a run with `mode = "tmux"` spawns backend invocations inside named tmux sessions with a deterministic naming scheme (e.g., `rb-<project-id>-<invocation-id>`).
- [ ] `ralph-burning run attach` connects the operator's terminal to the active tmux session for the current invocation; exits cleanly if no active session exists.
- [ ] `ralph-burning run tail --follow --logs` streams runtime log output incrementally (replacing 2-second poll with inotify/kqueue file watching where available, falling back to polling).
- [ ] Cancel (`run cancel` or Ctrl-C during `run start`) in tmux mode kills the tmux session and all child processes cleanly ‚Äî no orphan tmux sessions or child processes remain.
- [ ] Timeout in tmux mode triggers the same SIGTERM ‚Üí grace period ‚Üí SIGKILL ‚Üí session cleanup sequence as direct mode.
- [ ] Direct mode and tmux mode produce **equivalent durable history** (journal events, payloads, artifacts). Only ephemeral runtime-log metadata (e.g., tmux session name, attach/detach timestamps) may differ.
- [ ] If tmux is not installed or not found on `$PATH`, `mode = "tmux"` surfaces a clear readiness error at run start time and does not fall back silently.
- [ ] `backend check` (from Slice 5) reports tmux availability when `mode = "tmux"` is configured ‚Äî integrated as a new check kind, not a separate command.
- [ ] The tmux adapter satisfies `AgentExecutionPort` so it is transparent to the workflow engine and all upstream callers.

## Implementation Notes

### Tmux Session Lifecycle
The `TmuxAdapter` does not replace `ProcessBackendAdapter` ‚Äî it wraps it. The invocation flow:
1. `TmuxAdapter::invoke()` creates a named tmux session: `tmux new-session -d -s <session-name> -x 200 -y 50`
2. Inside the session, it runs the backend command (reusing args from extracted `build_command()`)
3. Output is tee'd to the runtime log path via `script` or pipe redirection so `tail --follow --logs` sees incremental output
4. `TmuxAdapter` monitors the tmux session for exit (polling `tmux has-session -t <name>`) while also reading the captured stdout for the structured response
5. On completion, the tmux session is killed if still alive and the structured response is returned identically to direct mode

### Cancel / Timeout Sequence (tmux mode)
1. Send `SIGTERM` to the process group inside the tmux session via `tmux send-keys -t <name> '' C-c`
2. Wait grace period (same as direct mode, from config)
3. If still alive: `tmux kill-session -t <name>` (force-kills all processes)
4. Verify no orphan with `tmux has-session -t <name>` returning error

### Durable History Equivalence
The tmux adapter must produce the same `InvocationEnvelope` as direct mode:
- Same `parsed_payload` (structured JSON from stdout)
- Same `InvocationMetadata` (duration, tokens, backend, model, session)
- `raw_output_reference` may point to a different runtime log path but must contain equivalent content
- The only allowed difference: runtime log may contain additional tmux lifecycle metadata lines (session created, attached, detached) clearly prefixed as ephemeral

### Config Schema Addition
```toml
# In workspace.toml or project config.toml
[execution]
mode = "direct"        # or "tmux"
stream_output = false  # or true
```

### State Invariants for Failure Modes
- **tmux session creation fails**: Invocation fails with `TransportFailure`; no tmux session is left behind; journal records the failure.
- **tmux binary disappears mid-run**: Current invocation completes or fails on its own (the child process is already running); next invocation fails at session creation. No silent fallback to direct mode mid-run.
- **Operator detaches from `run attach`**: Run continues unaffected. Detach is a terminal-only operation.
- **Cancel races with completion**: Same as direct mode ‚Äî if the process already exited, cancel is a no-op; if still running, cancel proceeds. The tmux session is cleaned up either way.
