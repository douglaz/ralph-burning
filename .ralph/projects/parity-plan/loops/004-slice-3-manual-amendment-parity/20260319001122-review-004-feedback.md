## Required Changes
1. **Amendment commit transactionality**: failed add/stage operations can still leave durable, visible amendment history behind.
   - Current: `add_manual_amendment` writes the amendment file at [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs#L812), appends `amendment_queued` at [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs#L826), and only rolls back the file if the snapshot write fails at [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs#L850). The journal event remains visible. The shared PR-review path has the same problem, and worse: `stage_amendment_batch` writes files and journal events per amendment at [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs#L1005) and [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs#L1018), then commits `run.json` only once at [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs#L1033) or [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs#L1044), with no compensation if that final write fails.
   - Expected: a failed add/stage must not leave a journal-visible amendment without the matching canonical snapshot state. Either make the journal/file/snapshot update failure-atomic, or add a compensating rollback path that restores the pre-command state for both manual and shared PR-review staging.
   - Reference: Slice 3 acceptance for durable history consistency and “if add fails before durable commit, no new amendment is visible”; fixed invariant “canonical state is structured state plus journal”.

2. **Remove/clear failure invariants**: single-amendment removal still reports success after a disk deletion failure, and clear can fall out of its exact partial-failure contract during repair.
   - Current: `remove_amendment` drops the amendment from `run.json` at [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs#L897) and ignores deletion errors at [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs#L905). That can leave the file on disk while the CLI reports success, and completion still blocks on disk-backed amendments at [engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L5144). `clear_amendments` only returns `AmendmentClearPartial` if the repair snapshot write succeeds; the second write is still fallible at [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs#L950) and [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs#L956).
   - Expected: `remove <id>` must fail cleanly when the durable deletion fails, leaving that amendment pending, and `clear` must preserve the exact removed/remaining invariant even if the repair snapshot write fails.
   - Reference: Slice 3 acceptance for `remove`/`clear` behavior and “if `remove <id>` fails, only that id remains pending” plus “if `clear` fails after partial deletion, the command returns the exact removed and remaining ids and `run.json` reflects only the remaining pending amendments”.
