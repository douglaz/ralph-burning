# Review: SUGGESTIONS

## Required Changes
1. **Durable history persistence**: successful amendment intake is still not guaranteed to produce the required `amendment_queued` history.
   - Current: after `run.json` is committed, manual add and shared batch staging both treat journal append as best-effort at [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs#L842) and [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs#L1059). At the same time, `read_journal` / serialization can still fail after the mutation is already durable at [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs#L845) and [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs#L1062). That means a command can succeed without the required history event, or fail after the amendment is already visible.
   - Expected: a successful add/stage must durably record the `amendment_queued` event, and journal errors must not create a “command failed but amendment committed” split outcome.
   - Reference: Slice 3 AC8, AC9; fixed invariant “canonical state is structured state plus journal”.

2. **Shared staging atomicity**: PR-review batch staging can still leak pre-commit amendment files.
   - Current: `stage_amendment_batch()` writes each file inside the loop at [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs#L1025). If a later `write_amendment()` fails, the `?` exits immediately from the loop starting at [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs#L1013), and the earlier files are never rolled back; cleanup only exists for snapshot-write failure at [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs#L1051).
   - Expected: any failure before the batch’s durable commit needs to restore the pre-command pending set, including rolling back earlier file writes in the same batch.
   - Reference: Slice 3 AC9, AC10.

3. **`clear` partial-failure invariant**: `run.json` can still be wrong after a reported partial clear.
   - Current: on partial deletion, the snapshot repair write is ignored at [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs#L975) and the command still returns `AmendmentClearPartial` at [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs#L979). If that repair write fails, `run.json` can continue to list removed amendments, and resume reconciliation only merges disk amendments into the snapshot at [engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L5153), so stale snapshot entries are not repaired away.
   - Expected: when `clear` reports exact `removed` and `remaining` IDs, `run.json` must already reflect only the remaining amendments; otherwise the service needs to restore disk state and fail differently.
   - Reference: Slice 3 AC10, AC11.
