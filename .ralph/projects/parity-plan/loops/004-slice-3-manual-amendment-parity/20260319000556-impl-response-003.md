# Implementation Response (Iteration 3)

## Changes Made
1. **Amendment mutation failure safety (Required Change 2)**: All mutation paths now drive dedup and existence checks from canonical snapshot state (`run.json`) instead of disk:
   - `add_manual_amendment`: dedup check changed from `amendment_queue.list_pending_amendments()` to `snapshot.amendment_queue.pending`. If the snapshot write fails after file+journal creation, the amendment file is rolled back to prevent the reconciliation-skip gap (journaled IDs missing from snapshot are permanently lost).
   - `remove_amendment`: existence verification and mutation now driven entirely from snapshot state. The snapshot is updated and written first; file deletion is best-effort afterward. If the snapshot write fails, no mutation is visible.
   - `clear_amendments`: reads pending set from snapshot instead of disk. The snapshot is optimistically cleared and committed first. Files are then deleted as best-effort. Un-deletable files are re-added to the snapshot in a second write.
   - `stage_amendment_batch`: dedup check changed from per-iteration disk reads to checking the in-memory snapshot pending queue (which accumulates staged amendments during the loop).

2. **Conformance: completion blocking (Required Change 1, sub-item 1)**: `parity_slice3_completion_blocking` executor at `scenarios.rs:17644` changed from `run start` to `run resume`, matching the CLI contract for projects in `paused` state after reopen.

3. **Conformance: clear partial failure (Required Change 1, sub-item 2)**: `parity_slice3_clear_partial_failure` replaced the filesystem permission-based approach with deterministic `RALPH_BURNING_TEST_AMENDMENT_REMOVE_FAIL_AFTER=1` failpoint injection. The executor now:
   - Extracts exact amendment IDs from both `add` commands
   - Asserts stderr contains `removed: {id}` and `remaining: {id}` for the specific IDs
   - Verifies `run.json` has exactly 1 remaining pending amendment
   - Cross-checks the remaining amendment ID in `run.json` matches the one reported in stderr

4. **Amendment remove failpoint (supporting infrastructure)**: Added `RALPH_BURNING_TEST_AMENDMENT_REMOVE_FAIL_AFTER` failpoint in `fs.rs` `remove_amendment()`, following the same pattern as the existing write failpoint.

5. **Snapshot-write failure unit tests (Recommended Improvement)**: Added 3 new tests with a `FailingRunSnapshotWriteStore` that always returns an I/O error:
   - `add_manual_amendment_rolls_back_file_on_snapshot_write_failure` ŌĆö amendment file is cleaned up
   - `remove_amendment_preserves_amendment_on_snapshot_write_failure` ŌĆö no mutation visible
   - `clear_amendments_preserves_all_on_snapshot_write_failure` ŌĆö no files deleted

6. **SharedRunSnapshotStore test fixture**: Added a combined read+write fake store so tests calling service functions multiple times see writes reflected in subsequent reads. Migrated dedup, remove, and clear tests to use it.
