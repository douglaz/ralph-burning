# Review: SUGGESTIONS

## Required Changes
1. **Canonical amendment state**: manual amendment mutations are not keeping `run.json` in sync with the durable queue.
Current: `RunSnapshot.amendment_queue` is part of canonical run state in [model.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/model.rs#L31), but [add_manual_amendment](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs#L755), [remove_amendment](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs#L851), [clear_amendments](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs#L868), and [reopen_completed_project](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs#L905) never update `snapshot.amendment_queue.pending`. I reproduced this with the built binary: after `project amend add`, the amendment file exists but `run.json` still shows `"pending":[]`, including the completed-project reopen path.
Expected: add/remove/clear/reopen must update the canonical pending queue in `run.json` atomically with the file-backed queue so pending amendments are visible in structured state, reopen writes the pending amendment into snapshot state, and partial clear leaves only remaining IDs reflected in `run.json`.
Reference: Slice 3 acceptance criteria for pending-amendment gating, reopen semantics, and partial-failure invariants (`AC 5`, `AC 7`, `AC 10`).

2. **PR-review staging parity**: automated PR-review amendments still bypass the shared staging path.
Current: [pr_review.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/pr_review.rs#L149) still converts amendments and writes them directly via [stage_amendments](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/pr_review.rs#L315); only the reopen helper is shared at [reopen_completed_project](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/pr_review.rs#L329). That means project-journal persistence and staging semantics are still split between manual and PR-review ingestion.
Expected: PR-review ingestion should go through the same shared amendment staging service as manual intake so dedup handling, journal persistence, reopen behavior, and snapshot updates are consistent between manual and automated amendments.
Reference: Slice 3 acceptance criteria for shared source metadata/journal persistence and explicit reuse of the shared amendment staging service (`AC 8`, `AC 9`).

3. **Operator-facing amendment contract**: the new CLI does not expose the metadata and failure details the spec requires.
Current: [project amend list](/root/new-ralph-burning/ralph-burning-rewrite/src/cli/project.rs#L655) prints only `id [source] <preview>` and omits `dedup_key`; long bodies are truncated rather than surfaced as durable metadata. On partial clear, [AmendmentClearPartial](/root/new-ralph-burning/ralph-burning-rewrite/src/shared/error.rs#L138) renders only counts, so operators do not get the exact removed and remaining IDs.
Expected: `project amend list` should surface the same metadata recorded in durable history for each pending amendment, and partial `clear` failures should return the exact removed and remaining IDs.
Reference: Slice 3 acceptance criteria for durable history metadata and `clear` partial-failure reporting (`AC 8`, `AC 10`).

4. **Conformance deliverables**: the new Slice 3 conformance coverage is incomplete and the registered executors do not match the implemented CLI.
Current: [manual_amendments.feature](/root/new-ralph-burning/ralph-burning-rewrite/tests/conformance/features/manual_amendments.feature#L1) covers only 8 scenarios and omits restart persistence, completion blocking, lease-conflict rejection, and clear partial-failure invariants. The registered executors in [scenarios.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/conformance_spec/scenarios.rs#L17294) also expect `Amendment: <id>` output, but the CLI prints a bare ID in [project.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/cli/project.rs#L641), so the add/remove scenarios would fail if executed.
Expected: add the missing `parity_slice3_*` cases and align the executor assertions with the actual CLI contract, or update the CLI contract consistently across product code, tests, and conformance.
Reference: Slice 3 acceptance criterion for conformance coverage (`AC 11`) and the prompt’s Required Deliverables / Conformance Requirements.

## Recommended Improvements
1. `project amend add` only probes the writer lock, releases it, and then mutates state in a separate step at [project.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/cli/project.rs#L606). Reusing the existing RAII lock flow in [cli_writer_lease.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/cli_writer_lease.rs#L136) like [run.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/cli/run.rs#L121) would avoid a race between the lease check and the actual mutation.
2. The preview truncation in [project.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/cli/project.rs#L670) slices the string by byte index and can panic on long UTF-8 amendment bodies. Use char-aware truncation or avoid slicing the raw `String` by byte offset.
