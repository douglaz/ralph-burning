# Feature: Slice 3 - Manual Amendment Parity

## Description
Loop artifacts and git checkpoint history show Slice 0, Slice 1, and Slice 2 are completed and approved, with no later parity-plan implementation commits. The baseline already has durable amendment queue storage, workflow-generated and final-review amendments, PR-review ingestion, resume reconciliation, completion blocking, and completed-project reopen for PR-review amendments. What is still missing is the operator-facing manual intake surface and the shared source-metadata and dedup contract, so the next logical feature is Slice 3 manual amendment parity.

## Acceptance Criteria
- [ ] Add `ralph-burning project amend add --text "..."` and `ralph-burning project amend add --file ./amendment.md`; exactly one input is required, empty bodies are rejected, and successful adds print a stable `amendment_id`.
- [ ] Add `ralph-burning project amend list`, `ralph-burning project amend remove <id>`, and `ralph-burning project amend clear`; `list` reads the canonical pending queue, and `remove` and `clear` affect only pending amendments.
- [ ] `QueuedAmendment` gains explicit source metadata and a stable dedup key. The schema must represent at least `manual`, `pr_review`, `issue_command`, and `workflow_stage`; this slice must populate `manual` and migrate existing PR-review and workflow producers onto the same contract.
- [ ] Manual dedup is deterministic: a pending manual amendment with the same normalized body and `manual` source metadata is a no-op that returns the existing amendment instead of silently creating another file or journal event.
- [ ] Pending manual amendments participate in the same gating as automated amendments: completion remains blocked while any pending amendment exists on disk or in `run.json`, and resume from any stage after planning rewinds to the flow planning stage before final-review or completion can succeed.
- [ ] Adding a manual amendment to a `completed` project reopens it immediately to a defined paused state: `status = paused`, `active_run = null`, `interrupted_run.stage_cursor.stage = <flow planning stage>`, and the last completed cycle and completion-round are preserved.
- [ ] Adding a manual amendment to `not_started`, `paused`, or `failed` updates durable pending state without changing unrelated snapshot fields; adding while a CLI or daemon writer lease is held fails cleanly with no mutation.
- [ ] Manual amendment ingestion appears in durable project history via `amendment_queued` events that include amendment id, source metadata, dedup key, and body; `project amend list` surfaces the same metadata.
- [ ] The shared amendment staging service is reused by PR-review ingestion so completed-project reopen, dedup handling, and journal persistence are consistent between manual and automated amendment paths.
- [ ] Failure invariants are explicit and tested: if add fails before durable commit, no new amendment is visible; if `remove <id>` fails, only that id remains pending and no other amendment changes; if `clear` fails after partial deletion, the command returns the exact removed and remaining ids and `run.json` reflects only the remaining pending amendments.
- [ ] Conformance adds `parity_slice3_*` coverage for manual add/list/remove, restart persistence, completion blocking, completed-project reopen, duplicate manual add, lease-conflict rejection, and `clear` partial-failure invariants.
- [ ] A Slice 3 report records the exact legacy manual-amendment references consulted, contract changes, tests run, results, and remaining gaps.
