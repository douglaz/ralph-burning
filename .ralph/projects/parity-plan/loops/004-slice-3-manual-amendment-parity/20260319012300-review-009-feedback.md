   Expected: Treat cleanup failure as a consistency error and include both failures in the returned error, or otherwise guarantee the staged files are removed before returning. Failed add/stage must not leave any visible pending amendments behind.

2. **Remove/clear failure invariants**
   Current: `remove_amendment()` ignores `write_amendment()` failure when restoring the deleted file after a snapshot error at [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs#L952), and `clear_amendments()` ignores restore failures after full or partial delete rollback at [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs#L1007) and [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs#L1027).    Expected: If rollback cannot fully restore the pre-command queue, return an explicit corruption/rollback error that includes both the original failure and the restore failure, matching the stronger pattern already used for journal rollback. `remove` must leave the target amendment pending on failure, and `clear` must either surface exact removed/remaining IDs with a repaired snapshot or clearly report unrecovered inconsistency.
   Reference: Slice 3 acceptance bullet “if remove <id> fails, only that id remains pending and no other amendment changes” and “if clear fails after partial deletion, the command returns the exact removed and remaining ids and run.json reflects only the remaining pending amendments.”
