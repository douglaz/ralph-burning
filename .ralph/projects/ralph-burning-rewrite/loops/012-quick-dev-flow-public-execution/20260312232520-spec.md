# Feature: Quick Dev Flow Public Execution

## Description
Loops 1 through 11 already delivered the rewrite foundation: workspace/bootstrap, flow discovery, stage contracts/renderers, effective config, canonical project/run storage, agent execution, standard/docs/CI orchestration, retry/resume/failure semantics, completion rounds/amendments, requirements drafting, and daemon routing/tasks/worktrees. The remaining core workflow gap is `quick_dev`: the preset, stages, contracts, renderer support, requirements recommendations, and daemon routing already exist, but `run start` and `run resume` still hard-fail with `UnsupportedFlow`, and current conformance/CLI coverage still encodes that deferral.

The next logical feature is to remove that public deferral and make `quick_dev` a first-class executable preset before planning the conformance/CI cutover. That keeps the scope focused on the missing behavior only, rather than re-planning engine generalization, daemon routing, or requirements work that is already complete.

## Acceptance Criteria
- [ ] `ralph-burning run start` accepts projects with flow `quick_dev`; the explicit `UnsupportedFlow` rejection path is removed for `quick_dev` while all existing run-state precondition checks remain unchanged.
- [ ] `ralph-burning run resume` accepts failed `quick_dev` runs from durable stage boundaries; resume must continue from canonical run snapshot + journal state, never from artifact scans or Markdown parsing.
- [ ] `quick_dev` executes the canonical stage order `plan_and_implement -> review -> apply_fixes -> final_review`; `prompt_review.enabled` has no effect on this stage plan and no prompt-review pause path is introduced.
- [ ] Happy-path `quick_dev` execution persists exactly one payload record and one rendered artifact per stage, appends the expected `run_started`, `stage_entered`, `stage_completed`, and `run_completed` journal events in sequence order, and finishes with `run.json` in `completed` status, no active run, and `completion_rounds = 1`.
- [ ] Preflight for `quick_dev` validates all four stages before any state mutation. If capability or availability checks fail for any `quick_dev` stage, `run.json`, `journal.ndjson`, payload history, artifact history, and sessions remain unchanged.
- [ ] On `review = approved` or `conditionally_approved`, the run continues to `apply_fixes` in the same cycle with the persisted review payload/artifact preserved in durable history.
- [ ] On `review = request_changes`, the non-passing review payload/artifact must still be durably committed before transition; then the engine emits `cycle_advanced`, increments the cycle monotonically, and restarts at `apply_fixes`. If `cycle_advanced` persistence fails, the run becomes failed and no partially-advanced cursor is committed.
- [ ] On `review = rejected`, the review payload/artifact remain durable history, the run transitions to `failed`, and no remediation-cycle advance occurs.
- [ ] On `final_review = approved`, the run completes normally. On `final_review = conditionally_approved` or `request_changes`, the final-review payload/artifact are committed first, durable amendments are queued, `completion_round_advanced` is appended, and the run restarts at `plan_and_implement` for the next completion round; cycle and completion-round counters remain monotonic and independent.
- [ ] Failure invariant for late-stage amendment persistence: if any amendment file in the `quick_dev` final-review batch fails to persist, any already-written files from that batch are removed, `run.json` amendment queue remains unchanged, and no completion-round advancement is recorded.
- [ ] `run status`, `run history`, and `run tail` correctly expose `quick_dev` stage names, cycle progression, and completion-round progression from canonical state with no special casing that treats the flow as deferred.
- [ ] A daemon task routed to `quick_dev` completes through the same workflow engine path without surfacing `UnsupportedFlow`, preserving the existing daemon task/lease invariants from loop 11.
- [ ] Conformance and CLI coverage are updated so `quick_dev` is tested as supported behavior, replacing the current “quick_dev remains deferred” assertions. Coverage must include: happy path, review remediation, final-review completion-round restart, failed-run resume, and preflight-no-mutation semantics.
- [ ] Existing `standard`, `docs_change`, `ci_improvement`, requirements-drafting, and daemon-routing behavior remains unchanged except for the removal of the `quick_dev` deferral.
