# Feature: Final Review Safety Hardening

## Description
Loops 001-016 already cover the v1 rewrite roadmap: workspace/init/config, fixed flow discovery, canonical project/run storage and journal queries, structured stage contracts and renderers, agent execution, standard/docs/ci/quick_dev execution, retry/resume/failure classes, completion/amendments, rollback, requirements drafting, daemon routing/tasks/leases/watchers, and conformance/CI. Loop 17 advanced through `completing -> final_review -> planning`, but the checkpoint history shows no post-review implementation commit, and the current code still contains accepted release-blocking gaps: CLI `run start`/`run resume` bypass the per-project writer lock, daemon dispatch mutates process-global CWD, claim-time journal failures can strand a task in `Claimed` with a live lease and abort the daemon, lease cleanup reports can claim success when cleanup failed, and transient `.claude`/`.ralph/tmp` artifacts remain tracked. This feature closes those remaining safety and release-hygiene gaps without introducing new product surface.

## Acceptance Criteria
- [ ] `ralph-burning run start` and `ralph-burning run resume` acquire the same per-project writer lock used by daemon task dispatch before any run-state mutation, and they release it on success, handled error, and early return.
- [ ] If the CLI cannot obtain the writer lock because another CLI or daemon worker already holds it, the command exits non-zero with an explicit lock-held message and does not create or resume a competing active run.
- [ ] Daemon worktree dispatch no longer calls `std::env::set_current_dir`; any execution path that needs a working directory receives it explicitly, so daemon behavior is independent of process-global CWD.
- [ ] `claim_task` becomes rollback-atomic across task state, lease state, and daemon journal writes: if any post-claim journal append fails, the durable end state must never be `Claimed` with a retained lease and writer lock.
- [ ] Claim failure invariants are explicit: if `LeaseAcquired` journaling fails before `TaskClaimed`, the task ends either `Pending` with no lease/worktree/writer lock or `Failed` with explicit failure class such as `claim_journal_failed`, and the journal contains enough compensating evidence to explain the cleanup outcome.
- [ ] A single task claim/journal failure is treated as a per-task failure only; `daemon start` continues scanning and may still process other eligible tasks in the same cycle.
- [ ] `LeaseService::reconcile` and daemon shutdown cleanup surface per-lease cleanup failures explicitly. `released_lease_ids` and user-visible release counts include only leases whose worktree removal, lease-file deletion, task lease-reference clearing, and writer-lock release all succeeded.
- [ ] Reconcile failure invariants are explicit: if task failure marking succeeds but lease cleanup fails, the task remains terminal but still recoverable for a later reconcile; if lease removal succeeds but task reference clearing fails, the lease is not reported as released and the task remains visibly inconsistent for operator repair.
- [ ] `ralph-burning daemon reconcile` prints a cleanup failure summary and exits non-zero when any stale lease cannot be fully cleaned up.
- [ ] Repository hygiene is enforced: `.claude/scheduled_tasks.lock`, `.ralph/tmp/`, and generated runtime-log files are ignored or removed from tracking, and the repository no longer carries committed transient session/runtime artifacts.
- [ ] Regression coverage is added for CLI writer-lock contention, panic/error-safe lock release, no process-global CWD dependency, claim-journal failure rollback, daemon continuation after claim failure, and reconcile partial-cleanup reporting.
