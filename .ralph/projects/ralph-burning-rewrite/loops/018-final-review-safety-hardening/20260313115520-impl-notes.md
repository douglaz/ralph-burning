# Implementation Notes

## Decisions Made
- **RAII writer-lock guard**: Implemented a `WriterLockGuard` struct in `cli/run.rs` with a `Drop` impl that calls `release_writer_lock`. This ensures the lock is released on success, error return (`?`), and panic unwind ‚Äî satisfying the "RAII-style" requirement without adding complexity to the engine call paths.
- **Lock holder ID for CLI**: Uses the string `"cli"` as the lease_id for CLI-acquired writer locks, distinguishing them from daemon-acquired locks that use `"lease-{task_id}"`.
- **CWD removal**: The `dispatch_in_worktree` method's `worktree_path` parameter is retained (as `_worktree_path`) for API stability but the `set_current_dir` calls are removed entirely. The engine functions already receive `base_dir` explicitly and all filesystem operations go through stores that use `base_dir`, so the CWD mutation was purely a process-global side effect.
- **Claim journal rollback**: If `LeaseAcquired` journal append fails, the claim attempts to roll back to `Pending` (no lease, no lock). If rollback itself fails, the task is marked `Failed` with `claim_journal_failed` failure class, ensuring the durable state is never `Claimed` with a live lease. If `TaskClaimed` journal fails, the task transitions to `Failed` with explicit class.
- **Per-task error handling**: `process_cycle` in the daemon loop now catches errors from `process_task` and logs them instead of propagating, so one task's claim/journal failure cannot abort the entire daemon scan cycle.
- **Reconcile accounting**: `ReconcileReport` gains a `cleanup_failures` field. A lease is only added to `released_lease_ids` when both `release()` and `clear_lease_reference()` succeed. Failed cleanups are captured with lease_id, task_id, and error details.
- **ReconcileCleanupFailed error variant**: Added to `AppError` so `daemon reconcile` can exit non-zero when cleanup fails, while still printing the reconcile summary first.

## Spec Deviations
- The spec mentions `adapters/fs.rs` for "shared project-lock access" ‚Äî the existing `FsDaemonStore` implementation of `acquire_writer_lock`/`release_writer_lock` was already sufficient for CLI use. The CLI calls the same `DaemonStorePort` trait methods through `FsDaemonStore`, so no new adapter code was needed.
- The `_worktree_path` parameter is kept in `dispatch_in_worktree` signature rather than removed, to avoid changing the call sites.
