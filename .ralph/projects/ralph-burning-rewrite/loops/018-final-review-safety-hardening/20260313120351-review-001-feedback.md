## Required Changes
1. **Claim/reconcile cleanup atomicity**: claim-time journal failures and partial lease cleanup are still not rollback-atomic.
   - Current: [task_service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/task_service.rs#L174) ignores `LeaseService::release(...)` failures in both journal-failure branches, and the successful rollback-to-`Pending` path returns with no compensating journal evidence. [lease_service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/lease_service.rs#L118) removes the lease file, releases the writer lock, and appends `LeaseReleased` before returning a worktree-removal error, which makes partial cleanup non-recoverable for later `reconcile` and can record a misleading success event.
   - Expected: treat lease/worktree/writer-lock cleanup as part of the claim rollback transaction, preserve durable evidence for the cleanup outcome, clear stale `lease_id` on terminal fallback, and keep enough durable lease state to allow later reconcile after partial cleanup failure.
   - Reference: Loop 18 feature spec acceptance criteria 4, 5, and 8.

2. **Daemon same-cycle scanning**: the daemon still stops after looking at one pending task, so it does not continue after claim failure or lock contention.
   - Current: [daemon_loop.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/daemon_loop.rs#L167) uses `.find(...)` to pick a single pending task, and [daemon_loop.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/daemon_loop.rs#L403) treats `ProjectWriterLockHeld` as a no-op. With `--single-iteration`, the daemon exits after that first skipped/failed task instead of moving on to the next eligible one.
   - Expected: one scan cycle should continue over remaining pending tasks after per-task claim/journal failure or writer-lock contention, including the `daemon start --single-iteration` case.
   - Reference: Loop 18 feature spec acceptance criterion 6 and [daemon_lifecycle.feature](/root/new-ralph-burning/ralph-burning-rewrite/tests/conformance/features/daemon_lifecycle.feature#L50).

3. **Regression coverage and conformance alignment**: the added tests do not yet cover the hardening cases the spec requires, and some registered conformance scenarios do not match their Gherkin.
   - Current: [automation_runtime_test.rs](/root/new-ralph-burning/ralph-burning-rewrite/tests/unit/automation_runtime_test.rs#L879) only exercises a successful reconcile path; there is no executable coverage for claim-journal rollback, partial reconcile cleanup failure, or panic-safe CLI lock release. [scenarios.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/conformance_spec/scenarios.rs#L5594), [scenarios.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/conformance_spec/scenarios.rs#L5604), and [scenarios.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/conformance_spec/scenarios.rs#L5617) implement trivial success cases for `DAEMON-LIFECYCLE-006/007/008` instead of the behavior described in [daemon_lifecycle.feature](/root/new-ralph-burning/ralph-burning-rewrite/tests/conformance/features/daemon_lifecycle.feature#L42).
   - Expected: add real unit/CLI/conformance coverage for claim rollback invariants, partial cleanup reporting, continued same-cycle daemon scanning, and no-global-CWD behavior, with executable scenario bodies that match the Gherkin text.
