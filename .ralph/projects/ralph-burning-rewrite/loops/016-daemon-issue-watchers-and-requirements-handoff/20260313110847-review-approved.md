## Acceptance Criteria Checklist
- [x] Watcher-backed issue ingestion is polled before each daemon cycle and creates tasks through the existing routing precedence.
- [x] Watched-issue tasks persist idempotency metadata including `issue_ref`, `source_revision`, resolved flow, and dispatch mode.
- [x] Re-polling the same `issue_ref` + `source_revision` is a no-op, while a newer revision after a terminal task can create a fresh task.
- [x] Daemon-side `/rb requirements draft` and `/rb requirements quick` commands are supported, and malformed/unknown requirements commands are rejected without task creation.
- [x] `requirements_quick` links a completed requirements run, persists `requirements_run_id`, and continues into project/workflow execution in the same daemon cycle.
- [x] `requirements_draft` links the requirements run and only enters `waiting_for_requirements` after the question set and answer template are durably committed.
- [x] Waiting-for-requirements tasks hold no lease and are naturally ignored by `daemon reconcile` because reconciliation only operates on leases.
- [x] Completed requirements runs resume waiting tasks on the next daemon cycle; incomplete runs remain waiting with no project/worktree creation.
- [x] Requirements-link failures leave the requirements run addressable and transition the task to `failed` with explicit linking failure handling.
- [x] Post-requirements seed/project failures preserve durable requirements history and fail the task without mutating an existing project flow.
- [x] Routed flow remains authoritative end-to-end, conflicting seed recommendations are recorded as warnings, and existing project flow immutability is enforced.
- [x] Conformance coverage includes watched-issue intake, idempotent re-polling, quick handoff, draft waiting/resume, duplicate rejection, routed-flow override, `SC-PROJ-009`, and the public conformance entrypoint passes.
- [x] `daemon status` shows the waiting state and linked `requirements_run_id` without regressing workflow-only task output.
