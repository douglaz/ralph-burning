# Implementation Response (Iteration 1)

## Changes Made

1. **Requirements handoff lifecycle** — Fixed all three components of the handoff:
   - **`handle_requirements_quick`** (`daemon_loop.rs`): No longer marks the task completed after seed extraction. Instead, it updates the task with seed-derived project metadata (project_id, project_name, prompt, resolved_flow) and switches dispatch_mode to Workflow, then returns Ok so the caller falls through into the standard claim/project-creation/workflow-dispatch path in the same daemon cycle.
   - **`handle_requirements_draft`** (`daemon_loop.rs`): Now properly transitions through `Pending → Claimed → Active` before invoking the draft service, so the subsequent `Active → WaitingForRequirements` transition succeeds. No worktree lease is needed for the draft path since it only generates questions.
   - **`check_waiting_tasks`** (`daemon_loop.rs`): Before resuming a waiting task, now extracts the seed handoff from the completed requirements run and populates the task's project metadata (project_id, project_name, prompt, resolved_flow). This ensures the next workflow dispatch cycle can create the project correctly. If seed extraction fails, the task transitions to failed while preserving the requirements run.
   - **`process_task`** (`daemon_loop.rs`): The requirements_quick match arm no longer returns early — it calls the handler and falls through. After the match block, the task is re-read from disk to pick up any mutations from requirements_quick before entering the standard claim path.

2. **Requirements command ingestion** — Separated requirements-command parsing from flow routing:
   - **`parse_requirements_command`** (`watcher.rs`): Now matches on `tokens.len() >= 2` instead of `== 3`, so malformed commands like `/rb requirements` (missing subcommand) or `/rb requirements draft extra` (extra tokens) return explicit errors instead of silently falling through to Workflow dispatch.
   - **`is_requirements_command`** (`watcher.rs`): New helper that checks whether a routing command string is a requirements command, used to prevent requirements commands from being passed to flow routing.
   - **`create_task_from_watched_issue`** (`task_service.rs`): Before calling `routing_engine.resolve_flow()`, filters out requirements commands from `routing_command` using `is_requirements_command()`. When a routing command is a requirements command, flow resolution uses labels and repo default only, preserving flow precedence without conflating requirements and flow commands.

3. **Conformance coverage** — Replaced all 8 placeholder DAEMON-INTAKE scenario executors with real assertions:
   - **DAEMON-INTAKE-001**: Creates a watched issue via `create_task_from_watched_issue`, asserts task fields (issue_ref, source_revision, dispatch_mode).
   - **DAEMON-INTAKE-002**: Creates a task, re-ingests the same issue, verifies idempotent no-op and single-task count.
   - **DAEMON-INTAKE-003**: Creates a RequirementsQuick task, verifies dispatch mode resolution and daemon status output.
   - **DAEMON-INTAKE-004**: Creates a RequirementsDraft task, exercises the full `Pending → Claimed → Active → WaitingForRequirements` transition, asserts final state (no lease, correct requirements_run_id).
   - **DAEMON-INTAKE-005**: Verifies `DuplicateWatchedIssue` error when different source_revision is submitted while a non-terminal task exists.
   - **DAEMON-INTAKE-006**: Creates a task with explicit `/rb flow quick_dev` routing command, verifies the routed flow overrides the default.
   - **DAEMON-INTAKE-007**: Tests all malformed requirements commands: unknown subcommand, bare `/rb requirements`, extra tokens, and valid non-requirements commands.
   - **DAEMON-INTAKE-008**: Writes a task in WaitingForRequirements state directly, runs `daemon status` via CLI, asserts output contains `waiting_for_requirements` and `requirements_run=req-123`.
   - SC-PROJ-009 fixture repair from the initial implementation is retained.

4. **Unit test additions** (`automation_runtime_test.rs`):
   - `parse_requirements_command_bare_requirements_fails`: verifies `/rb requirements` without subcommand is an error.
   - `parse_requirements_command_extra_tokens_fails`: verifies extra tokens after valid subcommand is an error.
   - `is_requirements_command_identifies_requirements_commands`: covers the new helper for both positive and negative cases.
   - `watched_issue_with_requirements_command_routes_flow_from_labels`: verifies that when routing_command is a requirements command, flow comes from labels instead.
