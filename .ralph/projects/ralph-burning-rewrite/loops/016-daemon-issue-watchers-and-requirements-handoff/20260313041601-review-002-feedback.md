## Required Changes
1. **Requirements draft lifecycle / public gate**: `requirements_draft` still does not meet the interactive draft contract end-to-end.
Current: `nix develop -c cargo run -- conformance run` fails on `RD-001` with `expected 'awaiting_answers', got 'completed'`. The default requirements stub returns an empty question set for every draft request in [stub_backend.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/adapters/stub_backend.rs#L376), so [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L253) skips straight to completion. The daemon then unconditionally moves `/rb requirements draft` tasks into waiting in [daemon_loop.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/daemon_loop.rs#L603), which I reproduced locally: the task was `waiting_for_requirements` while its linked requirements `run.json` was already `completed`.
Expected: normal draft mode must reach `awaiting_answers` for the non-empty-question path, and daemon tasks should only enter waiting when user answers are actually pending; the full conformance gate must pass afterward.
Reference: loop-16 acceptance criteria for `requirements_draft` waiting/resume and end-to-end `cargo run -- conformance run`; [requirements_drafting.feature](/root/new-ralph-burning/ralph-burning-rewrite/tests/conformance/features/requirements_drafting.feature#L8)

2. **Flow-override warning recording**: routed-flow overrides are only printed, not recorded.
Current: both [daemon_loop.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/daemon_loop.rs#L254) and [daemon_loop.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/daemon_loop.rs#L532) only `println!` when the seed suggests a different flow. Neither the task’s `routing_warnings` field in [model.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/model.rs#L64) nor the daemon journal gets a warning entry. A manual `/rb flow quick_dev` + `requirements quick` repro produced the stdout warning but no persisted warning in task JSON or `journal.ndjson`.
Expected: the override warning should be persisted on the task and/or journal so it survives beyond ephemeral runtime logs, while keeping the routed flow authoritative.
Reference: acceptance criterion “record a warning” for routed-flow overrides; [daemon_issue_intake.feature](/root/new-ralph-burning/ralph-burning-rewrite/tests/conformance/features/daemon_issue_intake.feature#L49)
