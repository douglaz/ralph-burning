# Feature: Daemon Issue Watchers and Requirements Handoff

## Description
Loops 5-14 and the last functional checkpoint at `6abf002` already delivered the core v1 runtime: canonical project/run storage, all four built-in flows, retry/resume/completion behavior, requirements drafting CLI, daemon task/lease/worktree lifecycle, rollback, and the conformance runner/CI wiring. The loop-15 checkpoints `fe45a1e` and `e671c00` only recorded the completion attempt; they did not land another runtime slice.

The remaining prompt gap is inside `automation_runtime`: the daemon still only scans pre-created task files and dispatches directly into workflow execution, so v1 is missing watcher-backed issue intake and the required workflow-or-requirements dispatch path. The public gate is also not green because `SC-PROJ-009` still writes a stale canonical `run.json` fixture. This feature completes the missing daemon intake/handoff layer without re-planning the already-shipped routing, lease, worktree, rollback, or conformance-runner foundations.

## Acceptance Criteria
- [ ] Add a watcher/ingestion port to `automation_runtime` that polls external issue candidates before each daemon cycle and creates daemon tasks through the existing routing rules (`/rb flow <flow-id>` over labels over repo default).
- [ ] Watched issue ingestion persists enough metadata to stay idempotent across polls, including `issue_ref`, a stable `source_revision`, the resolved flow, and a new dispatch mode such as `workflow`, `requirements_draft`, or `requirements_quick`.
- [ ] Re-polling the same `issue_ref` and `source_revision` must not create a second task. If a newer `source_revision` appears after a prior task is terminal, the daemon may create a fresh task for that newer revision.
- [ ] Add explicit daemon-side requirements commands for watched issues: `/rb requirements draft` enters the interactive requirements path, and `/rb requirements quick` enters the one-shot seed path. Unknown or malformed requirements commands fail ingestion with explicit diagnostics and create no task.
- [ ] `requirements_quick` tasks invoke `requirements_drafting` to a completed seed, persist the linked `requirements_run_id`, then continue through the existing project creation and workflow execution path in the same daemon cycle.
- [ ] `requirements_draft` tasks invoke `requirements_drafting::draft`, persist the linked `requirements_run_id`, and transition to a new non-terminal waiting state only after the question set and answer template are durably committed.
- [ ] State invariant: a task in the waiting-for-requirements state holds no writer lock, no worktree lease, and no active workflow run. `daemon reconcile` must ignore these tasks because there is no lease to recover.
- [ ] When `requirements answer <run-id>` completes the linked run, the next daemon cycle resumes the waiting task, derives the seed handoff, and only then creates or resumes the project workflow. If the requirements run is still incomplete, the task remains waiting and no project or lease is created.
- [ ] State invariant: if a requirements run is created but linking it back to the task fails, the requirements run remains addressable via `requirements show`, the task transitions to `failed` with an explicit failure class, and no project/worktree state is created.
- [ ] State invariant: if seed generation or project creation fails after a requirements run completes, the seed artifacts and requirements history remain durable, the task transitions to `failed`, and the daemon must not delete the requirements run or partially mutate an existing project flow.
- [ ] Routed flow precedence remains authoritative end-to-end. If a completed seed suggests a different flow than the task’s routed flow, the daemon must use the routed flow, record a warning, and never mutate an existing project’s immutable flow.
- [ ] Add conformance coverage for watched-issue ingestion, idempotent re-polling, `requirements_quick` handoff, `requirements_draft` waiting/resume behavior, duplicate issue rejection, and routed-flow override; repair `SC-PROJ-009` to use the current canonical `active_run.stage_cursor` shape; require `cargo run -- conformance run` to pass end-to-end afterward.
- [ ] `daemon status` surfaces the new waiting state and linked `requirements_run_id` while preserving existing output for workflow-only tasks.
