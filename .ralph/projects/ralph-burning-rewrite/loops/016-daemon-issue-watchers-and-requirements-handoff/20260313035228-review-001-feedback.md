## Required Changes
1. **Requirements handoff lifecycle**: integrate the new requirements dispatch modes with the daemon task state machine and the existing project/workflow handoff.
Current: `process_task` routes `requirements_quick` and `requirements_draft` before claim/activate, but the follow-on helpers only allow `active -> waiting_for_requirements` and `active -> completed`. In a temp workspace, a watched issue containing `/rb requirements quick` left the task `pending` with a linked `requirements_run_id` and no project under `.ralph-burning/projects/`; a watched issue containing `/rb requirements draft` failed immediately with `requirements_linking_failed` on `pending -> waiting_for_requirements`.
Expected: `requirements_quick` must persist the linked requirements run, derive the seed, and continue through project creation/workflow execution in the same daemon cycle. `requirements_draft` must durably enter `waiting_for_requirements`, and when answers are complete the next daemon cycle must derive the seed handoff before any project/worktree mutation.
Reference: loop-16 acceptance criteria for `requirements_quick`, `requirements_draft`, waiting-state invariants, and resume behavior; [daemon_loop.rs#L292](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/daemon_loop.rs#L292), [daemon_loop.rs#L417](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/daemon_loop.rs#L417), [daemon_loop.rs#L513](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/daemon_loop.rs#L513), [task_service.rs#L406](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/task_service.rs#L406), [task_service.rs#L430](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/task_service.rs#L430), [model.rs#L171](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/model.rs#L171), [daemon_loop.rs#L713](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/daemon_loop.rs#L713)

2. **Requirements command ingestion**: separate requirements-command parsing from flow routing and reject malformed requirements commands instead of silently falling back to workflow dispatch.
Current: `parse_requirements_command` only errors on unknown 3-token variants, so malformed input like `/rb requirements` falls through to `DispatchMode::Workflow`. Also, if `/rb requirements quick` is supplied via `routing_command`, the same field is passed into flow resolution and ingestion fails with a routing error instead of entering the requirements path.
Expected: unknown or malformed requirements commands must fail ingestion with explicit diagnostics and create no task. Valid requirements commands must still honor flow precedence via `/rb flow <flow-id>` over labels over repo default.
Reference: loop-16 acceptance criterion "Unknown or malformed requirements commands fail ingestion with explicit diagnostics and create no task"; [watcher.rs#L26](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/watcher.rs#L26), [watcher.rs#L54](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/watcher.rs#L54), [task_service.rs#L347](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/task_service.rs#L347), [routing.rs#L44](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/routing.rs#L44)
