# Review: SUGGESTIONS

## Required Changes
1. **Answers template round-trip safety**: valid `question_set` payloads can still produce an invalid or semantically altered `answers.toml`, which breaks the “editable template” contract for `requirements draft`.
   - Current: [contracts.rs#L121](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/contracts.rs#L121) only enforces non-empty, unique IDs and non-empty prompts, while [service.rs#L921](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L921) writes `q.id` as a bare TOML key and interpolates `prompt`, `rationale`, and `suggested_default` directly into comments/string literals. A payload with IDs like `team name` or `api.version`, or defaults/text containing quotes/newlines, can therefore generate invalid TOML or change key semantics before [service.rs#L980](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L980) validates the answers.
   - Expected: either constrain question IDs and free-text fields to a TOML-safe subset during domain validation, or render the template with proper TOML quoting/escaping and per-line comment escaping so every committed question set yields a valid editable `answers.toml` that round-trips through `requirements answer`.
   - Reference: Loop 10 acceptance criteria for “`question_set` payloads include stable question IDs … those IDs are the only keys accepted in `answers.toml`” and “`requirements draft` … commits … an editable `answers.toml` template.”

## Recommended Improvements
1. Add regression coverage for question IDs that require quoting and for prompts/defaults containing quotes or newlines, then assert the generated `answers.toml` still parses and resumes successfully.
