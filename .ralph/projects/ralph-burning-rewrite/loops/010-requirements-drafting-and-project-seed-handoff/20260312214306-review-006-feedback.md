# Review: SUGGESTIONS

## Required Changes
1. **`requirements show` reports stale pending questions after the run has already crossed the answer boundary**: the read model keeps and prints `pending_question_count` for any failed run, even when answers were already durably submitted and draft work has begun.
   - Current: [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L226) stores `pending_question_count` at question generation, [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L427) advances past the question boundary without clearing or recomputing it, [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L321) returns it for every failed run, and [requirements.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/cli/requirements.rs#L72) prints it directly. I reproduced this with a synthetic failed run at `question_round: 2` plus an `answers_submitted` journal event; `requirements show` still printed `Pending Questions: 1`.
   - Expected: Only surface pending questions when the latest durable boundary is still the committed question set. Once answers are durably stored or downstream draft/review state exists, `pending_question_count` should be cleared or derived as `0`/omitted.
   - Reference: Loop 10 acceptance criterion for `requirements show <run-id>` pending question count; prompt §11 requirement that reads derive from canonical run/journal state.

2. **Seed rollback violates the terminal-transition ordering invariant**: both seed-failure branches clean up external seed files before durably recording the failed terminal state.
   - Current: [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L675) and [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L699) call `remove_seed_pair()` before [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L803) persists `RunFailed`. If the process crashes or `fail_run()` fails after cleanup, canonical state can remain non-terminal even though external side effects were already removed.
   - Expected: Persist the failed run transition and `RunFailed` journal event before seed cleanup when possible, or durably record a cleanup-needed boundary before removing seed files.
   - Reference: Prompt §7 core invariant: terminal state transitions persist before cleanup of external side effects when possible; Loop 10 seed-write rollback acceptance criterion.
