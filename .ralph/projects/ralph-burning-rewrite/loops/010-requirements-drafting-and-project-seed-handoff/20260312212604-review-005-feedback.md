# Review: SUGGESTIONS

## Required Changes
1. **`requirements answer` durable-boundary handling**: accepted answers can still be resubmitted after `answers.json` is already durably replaced.
   - Current: [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L416) writes `answers.json` before [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L431) updates `run.json`, while the resume gate only checks `run.json` and the journal for `answers_submitted` in [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L351). If `write_run()` fails after `answers.json` commits, the latest durable boundary is no longer the committed question set, but a second `requirements answer` call is still accepted.
   - Expected: Treat persisted answers as the answers boundary. Either make `answers.json` + run/journal transition atomic, or extend the resumability gate so a run is no longer answerable once accepted answers have been durably stored.
   - Reference: Loop 10 acceptance criterion: `requirements answer <run-id>` is valid only for `awaiting_answers` runs and failed runs whose latest durable boundary is a committed question set.

2. **Seed rollback leaves seed history behind**: the seed-write failure path still preserves `project_seed` history artifacts that the spec says must not survive.
   - Current: [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L671) commits the `project_seed` payload/artifact pair before [fs.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/adapters/fs.rs#L1116) writes `seed/project.json` and `seed/prompt.md`. On prompt-write failure, [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L690) only removes the seed directory files, so `history/payloads/*seed*` and `history/artifacts/*seed*` remain visible on a failed run.
   - Expected: If `seed/prompt.md` fails after `seed/project.json` is written, no committed seed history should remain. Either write the seed files before committing `project_seed` history, or roll back the seed payload/artifact pair together with the seed directory outputs.
   - Reference: Loop 10 acceptance criteria for seed-write rollback and for failed runs retaining committed question-set/answers/draft/review artifacts.
