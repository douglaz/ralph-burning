# Feature: Requirements Drafting and Project Seed Handoff

## Description
Loops 1 through 9 already delivered the rewrite foundation: workspace init and flow discovery, stage contracts and renderers, effective config and active project resolution, canonical project/run storage and queries, agent execution, standard-flow orchestration, retry/resume semantics, completion rounds and amendments, and docs/CI flow execution. Git checkpoint history stops at loop 9 planning, and the entire `requirements` bounded context plus all four `requirements` CLI commands are still stubs at `HEAD`. The next missing roadmap slice is a canonical requirements-run workflow under `.ralph-burning/requirements/` that can draft requirements from an idea, pause for clarifying answers when needed, and emit a durable project-seed handoff without auto-creating a project or touching legacy `.ralph` PRD state.

## Acceptance Criteria
- [ ] Introduce a `RequirementsRun` state model persisted at `.ralph-burning/requirements/<run-id>/run.json` with, at minimum, `run_id`, `idea`, `mode` (`draft` or `quick`), `status` (`drafting`, `awaiting_answers`, `completed`, `failed`), `question_round`, `latest_question_set_id`, `latest_draft_id`, `latest_review_id`, `latest_seed_id`, `created_at`, `updated_at`, and `status_summary`.
- [ ] Each requirements run also persists `journal.ndjson`, `answers.toml`, `answers.json`, `history/payloads/*`, `history/artifacts/*`, `seed/project.json`, `seed/prompt.md`, and `runtime/{logs,backend}/*` under `.ralph-burning/requirements/<run-id>/`.
- [ ] `requirements show <run-id>` derives its read model from `run.json` and `journal.ndjson`, not by scanning artifacts, and prints the run ID, mode, status, question round, last failure summary if present, pending question count, recommended flow, and the seed prompt path when complete.
- [ ] Define structured contracts and deterministic renderers for four requirements-stage outputs: `question_set`, `requirements_draft`, `requirements_review`, and `project_seed`.
- [ ] Requirements contract validation follows the same invariant as project stages: schema validation occurs before any state mutation, domain validation occurs after schema validation and before transition commit, and rendered Markdown is never parsed for control flow.
- [ ] `question_set` payloads include stable question IDs, prompt text, rationale, `required` flag, and optional suggested defaults; those IDs are the only keys accepted in `answers.toml` and persisted `answers.json`.
- [ ] `requirements_draft` payloads include a problem summary, goals, non-goals, constraints, acceptance criteria, risks or open questions, and a recommended flow constrained to the built-in presets only: `standard`, `quick_dev`, `docs_change`, or `ci_improvement`.
- [ ] `project_seed` payloads include a recommended project `id`, project `name`, built-in `flow`, a final `prompt.md` body, and a short handoff summary suitable for manual `project create --prompt`.
- [ ] `requirements draft --idea "<text>"` creates a new run, durably records the idea, invokes question generation, and either transitions to `awaiting_answers` after committing the question payload/artifact plus an editable `answers.toml` template, or continues directly into draft/review/seed generation when no questions are required.
- [ ] `requirements quick --idea "<text>"` creates a new run in `quick` mode, skips question generation entirely, and executes draft -> review -> seed generation in one command using the same persistence and validation rules.
- [ ] `requirements answer <run-id>` is valid for runs in `awaiting_answers` and for failed runs whose latest durable boundary is a committed question set; it opens `<run-dir>/answers.toml` in `$EDITOR`, validates that every required question has a non-empty answer, persists `answers.json`, and resumes from the answers boundary instead of creating a new run.
- [ ] If `$EDITOR` fails or exits non-zero during `requirements answer`, the run state, journal, and previously accepted answers remain unchanged.
- [ ] Requirements drafting reuses `agent_execution`, but `agent_execution` must be generalized from workflow-only `StageContract` usage to a domain-neutral structured-contract interface so requirements contracts do not have to be added to the project workflow `StageId` enum.
- [ ] Planner defaults drive question generation, drafting, and project-seed generation; reviewer defaults drive requirements review; existing workspace default backend/model resolution and raw-output persistence rules continue to apply.
- [ ] Review outcomes have distinct semantics: `approved` allows seed generation and completion; `conditionally_approved` still completes but records follow-ups in canonical seed metadata and the rendered handoff summary; `request_changes` and `rejected` fail the run after the review payload/artifact are durably committed and before any seed files become visible.
- [ ] State invariants for partial failure are explicit: if the question payload/artifact pair fails to persist, the run must not transition to `awaiting_answers` and no partial question/template files may remain visible.
- [ ] State invariants for partial failure are explicit: if answer validation fails, `answers.json` is not replaced, no draft/review/seed stage runs, and the run remains at the same committed question boundary.
- [ ] State invariants for partial failure are explicit: if the draft or review payload/artifact pair fails to persist, the corresponding journal success event is not appended and no downstream stage begins.
- [ ] State invariants for partial failure are explicit: if `seed/project.json` is written but `seed/prompt.md` fails, both seed outputs are removed, the run remains non-completed, and only the already committed draft/review history survives.
- [ ] Completed requirements runs are immutable, failed runs retain the latest committed question set, answers, draft, and review artifacts, and `question_round` is monotonic across repeated answer cycles.
- [ ] `requirements show <run-id>` prints a suggested `ralph-burning project create --id ... --name ... --flow ... --prompt <seed/prompt.md>` command assembled from `seed/project.json` and `seed/prompt.md`.
- [ ] This loop does not auto-create, mutate, select, or delete projects; project handoff is durable seed output only, leaving `project create` as an explicit user action.
- [ ] Public behavior is covered by new Gherkin scenarios for: draft requiring answers, draft with no questions, quick-mode happy path, editor failure, answer validation failure, conditional approval handoff, review request_changes failure, and seed-write rollback.
- [ ] Existing workspace, project, and run behavior remains unchanged, including the current quick_dev run-execution deferral outside the `requirements` command surface.
