# Review: SUGGESTIONS

## Required Changes
1. **Requirements-run storage contract**: the run directory does not currently materialize the full file set the spec requires.
Current: `FsRequirementsStore::create_run_dir()` only scaffolds directories and `sessions.json`; `answers.toml` is only written on the non-empty-question branch, and `answers.json` is only written inside `requirements answer`. A manual `requirements quick --idea ...` run completed without either answers file present under the run directory.
Expected: Every requirements run should persist the full canonical file layout required by the spec, including `answers.toml` and `answers.json` even when questions are skipped or answers have not yet been submitted.
Reference: Loop 10 acceptance criterion: “Each requirements run also persists `journal.ndjson`, `answers.toml`, `answers.json`, ...`”.

2. **Canonical question-boundary failure state**: failures around question generation/template writing are not fully represented in canonical state.
Current: In `draft()`, both question payload/artifact persistence failure and `answers.toml` write failure set `run.json` to `failed` and return without appending a `run_failed` journal event. The template-write failure path also never records `pending_question_count`, and `show()` only exposes that field for `awaiting_answers`, so a failed run at the committed question boundary cannot satisfy the required `show` output.
Expected: When the run fails at or immediately after the committed question boundary, the terminal transition should be durably journaled, and canonical run state should retain enough data for `requirements show` to report the pending question count without artifact scanning.
Reference: Master Prompt §11 (`journal.ndjson` is the authoritative event source), plus the Loop 10 acceptance criteria for `requirements show <run-id>` and failed runs retaining the latest committed question set.

## Recommended Improvements
1. Add regression coverage for the paths above: quick/no-question runs should assert the required file layout, and question-boundary failure paths should assert both `run_failed` journaling and `requirements show` output.
2. `write_payload_artifact_pair_atomic()` should also remove the staged artifact file if the final artifact rename fails; otherwise `runtime/temp` can retain an orphaned staging file after an error.
