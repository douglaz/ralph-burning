# Review: SUGGESTIONS

## Required Changes
1. **Conditional approval semantics**: `conditionally_approved` reviews can complete without any follow-ups, so the run can reach completion without recording the required conditions.
   - Current: [contracts.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/contracts.rs#L170) only rejects non-approval outcomes with empty `findings`; [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L563) treats an empty `follow_ups` list as valid and proceeds to seed generation/completion.
   - Expected: Reject `conditionally_approved` review payloads that do not contain at least one follow-up, or otherwise guarantee a non-empty canonical follow-up list before completion.
   - Reference: Loop 10 acceptance criterion for review outcomes: `conditionally_approved` must complete while recording follow-ups in canonical seed metadata and the rendered handoff summary.

2. **`requirements answer` durable-boundary gating**: the resume gate still trusts `run.json` too much when a partial failure happens after `answers_submitted` is durably journaled.
   - Current: [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L336) accepts any `awaiting_answers` run without consulting the journal, but [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L382) appends `AnswersSubmitted` before [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L403) persists the run transition. If that later `write_run()` fails, the latest durable boundary is already past the question set, yet a second `requirements answer` call is still accepted.
   - Expected: Gate resumability from the latest durable boundary for both `awaiting_answers` and `failed` runs, or reorder the transition so `AnswersSubmitted` cannot be committed while the run still appears resumable from the question boundary.
   - Reference: Loop 10 acceptance criteria for `requirements answer <run-id>` and the durable-boundary invariant.
