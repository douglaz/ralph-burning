# Review: SUGGESTIONS

## Required Changes
1. **Requirements journal commit handling**: the requirements pipeline advances durable `run.json` state before the matching journal event is appended, but it does not recover if that append fails.
Current: [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L152) persists a new run before [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L156) appends `run_created`; [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L211) records the committed question boundary before [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L220) appends `questions_generated`; [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L446) advances past the answers boundary before [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L460) appends `answers_submitted`; later success boundaries repeat the same pattern at [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L529), [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L587), and [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L729). If any append fails, the command returns an error but leaves a mutated run snapshot with missing journal history, which can strand the requirements run in a non-resumable state and makes `journal.ndjson` non-authoritative.
Expected: handle requirements journal-append failures the same way the workflow engine already does: either roll back the just-written boundary or durably transition the run to `failed` before returning, so `run.json` and `journal.ndjson` stay consistent at every boundary.
Reference: Prompt §11 requires `journal.ndjson` to be the authoritative event source, and `requirements show` must derive from canonical run state plus journal state. The existing project pattern already enforces this in [engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L255) and [engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L1339), with regression coverage in [workflow_engine_test.rs](/root/new-ralph-burning/ralph-burning-rewrite/tests/unit/workflow_engine_test.rs#L1019) and [workflow_engine_test.rs](/root/new-ralph-burning/ralph-burning-rewrite/tests/unit/workflow_engine_test.rs#L1082).
