# Review: SUGGESTIONS

## Required Changes
1. **Answer validation and resume gating**: `requirements answer` does not enforce the committed question set or the durable-boundary rule.
Current: `answer()` accepts any failed run with `latest_question_set_id.is_some()` and `parse_and_validate_answers()` never loads the committed question payload, so unknown keys, missing required answers, and empty answer sets all pass ([service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L281), [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L767)). A manual CLI check completed a run from an empty `answers.toml` and wrote `answers.json` with an empty `answers` list.
Expected: Load the committed `question_set` payload, reject unknown IDs, require non-empty values for required questions, preserve `answers.json`/run/journal on validation failure, and only allow `answer` when the latest durable boundary is the committed question set.
Reference: Loop 10 acceptance criteria for `requirements answer`, question-ID validation, answer-validation failure invariants, and durable-boundary resume.

2. **`requirements show` canonical output**: the command cannot currently satisfy the required output contract.
Current: `show()` derives `pending_question_count` from `question_round`, not from committed question state, and the CLI prints `Recommended Flow: (see draft payload ...)` instead of the actual built-in flow ([service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L240), [requirements.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/cli/requirements.rs#L57)). A quick-run smoke test prints that placeholder verbatim.
Expected: Persist enough canonical summary in `run.json` and/or `journal.ndjson` to print the real pending question count and the real recommended flow without artifact scanning.
Reference: Loop 10 acceptance criteria for `requirements show <run-id>`.

3. **Partial-failure atomicity**: payload/artifact persistence still leaves partial durable state visible.
Current: `persist_payload_artifact_pair()` writes the payload first and explicitly leaves it behind if artifact persistence fails ([service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L651)). The question path also has no rollback if `answers.toml` write fails after the question payload/artifact pair commits ([service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L167)).
Expected: Make payload+artifact visibility atomic, and ensure question/template creation also rolls back cleanly so failed writes do not leave partial question, draft, or review files visible.
Reference: Loop 10 partial-failure invariants for question, draft, and review persistence.

4. **Conditional-approval handoff semantics**: follow-ups are not guaranteed to reach canonical seed output.
Current: on `conditionally_approved`, follow-ups are only interpolated into the planner prompt; the persisted seed payload and handoff artifact are taken verbatim from backend output, so follow-ups can be dropped entirely ([service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L488), [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L544)).
Expected: Merge review follow-ups into canonical seed metadata and the rendered handoff summary before persisting `seed/project.json` and the seed artifact.
Reference: Loop 10 acceptance criteria for `conditionally_approved` review outcomes.

5. **Workspace backend/model defaults are ignored for requirements runs**: requirements execution bypasses the existing config resolution rules.
Current: requirements stages call `resolver().resolve(role, None, None, None)`, so workspace defaults for planner/reviewer backend and model are never applied ([service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L606)).
Expected: Thread `EffectiveConfig` into requirements execution and resolve planner/reviewer targets the same way project workflow execution does.
Reference: Loop 10 acceptance criteria for planner/reviewer defaults and existing workspace backend/model resolution behavior.
