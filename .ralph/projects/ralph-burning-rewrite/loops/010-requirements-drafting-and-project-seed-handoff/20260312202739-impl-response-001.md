# Implementation Response (Iteration 1)

## Changes Made

1. **Answer validation and resume gating** (Required Change 1)
   - Rewrote `parse_and_validate_answers()` in `service.rs` to load the committed question set payload via new `read_payload()` store method, reject unknown answer keys, and require non-empty values for required questions.
   - Tightened `answer()` durable-boundary gating: failed runs are only resumable when `latest_question_set_id.is_some() && latest_draft_id.is_none()`, ensuring no draft/review has been committed past the question boundary.
   - Added `read_payload()` to `RequirementsStorePort` trait and `FsRequirementsStore`.

2. **`requirements show` canonical output** (Required Change 2)
   - Added `pending_question_count: Option<u32>` and `recommended_flow: Option<FlowPreset>` fields to `RequirementsRun` in `model.rs` (with serde defaults for backward compat).
   - `draft()` sets `pending_question_count` from the actual question set length when transitioning to `awaiting_answers`.
   - `continue_after_answers()` sets `recommended_flow` from the draft payload's `recommended_flow` field.
   - `show()` reads both from `run.json` directly â€” no artifact scanning.
   - CLI `show` prints `Recommended Flow: {flow}` using the actual flow value, replacing the placeholder `(see draft payload ...)`.
   - Added `recommended_flow: Option<FlowPreset>` to `RequirementsShowResult`.

3. **Partial-failure atomicity** (Required Change 3)
   - Added `write_payload_artifact_pair_atomic()` to `RequirementsStorePort` and `FsRequirementsStore` using staging directory (`runtime/temp/`), matching the project store's atomic write pattern.
   - Added `"runtime/temp"` to `REQUIREMENTS_RUN_SUBDIRS`.
   - All `persist_payload_artifact_pair` calls in the service now use the atomic store method.
   - In `draft()`, answers.toml write failure after question payload/artifact commit now fails the run explicitly (previously used `?` with no cleanup).

4. **Conditional-approval handoff semantics** (Required Change 4)
   - In `continue_after_answers()`, after receiving the seed payload from the backend, review follow-ups are merged into `seed_payload.follow_ups` (deduped) and appended to `seed_payload.handoff_summary`.
   - The seed artifact is re-rendered via `renderers::render_project_seed()` after merging, ensuring both `seed/project.json` and the history artifact include follow-ups.

5. **Workspace backend/model defaults** (Required Change 5)
   - Added `workspace_defaults: Option<BackendSelectionConfig>` to `RequirementsService` with `with_workspace_defaults()` builder method.
   - `invoke_stage()` now passes `self.workspace_defaults.as_ref()` to `resolver.resolve()` instead of `None`.
   - CLI handler loads `EffectiveConfig`, converts to `BackendSelectionConfig`, and passes to the service via `with_workspace_defaults()`.

6. **Acceptance coverage** (Required Change 6)
   - Added 4 new Gherkin scenarios (RD-011 through RD-014): editor failure preserves state, answer validation rejects unknown IDs, answer validation rejects empty required answers, conditional approval includes follow-ups in seed.
   - Added 5 new service integration tests: `show_returns_recommended_flow_from_completed_run`, `answer_validation_rejects_unknown_question_ids`, `answer_validation_rejects_empty_required_answers`, `conditional_approval_merges_follow_ups_into_seed`, `review_rejection_fails_run_and_preserves_review_artifact`.
   - Updated `draft_with_questions_transitions_to_awaiting_answers` to assert `pending_question_count`.
   - Added 5 new CLI tests: `requirements_quick_creates_completed_run`, `requirements_show_displays_completed_run`, `requirements_draft_with_empty_questions_completes`, `requirements_show_on_nonexistent_run_fails`, `requirements_answer_on_nonexistent_run_fails`.
