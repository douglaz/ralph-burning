# Review: SUGGESTIONS

## Required Changes
1. **Durable-boundary gating for `requirements answer`**: failed runs can still be resumed even after they have already crossed the question boundary.
   - Current: `answer()` allows any failed run with `latest_question_set_id.is_some()` and `latest_draft_id.is_none()` in [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L327), but the command itself writes `answers.json` and appends `answers_submitted` before draft generation in [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L356). That means a run that already durably recorded answers, then failed before any draft payload committed, still passes the gate. I reproduced this by seeding a failed run with an `answers_submitted` journal event; `ralph-burning requirements answer` resumed it to completion instead of rejecting it.
   - Expected: gate on the actual latest durable boundary, not just `latest_draft_id`. A failed run should only accept `requirements answer` when the committed question set is still the latest durable boundary; once answers have already been durably recorded, the command must reject resume.
   - Reference: acceptance criteria for `requirements answer <run-id>` being valid only for `awaiting_answers` runs and failed runs whose latest durable boundary is a committed question set.

2. **Canonical question-set state retention**: successful question generation is still not recorded canonically on every path.
   - Current: after the question payload/artifact pair is committed in [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L172), `latest_question_set_id` and the `questions_generated` journal event are only written in the non-empty-question branch at [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L210) and [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L220). On the default empty-question draft path, the run completes with `history/payloads/*-qs-1.json` present but `latest_question_set_id: null` in `run.json` and no question-stage event in `journal.ndjson`. The same omission exists on the `answers.toml` write-failure path in [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/requirements_drafting/service.rs#L196), where the run fails without ever recording the committed question set.
   - Expected: record the committed question-set boundary in canonical run state and journal immediately after the payload/artifact pair succeeds, before branching into `awaiting_answers`, empty-question continuation, or template-write failure handling. Failed runs must still retain the latest committed question set through `run.json` and `journal.ndjson`.
   - Reference: acceptance criteria for the `RequirementsRun` state model, `requirements show <run-id>` deriving from `run.json` and `journal.ndjson`, and failed runs retaining the latest committed question set.
