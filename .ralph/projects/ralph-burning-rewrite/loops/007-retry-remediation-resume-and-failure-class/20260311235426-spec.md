# Feature: Retry, Remediation, Resume, and Failure Class Semantics

## Description
Implement the retry/remediation layer, run resume capability, prompt review pause behavior, and failure-class-driven retry policies for the workflow engine. Currently (Loop 6), any stage failure immediately terminates the run. This feature adds: (1) configurable retry policies mapped to each `FailureClass`, with per-stage attempt tracking; (2) remediation cycles when QA or Review stages return `RequestChanges`, looping back to the implementation stage with an incremented cycle number; (3) `run resume` to continue a failed or paused run from the last durable stage boundary; (4) prompt review pause semantics where `readiness.ready == false` pauses the run for user revision. These behaviors are required by delivery roadmap item 7 and core invariants: "Cancellation and timeout must immediately halt retries," "Resume is allowed only from durable stage boundaries," and "Cycle and completion round numbers are monotonic and independent."

## Acceptance Criteria
- [ ] A `RetryPolicy` struct maps each `FailureClass` to max attempts and terminal/retryable classification. Default policies: `TransportFailure` → max 3 attempts; `SchemaValidationFailure` → max 2; `DomainValidationFailure` → max 2; `Timeout` → max 2; `Cancellation` → terminal (0 retries); `QaReviewOutcomeFailure` → not retried (triggers remediation or terminal depending on outcome).
- [ ] The engine retry loop increments `StageCursor.attempt` on each retry and emits `StageFailed` + `StageEntered` journal events per attempt. Cancellation signal during a retry loop immediately halts all further retries for that stage.
- [ ] When a QA or Review stage (but NOT `CompletionPanel`, `AcceptanceQa`, or `FinalReview`) returns `RequestChanges`, the engine persists the validation payload/artifact, emits a `CycleAdvanced` journal event, increments the cycle number, and loops execution back to the `Implementation` stage. Follow-up amendments from the review payload are passed as invocation context to the re-executed implementation stage.
- [ ] When a QA or Review stage returns `Rejected`, the run fails terminally with `QaReviewOutcomeFailure`. When `ConditionallyApproved`, execution continues to the next stage and the `follow_up_or_amendments` are appended to the `amendment_queue` in the run snapshot.
- [ ] Maximum remediation cycles per run is configurable (default 3). Exceeding the limit fails the run with a `RemediationExhausted` error.
- [ ] `StageContract::evaluate_permissive` performs schema validation + semantic validation + rendering WITHOUT the outcome check, returning `ValidatedBundle` even for non-passing validation payloads. The engine calls this, persists the bundle, then separately inspects the `ReviewOutcome` to decide remediation vs continue vs fail.
- [ ] `run resume` reads the snapshot (must be `Failed` or `Paused`) and journal, identifies completed stages via `StageCompleted` journal events, sets status back to `Running`, restores `active_run` with the cursor at the first incomplete stage (attempt reset to 1), appends a `RunResumed` journal event, and continues execution. The run_id is preserved across resume.
- [ ] `run resume` on a `NotStarted`, `Running`, or `Completed` snapshot returns a clear error.
- [ ] When the `PromptReview` stage produces a payload with `readiness.ready == false`, the engine persists the payload/artifact, sets `RunStatus::Paused`, clears `active_run`, and returns with a descriptive message. The user may revise the prompt and run `ralph-burning run resume` to continue from the `Planning` stage.
- [ ] `run start` on a project with status `Failed` or `Paused` returns a clear error directing the user to `run resume`.
- [ ] New `JournalEventType` variants: `StageFailed` (stage_id, cycle, attempt, failure_class, message, will_retry: bool) and `RunResumed` (run_id, resume_stage, cycle). Existing `CycleAdvanced` event builder is implemented (builder already referenced in enum but never constructed).
- [ ] `StubBackendAdapter` gains `with_transient_failure(stage_id, fail_count)` that fails the first N invocations for a stage and then succeeds, enabling retry and remediation test scenarios.
- [ ] Unit tests cover: retry exhaustion → run failure; retry success on second attempt; remediation cycle triggered by QA `RequestChanges`; remediation limit exceeded; resume from failed run skips completed stages; resume from paused prompt-review run; cancellation halts retry loop; `ConditionallyApproved` queues amendments and proceeds.
- [ ] State invariant: after any failure mode (retry exhaustion, remediation exhaustion, cancellation during retry, resume validation failure), the run snapshot is in a consistent terminal or resumable state — `Failed` with `active_run == None`, or `Paused` with `active_run == None`. No partial `Running` state is left behind.
