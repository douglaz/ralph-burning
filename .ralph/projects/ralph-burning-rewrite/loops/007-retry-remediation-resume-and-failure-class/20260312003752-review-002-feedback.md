# Review: SUGGESTIONS

## Required Changes
1. **Resume remediation context**: `run resume` restores the remediation stage boundary, but it does not restore the QA/Review follow-up context that the spec requires to be passed back into the re-executed `implementation` stage.
Current: the live remediation path only stores the follow-up bundle in the in-memory `implementation_context` before looping back to `implementation` ([engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L680)). On resume, the engine always re-enters with `implementation_context` set to `None` ([engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L437)), so `invocation_context()` rebuilds the resumed implementation request without any `"remediation"` payload ([engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L1298)). The added regression test for the remediation-boundary failure only checks that resume restarts at `implementation`, not that the persisted follow-ups are replayed into the resumed invocation ([workflow_engine_test.rs](/root/new-ralph-burning/ralph-burning-rewrite/tests/unit/workflow_engine_test.rs#L1415)).
Expected: when a QA/Review `RequestChanges` result has already been persisted and the run is later resumed from that durable boundary, the engine should durably preserve or reconstruct the remediation follow-ups and pass them into the resumed `implementation` invocation exactly as it does in the uninterrupted path.
Reference: loop 7 acceptance criteria for QA/Review `RequestChanges` ("Follow-up amendments from the review payload are passed as invocation context to the re-executed implementation stage") together with the `run resume` requirement to continue from the durable boundary.
