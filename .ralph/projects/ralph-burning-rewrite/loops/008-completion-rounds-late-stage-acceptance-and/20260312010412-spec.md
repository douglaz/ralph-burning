# Feature: Completion Rounds, Late-Stage Acceptance, and Durable Amendments

## Description
Approved loop artifacts `001` through `007` and checkpoint history through `4b10f4e` show the rewrite has completed the roadmap through retry, remediation, pause/resume, and failure-class handling. The next missing slice is roadmap step 8. The current standard-flow engine already executes `completion_panel`, `acceptance_qa`, and `final_review`, but it still treats them as generic validation stages: `completion_rounds` never advances beyond `1`, `StageCursor::advance_completion_round` and `JournalEventType::CompletionRoundAdvanced` are unused, `AmendmentQueued` and `processed_count` are unused, `projects/<id>/amendments/` is only scaffolding, and a run can currently complete while `amendment_queue.pending` is non-empty. This feature should add the real late-stage state machine for the standard preset: completion-round advancement, durable amendment queue persistence/application, and stage-specific continue-vs-complete behavior for `completion_panel`, `acceptance_qa`, and `final_review`, without re-implementing loops 1-7 or enabling non-standard flows yet.

## Acceptance Criteria
- [ ] Loop-7 behavior remains unchanged for `qa` and `review`: their `RequestChanges` remediation path still advances work cycles directly back to `implementation`, and the new logic applies only to `completion_panel`, `acceptance_qa`, and `final_review`.
- [ ] Late-stage transition rules are explicit and stage-specific: `completion_panel` `Approved` advances to `acceptance_qa`; `acceptance_qa` `Approved` advances to `final_review` when the flow definition enables final review and otherwise completes the run; `final_review` `Approved` completes the run; `Rejected` at any late stage fails the run terminally with `QaReviewOutcomeFailure`.
- [ ] `ConditionallyApproved` or `RequestChanges` at any late stage does not silently proceed and does not terminally fail; after the late-stage payload/artifact pair is durably committed, the engine queues every `follow_up_or_amendments` item as a durable amendment, emits `amendment_queued` journal events, emits one `completion_round_advanced` journal event, and moves the cursor to `planning` for the next completion round.
- [ ] On a `completion_round_advanced` transition, `StageCursor.completion_round` increments, `attempt` resets to `1`, the next stage becomes `planning`, and `cycle` stays unchanged at that boundary. `run.json.completion_rounds` tracks the highest round reached and remains monotonic independently of `cycle_history`.
- [ ] When the next completion round successfully leaves `planning` and re-enters `implementation`, the engine emits `cycle_advanced`, increments the work-cycle number, and records the new cycle in `cycle_history`. Completion-round continuation must therefore advance completion round first and work cycle only when new implementation work actually begins.
- [ ] Queued amendments are persisted under `projects/<id>/amendments/` using atomic writes and a typed canonical record that includes at least amendment id, source stage, source cycle, source completion round, body, and created timestamp. `run.json` stores the same queue in structured form; raw `serde_json::Value` placeholders are no longer the canonical queue model in this slice.
- [ ] Pending queued amendments are injected into the next `planning` invocation in deterministic order. They are marked processed and removed from pending on-disk state only after the `planning` stage commit succeeds; planning invocation failure or planning commit failure must leave them pending and reusable on `run resume`.
- [ ] `JournalEventType::CompletionRoundAdvanced` and `JournalEventType::AmendmentQueued` have concrete builders and are emitted in canonical order after the source late-stage commit. Public `run history`/`run tail` output includes those events without ever treating amendment files or runtime logs as durable history artifacts.
- [ ] `run_completed` may be persisted only when both the structured amendment queue and the `amendments/` directory contain no pending items. A late-stage conditional approval must never allow a completed snapshot with leftover queued amendments.
- [ ] `run resume` reconstructs pending amendments and the current completion round from canonical state and resumes from the first incomplete durable stage without duplicating already-processed amendments or losing queued ones.
- [ ] Failure invariant: if amendment persistence fails before any `amendment_queued` event is committed, no queue entry becomes visible in `run.json` and no `completion_round_advanced` event is written.
- [ ] Failure invariant: if some amendment files and `amendment_queued` events are durable but snapshot or cursor advancement fails afterward, those amendments remain pending and discoverable for resume; the run must end in a non-completed state with `active_run == null`.
- [ ] Failure invariant: if completion is otherwise ready but pending amendments are detected during terminalization, completion is blocked and the run remains resumable; that guard must not delete, drain, or silently mark any pending amendment as processed.
- [ ] Conformance and unit coverage are added for: completion-panel-driven continuation into round 2, acceptance-QA-driven continuation, final-review-driven continuation, terminal rejection at late stages, successful completion after a second round, resume with pending late-stage amendments, and the completion guard that blocks success while amendments remain queued.
- [ ] `quick_dev`, `docs_change`, and `ci_improvement` remain unsupported in `run start` after this slice; this feature hardens only the standard-flow late-stage semantics.
