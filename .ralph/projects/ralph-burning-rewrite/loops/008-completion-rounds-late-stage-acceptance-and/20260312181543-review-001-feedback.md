# Review: SUGGESTIONS

## Required Changes
1. **Completion guard resumability**
   - Current: `complete_run()` returns `CompletionBlocked` directly from [engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L1653), and the caller just propagates that error from [engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L1021). The last durable snapshot is still the stage-entered `running` snapshot written in [engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L1170), which means `run resume` will reject it as non-resumable in [engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L364).
   - Expected: when the completion guard fires, persist a non-completed snapshot with `active_run == null` and a resumable status, leaving queued amendments untouched. The command should still exit non-zero, but it must not strand the project in `running` state.
   - Reference: Loop 8 acceptance criterion “if completion is otherwise ready but pending amendments are detected during terminalization, completion is blocked and the run remains resumable”.

2. **Amendment queue failure semantics and deterministic ordering**
   - Current: late-stage amendments are written one file at a time in [engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L633) and a later write failure returns immediately without cleaning up earlier files. On resume, [engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L1680) blindly rehydrates whatever is on disk, while [fs.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/adapters/fs.rs#L767) orders amendments only by `created_at` even though each batch gets the same timestamp in [engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L1632). That can surface orphaned partial batches after a failed write and makes same-batch resume ordering non-deterministic.
   - Expected: persist amendment batches transactionally, or roll back any earlier files/events if the batch cannot be completed, and use a stable ordering key so resumed planning receives amendments in deterministic order.
   - Reference: Loop 8 acceptance criteria for amendment-persistence failure invariants and “Pending queued amendments are injected into the next planning invocation in deterministic order”.
