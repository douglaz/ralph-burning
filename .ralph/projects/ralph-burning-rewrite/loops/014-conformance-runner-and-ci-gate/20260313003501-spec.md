# Feature: Conformance Runner and CI Gate

## Description
Loops 1-13, confirmed by the approved loop specs and git checkpoints through `c61672d`, already delivered the v1 runtime: workspace/bootstrap, flow presets and stage contracts, workspace governance, canonical project/run storage and queries, agent execution, standard/docs/CI/quick_dev execution, retry/resume semantics, completion rounds and amendments, requirements drafting, daemon routing/tasks/leases/worktrees, and `run rollback`. The remaining roadmap gap is the conformance/cutover slice: the repo already contains the Gherkin source of truth under `tests/conformance/features/` (18 feature files, 202 scenarios), but `src/cli/conformance.rs` still returns `NotYetImplemented`, `src/contexts/conformance_spec/` is a placeholder, and there is no CI workflow enforcing the suite.

The next logical feature is to implement an executable conformance layer for the existing Gherkin corpus and wire it into CI. This slice should not add new workflow behavior; it should make the already-implemented public contract discoverable, runnable, filterable, fail-fast, and gated in CI, while adding a regression guard against reintroducing legacy runtime paths.

## Acceptance Criteria
- [ ] `ralph-burning conformance list` discovers every executable scenario defined in `tests/conformance/features/` and prints a deterministic table containing scenario ID, feature title, scenario title, and source file.
- [ ] Scenario IDs are resolved without rewriting the existing corpus: prefer a `# <scenario-id>` comment immediately preceding a `Scenario` or `Scenario Outline`; otherwise use the scenario-level tag. Feature-level tags are metadata only.
- [ ] Missing scenario IDs, duplicate scenario IDs, or malformed scenario IDs fail `conformance list` and `conformance run` with file-and-line diagnostics before any scenario executes.
- [ ] `ralph-burning conformance run` with no filter executes the full discovered suite in deterministic order: feature file path order, then in-file scenario order.
- [ ] `ralph-burning conformance run --filter <scenario-id>` performs exact scenario-ID matching only. Unknown IDs exit non-zero and execute zero scenarios.
- [ ] Scenario outlines are listed once under their scenario ID, and their executor covers every example row for that outline.
- [ ] Every discovered scenario ID has exactly one registered executor in `contexts/conformance_spec`, and every registered executor maps to a discovered scenario ID. Registry drift fails tests and causes `conformance run` to exit non-zero before execution.
- [ ] The runner executes scenarios in isolated temp workspaces using only the rewrite runtime (`.ralph-burning/`), the stub backend, and fake adapters or direct domain services as appropriate.
- [ ] Failure invariant: if feature parsing, registry validation, or filter resolution fails, zero scenarios are executed.
- [ ] Failure invariant: if one selected scenario fails, the runner stops immediately, reports the failing scenario ID and reason once, marks remaining selected scenarios as not run, and exits non-zero.
- [ ] Failure invariant: a scenario failure or panic must clean up its temp workspace and must not leak filesystem state, environment variables, or session state into later scenarios.
- [ ] `conformance run` prints a final summary with selected, passed, failed, and not-run counts plus wall-clock duration.
- [ ] The runner does not invent hidden scenarios outside `tests/conformance/features/`; the checked-in Gherkin corpus remains the sole public source of truth.
- [ ] Add a cutover regression guard that fails if production runtime code under `src/` introduces legacy workspace/runtime references such as `.ralph/` state roots or `multibackend-orchestration` imports or shell paths.
- [ ] `src/cli/conformance.rs` no longer returns `NotYetImplemented`, and `shared/error.rs` gains explicit conformance-specific errors for parse, discovery, registry, filter, cutover-guard, and scenario-execution failures.
- [ ] Add CLI/integration coverage for `conformance list`, successful filtered `conformance run`, unknown-filter failure, duplicate-ID detection, and fail-fast stopping behavior.
- [ ] Add unit coverage for scenario ID extraction, outline handling, deterministic ordering, registry completeness, fail-fast accounting, temp-workspace isolation cleanup, and the cutover guard.
- [ ] Add a CI workflow under `.github/workflows/` that runs the rewrite crate on every push and pull request with at least `cargo test` and `cargo run --locked -- conformance run`.
- [ ] The CI workflow uses the rewrite binary only and must not invoke legacy workspace fixtures or commands from `multibackend-orchestration`.
- [ ] After this slice, every v1 CLI command named in the master prompt is implemented and no additional blocking feature remains for functional completeness.
