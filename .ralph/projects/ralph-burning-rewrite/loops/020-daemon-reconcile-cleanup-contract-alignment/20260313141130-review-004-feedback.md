## Required Changes
1. **Claim rollback partial-cleanup handling**: The shared `release()` contract is still not applied correctly in the `claim_task` rollback path.
   - Current: After a `LeaseAcquired` journal failure, [task_service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/task_service.rs#L191) correctly computes `resources_released = false` for `Ok(ReleaseResult { ... })`, but [task_service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/task_service.rs#L240) still calls `release_result.unwrap_err()`. With the new `release()` behavior in [lease_service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/lease_service.rs#L213), a partial cleanup now returns `Ok(...)`, so this branch will panic instead of persisting a failed task with the lease retained.
   - Expected: Handle `Ok` partial-cleanup results explicitly in this branch, preserve `lease_id`, persist the terminal task state/journal consistently, and add regression coverage for a journal-failure path where `release()` returns `Ok` with `resources_released = false`.
   - Reference: Loop 20 acceptance criteria 4 and 5; Master Prompt §7 Core Invariants; Master Prompt §15 Testability Requirements.

2. **Reconcile success accounting for worktree disappearance races**: `daemon reconcile` still relies on a pre-check instead of the actual release outcome to enforce the missing-worktree contract.
   - Current: [lease_service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/lease_service.rs#L271) rejects an obviously missing worktree before release, but the later success path at [lease_service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/lease_service.rs#L288) never inspects `outcome.worktree_already_absent` or `outcome.resources_released`. If the worktree disappears between that pre-check and [lease_service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/lease_service.rs#L161), reconcile can still clear the task lease reference and add the lease to `released_lease_ids`.
   - Expected: Gate reconcile success on the full `ReleaseResult` (or explicitly handle `worktree_already_absent`) so `already absent` is never reported as a successful release, and add a regression test for that race.
   - Reference: Loop 20 acceptance criteria 2 and 6.
