# Feature: Daemon Reconcile Cleanup Contract Alignment

## Description
Loops 001-018 already cover the v1 rewrite roadmap: workspace/init/config, fixed flow presets, canonical project/run storage and journal queries, stage contracts and renderers, agent execution, standard/docs/ci/quick_dev execution, retry/resume, completion/amendments, rollback, requirements drafting, daemon routing/watchers, and the conformance runner. Git checkpoint history shows loop 19 returned from `completing` to `planning` without a follow-up implementation commit, and the current source reflects that: the earlier loop 18 hardening work is present in code (`run start`/`resume` writer locks, claim rollback handling, no process-global `set_current_dir`, reconcile cleanup reporting), but the built binary still fails the public conformance gate at `DAEMON-LIFECYCLE-006`. The unresolved gap is narrower than loop 18: `daemon reconcile` still treats a stale lease whose referenced worktree is already missing as a successful cleanup path, while the public conformance contract expects that condition to be surfaced as a cleanup failure with non-zero exit. The next feature is to align daemon reconcile cleanup semantics, durable-state invariants, and conformance coverage around that remaining mismatch.

## Acceptance Criteria
- [ ] `./target/debug/ralph-burning conformance run --filter DAEMON-LIFECYCLE-006` passes from a built binary, and the result is backed by real CLI behavior rather than test-only adapters.
- [ ] `daemon reconcile` does not count a stale lease as successfully released unless worktree cleanup, lease-file deletion, writer-lock release, and task lease-reference cleanup all complete under the chosen contract.
- [ ] A stale lease whose referenced worktree path is missing or otherwise cannot be positively cleaned up is reported as a cleanup failure: output includes `Cleanup Failures`, the lease id, and the task id, and the command exits non-zero.
- [ ] Failure invariant: if task failure marking succeeds but worktree cleanup fails, the task remains terminal, `released_lease_ids` does not include that lease, and the remaining durable lease state stays visible for later/operator recovery instead of being silently cleared.
- [ ] Failure invariant: if worktree cleanup succeeds but lease-file deletion, writer-lock release, or task lease-reference clearing fails, reconcile records the specific failed sub-step, does not report the lease as released, and preserves whatever inconsistent durable state remains.
- [ ] The implementation does not silently collapse `already absent` and `removed successfully` into the same cleanup outcome; the chosen policy is explicit in code and mirrored consistently in unit tests, CLI tests, and conformance scenarios.
- [ ] Existing unit coverage that currently assumes a missing worktree is a successful reconcile release is updated to the chosen contract, and new regression coverage is added for the non-zero CLI path.
