# Feature: Docs Change and CI Improvement Flow Execution

## Description
Generalize the workflow engine to execute `docs_change` and `ci_improvement` flow presets on the same orchestration core that currently drives the `standard` flow. Today, `execute_standard_run` and `resume_standard_run` contain hardcoded references to standard-flow stage identifiers (`Planning`, `Implementation`, `PromptReview`, `CompletionPanel`, `AcceptanceQa`, `FinalReview`). This feature introduces a `FlowSemantics` descriptor that maps each flow preset's structural roles (planning stage, execution stage, remediation target, late stages, amendment injection point) so the engine can orchestrate any built-in flow without per-flow branching. The CLI `run start` and `run resume` commands are updated to accept `docs_change` and `ci_improvement` projects, and new conformance scenarios verify end-to-end behavior for both presets.

## Acceptance Criteria
- [ ] A `FlowSemantics` struct is introduced that captures per-flow structural roles: `planning_stage` (the stage that receives amendment injection and drains amendments after commit), `execution_stage` (the stage that remediation cycles restart from), `remediation_trigger_stages` (stages whose `RequestChanges` outcome triggers a remediation cycle, analogous to today's `is_remediation_stage`), `late_stages` (stages whose `RequestChanges`/`ConditionallyApproved` outcomes trigger completion-round advancement), and `prompt_review_stage` (optional, for flows that include prompt review).
- [ ] `FlowSemantics` is derivable from a `FlowPreset` via a `flow_semantics(preset: FlowPreset) -> FlowSemantics` function. For `docs_change`: planning=`DocsPlan`, execution=`DocsUpdate`, remediation_triggers=[`DocsValidation`, `Review`], late_stages=[], prompt_review=None. For `ci_improvement`: planning=`CiPlan`, execution=`CiUpdate`, remediation_triggers=[`CiValidation`, `Review`], late_stages=[], prompt_review=None. For `standard`: planning=`Planning`, execution=`Implementation`, remediation_triggers=[`Qa`, `Review`], late_stages=[`CompletionPanel`, `AcceptanceQa`, `FinalReview`], prompt_review=Some(`PromptReview`).
- [ ] The engine functions `execute_standard_run`, `execute_standard_run_with_retry`, `execute_standard_run_internal`, `resume_standard_run`, and `resume_standard_run_with_retry` are generalized to accept a `FlowPreset` parameter (or derive `FlowSemantics` internally) instead of hardcoding standard-flow stage IDs. All existing hardcoded references to `StageId::Planning`, `StageId::Implementation`, `StageId::PromptReview` in the orchestration loop are replaced with lookups against `FlowSemantics`.
- [ ] The public engine API is renamed from `execute_standard_run` → `execute_run` and `resume_standard_run` → `resume_run` (keeping the old names as thin wrappers or removing them). The `_with_retry` variants follow the same rename.
- [ ] `standard_stage_plan()` is generalized to `stage_plan_for_flow(preset: FlowPreset, prompt_review_enabled: bool) -> Vec<StageId>` that derives the executable stage sequence for any preset. For `docs_change` and `ci_improvement`, `prompt_review_enabled` has no effect (no prompt review stage exists). For `standard`, behavior is unchanged.
- [ ] `standard_stage_plan_for_resume()` is generalized to `stage_plan_for_resume(preset: FlowPreset, events: &[JournalEvent], config: &EffectiveConfig)` with the same journal-reconstruction logic but using flow-specific stage definitions.
- [ ] `is_remediation_stage()` and `is_late_stage()` are replaced by lookups against `FlowSemantics::remediation_trigger_stages` and `FlowSemantics::late_stages` respectively.
- [ ] `record_cycle_advance()` uses `FlowSemantics::execution_stage` instead of hardcoded `StageId::Implementation`.
- [ ] Amendment drain logic (currently keyed to `stage_id == StageId::Planning`) uses `FlowSemantics::planning_stage`.
- [ ] Completion-round restart (currently targets `StageId::Planning`) uses `FlowSemantics::planning_stage`.
- [ ] Remediation-cycle restart (currently targets `StageId::Implementation`) uses `FlowSemantics::execution_stage`.
- [ ] Prompt-review pause check uses `FlowSemantics::prompt_review_stage` (skipped for flows without prompt review).
- [ ] `docs_change` happy path: `run start` on a docs_change project executes `docs_plan → docs_update → docs_validation → review` in sequence, persisting payload + artifact + journal events at each stage boundary, completing with `RunStatus::Completed`.
- [ ] `ci_improvement` happy path: `run start` on a ci_improvement project executes `ci_plan → ci_update → ci_validation → review` in sequence with the same persistence guarantees.
- [ ] `docs_change` remediation: when `docs_validation` or `review` returns `RequestChanges`, the engine triggers a remediation cycle restarting from `docs_update` (not `implementation`), with `CycleAdvanced` journal event emitted.
- [ ] `ci_improvement` remediation: when `ci_validation` or `review` returns `RequestChanges`, the engine triggers a remediation cycle restarting from `ci_update`.
- [ ] `docs_change` and `ci_improvement` have no late stages: `ConditionallyApproved` from `docs_validation`, `ci_validation`, or `review` does not trigger completion-round advancement; it is treated as non-late-stage conditional approval (amendments accumulated in snapshot but no completion-round loop).
- [ ] `Rejected` from any validation stage in docs_change/ci_improvement terminates the run with `QaReviewOutcomeFailure`, same as standard flow.
- [ ] `run resume` works for `docs_change` and `ci_improvement` projects that are in `Failed` or `Paused` status, reconstructing resume state from journal using flow-specific stage definitions.
- [ ] Retry semantics (transport failure, schema validation, timeout, cancellation) apply identically to docs_change/ci_improvement stages as they do to standard stages — no flow-specific retry policy changes.
- [ ] The CLI `run start` and `run resume` handlers in `src/cli/run.rs` remove the `UnsupportedFlow` guard for `DocsChange` and `CiImprovement` (but keep it for `QuickDev` which is deferred to a later loop). They pass the project's `FlowPreset` to the generalized engine.
- [ ] Completion guard still runs for docs_change/ci_improvement (it's a no-op since these flows have no late stages and thus never queue durable amendments, but the guard must not crash or produce false positives).
- [ ] All existing standard-flow tests continue to pass without modification (behavioral regression gate).
- [ ] New conformance Gherkin scenarios are added under `tests/conformance/features/`:
  - `run_start_docs_change.feature` — happy path, remediation cycle, rejection, retry on failure
  - `run_start_ci_improvement.feature` — happy path, remediation cycle, rejection, retry on failure
  - `run_resume_non_standard.feature` — resume from failed/paused for docs_change and ci_improvement
- [ ] New unit tests cover `FlowSemantics` derivation for all four presets, verifying each structural role field.
- [ ] New engine-level tests verify that `execute_run` with `DocsChange` and `CiImprovement` presets produce correct journal event sequences and final snapshot states.
