# Implementation Notes

## Decisions Made
- **Three payload families cover all 16 stages.** Planning (4 stages), Execution (5 stages), and Validation (7 stages) share typed payload structs matching the spec's field groupings.
- **Schema validation via serde deserialization.** `schemars` generates JSON Schemas for external consumers; actual validation uses `serde_json::from_value`. Avoids extra `jsonschema` dependency.
- **`EvalSuccess` carries both bundle and optional outcome failure.** Hard failures (schema/domain) return no bundle; QA/review outcome failures return the bundle with a `QaReviewOutcomeFailure` flag.
- **Only `Approved` is passing.** `ConditionallyApproved`, `RequestChanges`, and `Rejected` all map to `QaReviewOutcomeFailure`.
- **`StageId::ALL` and `display_name()` added** without changing any existing enum variants or flow orderings.

## Spec Deviations
- JSON Schema validation uses serde rather than a dedicated `jsonschema` crate. Schemas are still generated via `schemars` and exposed via `StageContract::json_schema()`.
- No separate `RenderFailure` in `FailureClass` — render errors map to `DomainValidationFailure`.

## Testing
- **48 tests pass** (up from 12), covering registry completeness, schema/domain short-circuiting, successful evaluation per family, QA outcome failure mapping, JSON Schema generation, deterministic rendering, and conformance scenario IDs (SC-EVAL-001 through SC-EVAL-007).
- **`cargo clippy --all-targets --all-features -- -D warnings` passes clean.**
- **9 Gherkin conformance scenarios** in `stage_contracts.feature`.
