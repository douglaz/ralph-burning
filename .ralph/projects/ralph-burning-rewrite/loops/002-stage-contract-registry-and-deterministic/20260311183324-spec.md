# Feature: Stage Contract Registry and Deterministic Artifact Rendering

## Description
Loop 1 is complete: the approved artifact and git checkpoints `1dbe79b` through `60e6d0c` delivered the Rust crate skeleton, bounded-context module layout, workspace init, built-in flow discovery, and the shared `FlowPreset` / `StageId` foundation. The only later change at `cac0f25` adds `flake.nix` support and does not add workflow behavior. The next missing slice is the unfinished remainder of roadmap step 2: a `StageContract` and renderer layer for the already-defined stages. The current codebase has no stage payload schemas, no semantic validators, no failure-class mapping, and no deterministic Markdown artifact rendering, so later work on agent execution and `run start` would otherwise have no canonical contract to target.

## Acceptance Criteria
- [ ] Add a `StageContract` registry keyed by the existing `StageId`, and every stage already present in the four built-in presets resolves to exactly one contract without changing any approved flow IDs or stage ordering.
- [ ] Introduce typed stage payload models plus JSON Schema generation/validation for all supported stages: `prompt_review`, `planning`, `implementation`, `qa`, `review`, `completion_panel`, `acceptance_qa`, `final_review`, `plan_and_implement`, `apply_fixes`, `docs_plan`, `docs_update`, `docs_validation`, `ci_plan`, `ci_update`, and `ci_validation`.
- [ ] Planning-style contracts (`prompt_review`, `planning`, `docs_plan`, `ci_plan`) require structured problem framing, assumptions or open questions, ordered proposed work, and explicit readiness or risk fields.
- [ ] Execution/update contracts (`implementation`, `plan_and_implement`, `apply_fixes`, `docs_update`, `ci_update`) require structured change summaries, intended or completed steps, validation evidence or plans, and outstanding-risk fields.
- [ ] Validation/review contracts (`qa`, `docs_validation`, `ci_validation`, `acceptance_qa`, `review`, `final_review`, `completion_panel`) require an explicit outcome/decision enum plus evidence, findings or gaps, and follow-up or amendment data when the outcome is not cleanly approved.
- [ ] Contract evaluation order is fixed: schema validation runs first, semantic validation runs only on schema-valid payloads, and rendering runs only on payloads that passed both prior steps.
- [ ] Introduce the canonical failure-class enum needed by later slices, including `transport_failure`, `schema_validation_failure`, `domain_validation_failure`, `timeout`, `cancellation`, and `qa_review_outcome_failure`.
- [ ] Stage contracts map non-passing QA/review/completion decisions to `qa_review_outcome_failure` without treating them as schema or domain failures.
- [ ] State invariants for this slice are explicit: on schema failure, semantic validation and rendering do not run; on domain failure, rendering does not run; on any failure, no success bundle is returned; callers receive either `validated payload + rendered artifact` together or an error, never one without the other.
- [ ] Markdown renderers are deterministic and stage-specific: the same validated payload always renders byte-identical output, uses canonical `cycle` terminology, and is derived only from validated structured fields rather than raw backend text.
- [ ] Raw backend output remains diagnostic-only input to the contract evaluator and is kept separate from payload and artifact outputs so later persistence work can store diagnostics without using Markdown as protocol.
- [ ] Add unit tests proving every stage in every built-in flow has contract coverage, schema failures short-circuit semantic validation, semantic failures short-circuit rendering, and repeated renders of the same payload are identical.
- [ ] Add conformance scenario IDs for successful contract evaluation, schema failure, domain failure, and QA/review outcome failure so later `run start` work inherits fixed public expectations.
- [ ] `cargo test` and `cargo clippy --all-targets --all-features -- -D warnings` pass after the contract and renderer modules are added.
