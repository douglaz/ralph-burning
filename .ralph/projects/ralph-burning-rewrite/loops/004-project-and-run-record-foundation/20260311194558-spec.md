# Feature: Project and Run Record Foundation

## Description
Loops 1 through 3 are complete: the rewrite already has workspace bootstrap, built-in flow discovery, stage-contract/rendering infrastructure, and workspace governance with effective config plus active-project resolution. The next missing slice is roadmap step 4: make `project_run_record` the durable source of truth by implementing canonical project storage, initial run/journal state, project CRUD, and read-only run query commands.

Current HEAD still has `src/contexts/project_run_record/mod.rs` as a stub, `project create/list/show/delete` unimplemented, and all `run` commands unimplemented. This feature should fill that gap without re-planning completed flow, config, or active-project work.

## Acceptance Criteria
- [ ] `ralph-burning project create --id <id> --name <name> --prompt <file> --flow <flow-id>` validates workspace version, project ID, flow preset, prompt readability, and duplicate IDs before any filesystem mutation.
- [ ] Successful project creation writes `.ralph-burning/projects/<project-id>/project.toml`, `prompt.md`, `run.json`, `journal.ndjson`, `sessions.json`, `history/payloads/`, `history/artifacts/`, `runtime/logs/`, `runtime/backend/`, `runtime/temp/`, `amendments/`, and `rollback/`.
- [ ] `project.toml` persists immutable project metadata: `id`, `name`, fixed `flow`, copied prompt reference, prompt hash, creation timestamp, and durable status summary. Flow cannot be changed after creation by any CLI or store path.
- [ ] `run.json` persists canonical run state only: active run pointer, run status, optional stage cursor, cycle history, completion rounds, rollback-point metadata, amendment-queue state, and status summary. It must not infer state from artifacts or logs.
- [ ] `journal.ndjson` is append-only, sequence-ordered, and initialized with a `project_created` event. Queries derive state from canonical run state plus journal, never from Markdown scans.
- [ ] `sessions.json` initializes as a valid empty session store, separate from durable history.
- [ ] `project create` is atomic. If prompt copy, snapshot write, or journal initialization fails, the new project is not visible to `project list/show`, no partial canonical files remain exposed, and `.ralph-burning/active-project` is unchanged.
- [ ] Duplicate project creation fails non-zero with a clear error and leaves existing project state unchanged.
- [ ] `project create` does not implicitly select the new project; active-project mutation remains the job of the already-implemented `project select`.
- [ ] `ralph-burning project list` reads canonical project records, not raw directory names, and shows enough summary data to identify each project plus whether it is the active selection.
- [ ] `ralph-burning project show [<id>]` shows explicit project data when `<id>` is provided and otherwise resolves the active project through workspace governance. Output includes flow, prompt hash/reference, created timestamp, run summary, and rollback summary.
- [ ] `ralph-burning project delete <id>` deletes only projects with no active run. If the deleted project was selected, `.ralph-burning/active-project` is cleared only after the delete succeeds. If delete fails after validation, the project remains addressable and the active-project pointer is unchanged.
- [ ] Add payload/artifact store models and enumeration APIs for durable history. If a payload write succeeds but its artifact companion write/index fails, `run history` and `run tail` must not expose a partial durable-history record.
- [ ] Runtime logs are stored and queried separately from durable history. Runtime-log write failures must not fabricate, remove, or reorder durable history.
- [ ] `ralph-burning run status` reads canonical state for the active project and reports either “not started” or the persisted run snapshot without inferring from artifacts.
- [ ] `ralph-burning run history` shows durable history only: journal events plus payload/artifact records in canonical order. Runtime logs never appear there.
- [ ] `ralph-burning run tail` shows durable history only by default, and `ralph-burning run tail --logs` appends the newest runtime logs after the durable section without changing the durable-history portion.
- [ ] Corrupt or inconsistent canonical files (`project.toml`, `run.json`, `journal.ndjson`, payload/artifact indexes) fail fast with file-specific errors. Commands must not fall back to rendered Markdown or directory scans when canonical state is invalid.
- [ ] Introduce `ProjectStorePort`, `JournalStorePort`, `ArtifactStorePort`, and `RuntimeLogStorePort` boundaries inside `contexts/project_run_record`, with filesystem-backed adapters using atomic writes/renames.
- [ ] Add domain/unit tests for immutable flow, journal sequencing, and history/log separation; service tests with fake ports; adapter contract tests for filesystem round trips and corruption visibility; and CLI integration tests for project CRUD plus run queries.
- [ ] Add public Gherkin coverage for project creation, duplicate IDs, immutable flow, list/show/delete behavior, no-active-run status, durable-history-only queries, and `run tail --logs` separation semantics.
- [ ] `cargo test` and `cargo clippy --all-targets --all-features -- -D warnings` pass.
