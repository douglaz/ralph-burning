# Review: SUGGESTIONS

## Required Changes
1. **Canonical project validation is still incomplete on `project select`**: structurally invalid `project.toml` files are still accepted as selectable projects.
Current: [`project.rs`](/root/new-ralph-burning/ralph-burning-rewrite/src/cli/project.rs#L44) routes `project select` through [`set_active_project`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workspace_governance/mod.rs#L84), and [`validate_project_exists`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workspace_governance/mod.rs#L104) only checks directory existence, file presence, and TOML syntax via `toml::Value`. It never deserializes a canonical `ProjectRecord`. I reproduced this with the built binary: a project directory whose `project.toml` contained only `id = "alpha"` was accepted, and `project select alpha` exited `0`.
Expected: `project select` should fail fast with a file-specific corruption error when `project.toml` is missing required canonical fields or otherwise cannot deserialize into the canonical project record.
Reference: Loop 4 acceptance criterion “Corrupt or inconsistent canonical files (`project.toml`, `run.json`, `journal.ndjson`, payload/artifact indexes) fail fast with file-specific errors”; master prompt §11 “Project and Run Record Rules”.

2. **`project delete` is not transactional across active-pointer cleanup**: a post-delete failure can still leave the project gone even though the command fails.
Current: [`delete_project`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs#L200) deletes the project first and only then calls [`clear_active_project`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs#L228). The filesystem adapter’s [`clear_active_project`](/root/new-ralph-burning/ralph-burning-rewrite/src/adapters/fs.rs#L590) can return I/O errors. If that happens, the command returns an error after validation, but there is no rollback path to restore the deleted project, so the project is no longer addressable.
Expected: a failure while clearing `.ralph-burning/active-project` must not strand the delete half-complete. Either make the pointer-clear step part of a transactional delete workflow or restore the project before returning an error.
Reference: Loop 4 acceptance criterion for `project delete`: if delete fails after validation, “the project remains addressable and the active-project pointer is unchanged.”

## Recommended Improvements
1. Add regression coverage for both gaps: a CLI test for `project select` against syntactically valid but schema-invalid `project.toml`, and a service/adapter test where `clear_active_project` fails after a successful directory delete.
