# Review: SUGGESTIONS

## Required Changes
1. **`project delete` transactional semantics**: the active-project pointer is cleared too early and rollback is only best-effort.
   - Current: [`delete_project`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs#L201) calls `clear_active_project` before `store.delete_project(...)` at [`service.rs` lines 230-234](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs#L230), even though the spec requires clearing the pointer only after a successful delete. On delete failure it only does a best-effort `write_active_project(...)` restore at [`service.rs` lines 236-240](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs#L236), so a restore failure would still leave the pointer changed.
   - Expected: Keep `.ralph-burning/active-project` intact until the delete has committed, or move the pointer update into the same transactional workflow as the trash-path delete so both sides can be rolled back on any failure.
   - Reference: Loop 4 acceptance criterion for `project delete`: if the deleted project was selected, `.ralph-burning/active-project` is cleared only after the delete succeeds; if delete fails after validation, the project remains addressable and the active-project pointer is unchanged.

## Recommended Improvements
1. Add a regression test for the restore-failure path itself. The current suite checks that `write_active_project` is attempted, but not that the spec guarantee still holds if that restore write fails.
