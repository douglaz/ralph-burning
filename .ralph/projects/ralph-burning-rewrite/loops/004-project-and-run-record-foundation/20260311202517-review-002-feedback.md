# Review: SUGGESTIONS

## Required Changes
1. **Run snapshot interpretation and semantic validation**: `run status` and `project delete` currently accept invalid canonical run state and mis-handle valid terminal snapshots.
   - Current: [`RunStatusView::from_snapshot` in queries.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/queries.rs#L19) treats any snapshot with `active_run = None` as `not started`, and [`FsRunSnapshotStore::read_run_snapshot` in fs.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/adapters/fs.rs#L503) only does JSON deserialization, not domain validation. I reproduced both cases with the built binary: `status = "completed"` plus `active_run = null` prints `Status: not started`, and `status = "running"` plus `active_run = null` still allows deletion because [`delete_project` in service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs#L200) only checks `has_active_run()`.
   - Expected: `run status` should report terminal persisted states even when no active run exists, and semantically inconsistent snapshots such as `running` or `paused` without an active cursor should fail fast as corrupt canonical state before status/delete queries proceed.
   - Reference: Loop 4 acceptance criteria for `run status`, `project delete`, and corrupt/inconsistent canonical files; master prompt §7 core invariants and §11 Project and Run Record Rules.

2. **Missing `project.toml` must fail fast instead of being hidden or downgraded to “not found”**: project record corruption is not surfaced according to the spec.
   - Current: [`FsProjectStore::project_exists` in fs.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/adapters/fs.rs#L197) and [`list_project_ids` in fs.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/adapters/fs.rs#L223) use `project.toml` presence as a filter, while [`read_project_record` in fs.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/adapters/fs.rs#L202) maps a missing file to `ProjectNotFound`. In a black-box repro, deleting `.ralph-burning/projects/alpha/project.toml` made `project show alpha` return “not found” and `project list` succeed with `No projects found.` instead of failing on the missing canonical file.
   - Expected: Missing or inconsistent `project.toml` should raise a file-specific corruption error, and `project list` should not silently skip directories whose canonical project record is incomplete.
   - Reference: Loop 4 acceptance criterion `Corrupt or inconsistent canonical files (project.toml, run.json, journal.ndjson, payload/artifact indexes) fail fast with file-specific errors.`
