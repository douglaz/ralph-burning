# Implementation Notes

## Decisions Made
- **Port/adapter architecture**: Introduced `ProjectStorePort`, `JournalStorePort`, `ArtifactStorePort`, `RuntimeLogStorePort`, `RunSnapshotPort`, and `ActiveProjectPort` as trait boundaries in `contexts/project_run_record/service.rs`, with `Fs*` filesystem-backed implementations in `adapters/fs.rs`. This enables fake-port testing without filesystem I/O.
- **Atomic project creation**: Uses a staging directory (`.{id}.staging.{pid}`) alongside the final project path, writes all canonical files there, then performs a single `rename()` to the final location. On any failure, the staging dir is cleaned up, ensuring no partial projects are visible to `project list` or `project show`.
- **Journal as NDJSON**: `journal.ndjson` uses one JSON object per line with monotonically increasing `sequence` numbers. Parsing validates sequence ordering and fails fast on gaps or corruption.
- **Prompt hash**: Used `std::hash::DefaultHasher` (SipHash) for a lightweight, deterministic prompt content hash. This is sufficient for integrity tracking; a cryptographic hash can be substituted later if needed.
- **Run snapshot separation**: `run.json` stores canonical run state independently. Status, history, and tail commands read from canonical files, never inferring from artifacts or Markdown. The `RunHistoryView` type has no runtime log field, enforcing separation at the type level.
- **Flow immutability**: `ProjectRecord.flow` is a plain field with no mutation methods. The `create_project` service writes it once; no CLI path or store method allows changing it afterward.
- **Active-project pointer semantics**: `project create` does not set the active-project pointer (preserving the existing `project select` responsibility). `project delete` clears the pointer only after successful deletion.
- **Error variants**: Added `DuplicateProject`, `CorruptRecord`, `InvalidPrompt`, `ActiveRunDelete`, `JournalSequence`, and `SerdeJson` to `AppError` for precise error reporting.

## Spec Deviations
- **Payload/artifact write atomicity**: The spec requires that if a payload write succeeds but its artifact companion fails, `run history` and `run tail` must not expose a partial record. This is enforced at query time via `validate_history_consistency()` which rejects orphaned artifacts. The actual atomic write of payload+artifact pairs will be implemented in the `run start` slice (loop 6) since this loop only adds the store models and enumeration APIs.
- **`run start`, `run resume`, `run rollback`**: These remain as `NotYetImplemented` stubs per the spec's "keep as later-slice work" directive.
- **Adapter contract tests for filesystem round trips**: The filesystem round-trip behavior is covered by the CLI integration tests (which create real projects via the binary and verify file contents). Dedicated adapter-only contract tests were not added separately since the CLI tests provide equivalent coverage with less indirection.
