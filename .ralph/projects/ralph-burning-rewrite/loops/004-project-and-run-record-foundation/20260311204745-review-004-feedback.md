# Review: SUGGESTIONS

## Required Changes
1. **Prompt reference durability**: `project.toml` records the source prompt path instead of the canonical copied prompt.
   - Current: [project.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/cli/project.rs#L97) forwards `args.prompt` into `CreateProjectInput.prompt_path`, and [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs#L91) persists that unchanged as `ProjectRecord.prompt_reference`. In a black-box check, creating with `--prompt external.md` produced `prompt_reference = "external.md"` and `project show` reported `Prompt reference: external.md`, even though the durable copy lives at `.ralph-burning/projects/<id>/prompt.md`.
   - Expected: Persist the copied prompt reference (`prompt.md` or an equivalent project-relative canonical path) so the durable project record stays self-contained after the source file moves or disappears.
   - Reference: Loop 4 acceptance criterion "`project.toml` persists immutable project metadata: `id`, `name`, fixed `flow`, copied prompt reference, prompt hash, creation timestamp, and durable status summary"; master prompt §6 storage layout.

2. **Delete failure addressability**: failed post-validation deletes can still strand a project outside its canonical path.
   - Current: [fs.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/adapters/fs.rs#L278) renames the project to `.<id>.deleting.<pid>`, then removes it; if removal fails and the restore rename also fails, the code returns the original error and explicitly leaves the project "stranded at the trash path". That violates the spec guarantee that a failed delete leaves the project addressable.
   - Expected: A delete that fails after validation must leave the project addressable at `.ralph-burning/projects/<id>` and keep the active-project pointer unchanged. The restore-failure branch needs a stronger transactional guarantee than best-effort rename-back.
   - Reference: Loop 4 acceptance criterion for `project delete`: if delete fails after validation, "the project remains addressable and the active-project pointer is unchanged."

3. **Filesystem adapter contract coverage**: the required adapter-level test slice is still missing.
   - Current: the test tree includes unit/domain/service coverage and CLI integration coverage, but there are no dedicated filesystem adapter contract tests for the `Fs*` stores under `tests/`.
   - Expected: Add adapter contract tests for filesystem round trips and corruption visibility for the project, run snapshot, journal, artifact, and runtime log adapters, as explicitly required by the loop 4 acceptance criteria.
   - Reference: Loop 4 acceptance criterion: "Add domain/unit tests ... service tests with fake ports; adapter contract tests for filesystem round trips and corruption visibility; and CLI integration tests for project CRUD plus run queries."
