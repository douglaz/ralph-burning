# Review: SUGGESTIONS

## Required Changes
1. **`project delete` is still not transactional across pointer cleanup**: the command can fail after validation with the project already gone.
   - Current: [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs#L226) deletes the project first, then [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs#L233) clears `.ralph-burning/active-project`. If `clear_active_project` fails, the command returns an error after the project has already been removed, and the new regression test [project_run_record_test.rs](/root/new-ralph-burning/ralph-burning-rewrite/tests/unit/project_run_record_test.rs#L653) explicitly locks in that behavior.
   - Expected: Any post-validation failure in `project delete` must leave the project addressable at `.ralph-burning/projects/<id>` and the active-project pointer unchanged. Restore the project on pointer-clear failure, or make delete plus pointer update one transactional workflow.
   - Reference: Loop 4 acceptance criterion for `project delete`: if delete fails after validation, “the project remains addressable and the active-project pointer is unchanged.”

2. **Empty `journal.ndjson` is accepted as valid canonical state**: journal corruption does not fail fast.
   - Current: [journal.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/journal.rs#L42) returns `Ok(Vec::new())` for an empty journal, and [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs#L177) plus [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs#L257) treat that as normal. In a black-box check, truncating `.ralph-burning/projects/alpha/journal.ndjson` made `run history` exit `0` with an empty journal section, and `project show alpha` exit `0` with `Journal events: 0`.
   - Expected: Treat an empty journal, or any journal whose first event is not `project_created` with sequence `1`, as a corrupt canonical file and fail fast in `project show`, `run history`, and `run tail`.
   - Reference: Loop 4 acceptance criteria for `journal.ndjson` initialization and for corrupt/inconsistent canonical files failing fast; master prompt §11 (“`journal.ndjson` is the authoritative event source”).
