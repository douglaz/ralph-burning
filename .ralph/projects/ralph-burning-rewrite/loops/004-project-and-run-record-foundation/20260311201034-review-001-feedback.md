# Review: SUGGESTIONS

## Required Changes
1. **Canonical run snapshot schema**: `run.json` does not persist the full canonical state required by the loop 4 spec.
Current: [`RunSnapshot` in model.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/model.rs:30) only stores `active_run`, `status`, and `status_summary`, so created `run.json` files contain only those three fields.
Expected: Persist the full canonical snapshot for this slice, including empty/default `cycle_history`, `completion_rounds`, rollback-point metadata, amendment-queue state, and the stage cursor state expected by the spec.
Reference: Loop 4 acceptance criterion for `run.json`; master prompt §11 "Project and Run Record Rules".

2. **Durable history completeness**: partial payload/artifact records are still exposed, and `run tail` does not render the full durable history.
Current: [`validate_history_consistency` in queries.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/queries.rs:103) only rejects orphaned artifacts, not orphaned payloads. I verified that a payload file without an artifact is shown by `run history`, while [`run tail` in run.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/cli/run.rs:149) prints only journal events and omits payload/artifact records entirely.
Expected: Treat payload + artifact as a paired durable-history unit in both directions, fail fast on either orphaned side, and make the durable section of `run tail` include the same payload/artifact history that `run history` exposes.
Reference: Loop 4 acceptance criteria for payload/artifact atomicity, `run history`, and `run tail`; master prompt §10 "Structured Stage Contracts".

3. **Fail-fast canonical reads**: missing canonical files are silently treated as empty state instead of invalid state.
Current: [`FsJournalStore::read_journal` in fs.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/adapters/fs.rs:336) returns `Ok(Vec::new())` when `journal.ndjson` is missing, and [`FsRunSnapshotStore::read_run_snapshot` in fs.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/adapters/fs.rs:477) returns `RunSnapshot::initial()` when `run.json` is missing. I reproduced this locally: deleting `run.json` still lets `run status` succeed as `not started`, and `project delete` then succeeds.
Expected: Missing or inconsistent canonical files should raise file-specific corruption/inconsistency errors, with no synthesized fallback state.
Reference: Loop 4 acceptance criterion "Corrupt or inconsistent canonical files ... fail fast"; master prompt §11.

4. **Delete failure semantics**: project deletion is not transactional enough to satisfy the spec guarantee that a failed delete leaves the project addressable.
Current: [`FsProjectStore::delete_project` in fs.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/adapters/fs.rs:249) calls `fs::remove_dir_all` directly. If deletion fails mid-way, [`delete_project` in service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs:200) has no way to preserve addressability beyond keeping the active-project pointer unchanged.
Expected: Delete via a reversible rename/move step or equivalent transactional handling so post-validation failure cannot strand the project in a partially removed state.
Reference: Loop 4 acceptance criterion for `project delete` failure behavior; [`project_records.feature`](/root/new-ralph-burning/ralph-burning-rewrite/tests/conformance/features/project_records.feature:49).
