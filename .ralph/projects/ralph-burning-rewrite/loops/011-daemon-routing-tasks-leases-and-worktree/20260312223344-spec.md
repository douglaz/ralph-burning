# Feature: Daemon Routing, Tasks, Leases, and Worktree Management

## Description
Implement the `automation_runtime` bounded context and daemon CLI commands, delivering the full daemon task lifecycle: routing resolution (command > label > repo default), durable task and lease persistence, git worktree isolation per task, reconciliation of orphaned state, and integration with the existing workflow engine for automated project execution. This covers delivery roadmap item #11: "Daemon routing, tasks, leases, watchers, rebase, interactive requirements path."

The daemon is a long-running foreground process that polls for task files, acquires worktree leases, dispatches work through the generalized `execute_run` / `resume_run` engine, and cleans up on completion. Tasks are created externally (by future GitHub watchers or manually) and placed in `daemon/tasks/`. The daemon picks up pending tasks, enforces invariants (one task per issue, one writer per project, one worktree per lease), and drives them to completion or failure.

## Acceptance Criteria
- [ ] `DaemonTask` model with lifecycle states: `pending`, `claimed`, `active`, `completed`, `failed`, `aborted` тАФ persisted as JSON under `.ralph-burning/daemon/tasks/<task-id>.json`
- [ ] `WorktreeLease` model with fields: lease ID, task ID, worktree path, branch name, acquired-at timestamp, TTL, heartbeat timestamp тАФ persisted under `.ralph-burning/daemon/leases/<lease-id>.json`
- [ ] **Routing engine** resolves flow preset from task metadata using strict precedence: (1) explicit `routing_command` field тЖТ (2) `routing_labels` array тЖТ (3) workspace `default_flow` config. Command and label values validated against built-in presets only; unknown values produce `RoutingResolutionFailed` error.
- [ ] **Label parser** accepts `rb:flow:<preset>` format (e.g., `rb:flow:standard`, `rb:flow:quick_dev`, `rb:flow:docs_change`, `rb:flow:ci_improvement`). Malformed labels are ignored with a warning; conflicting flow labels produce `AmbiguousRouting` error.
- [ ] **Task creation** via `DaemonTaskService::create_task()` validates: no existing active task for the same `issue_ref`, routing resolves to a valid flow preset, and the task file is atomically written.
- [ ] **Task claim** acquires a writer lock for the target project (advisory file lock under `daemon/leases/`), creates a worktree lease, and transitions task to `claimed`. If the writer lock is held, the task stays `pending` and is retried on next poll.
- [ ] **Worktree lifecycle**: on claim, create a git worktree at a deterministic path (`.ralph-burning/worktrees/<task-id>/`); on task terminal state, remove the worktree and release the lease. Worktree creation failure transitions task to `failed` with explicit error.
- [ ] **Task dispatch**: once claimed, the daemon creates a project (if needed) and invokes `execute_run` or `resume_run` from the workflow engine within the worktree context. The task transitions to `active` during execution and to `completed` or `failed` based on run outcome.
- [ ] **Heartbeat**: active tasks update `WorktreeLease.last_heartbeat` periodically. Reconciliation uses heartbeat staleness to detect orphaned leases.
- [ ] **Invariant: one writer lock per project** тАФ enforced by advisory lease file; attempting to claim a second task for the same project while a lease is held returns `ProjectWriterLockHeld`.
- [ ] **Invariant: one active daemon task per issue** тАФ enforced at task creation; `create_task` with a duplicate `issue_ref` where existing task is non-terminal returns `DuplicateTaskForIssue`.
- [ ] **Invariant: one worktree lease per task** тАФ enforced at claim; a task that already has a lease cannot acquire another.
- [ ] **`daemon start`** CLI: runs the daemon event loop. Polls `daemon/tasks/` for `pending` tasks at a configurable interval (default 10s). Processes one task at a time per poll cycle. Logs task lifecycle transitions. Exits cleanly on SIGINT/SIGTERM with lease cleanup.
- [ ] **`daemon status`** CLI: lists all tasks with their current state, associated lease (if any), and last heartbeat. Non-terminal tasks shown first.
- [ ] **`daemon abort <task-id>`** CLI: transitions a `claimed` or `active` task to `aborted`, cancels any in-progress backend invocation (via existing `CancellationToken`), releases the worktree lease, and removes the worktree. Aborting a terminal task returns an error.
- [ ] **`daemon retry <task-id>`** CLI: transitions a `failed` task back to `pending` (incrementing attempt count). Retrying a non-failed task returns an error.
- [ ] **`daemon reconcile`** CLI: scans leases for stale heartbeats (configurable TTL, default 5 minutes), removes orphaned worktrees, releases stale leases, and transitions any `claimed`/`active` tasks with stale leases to `failed` with `reconciliation_timeout` failure class. Reports actions taken.
- [ ] **Rebase support**: before dispatching a claimed task, the daemon attempts `git rebase` of the worktree branch onto the upstream default branch. If rebase fails (conflicts), the task transitions to `failed` with `rebase_conflict` details rather than silently proceeding with a stale base.
- [ ] **Journal integration**: daemon lifecycle events (task_created, task_claimed, task_completed, task_failed, task_aborted, lease_acquired, lease_released, reconciliation_run) are appended to a daemon-level journal at `.ralph-burning/daemon/journal.ndjson`.
- [ ] **Error variants** added to `AppError`: `RoutingResolutionFailed`, `AmbiguousRouting`, `DuplicateTaskForIssue`, `ProjectWriterLockHeld`, `WorktreeCreationFailed`, `WorktreeRemovalFailed`, `LeaseStale`, `TaskStateTransitionInvalid`, `RebaseConflict`.
- [ ] **Conformance scenarios** in `tests/conformance/features/daemon_routing.feature` covering: routing precedence (command > label > default), label parsing valid/invalid, conflicting labels, task lifecycle state machine, lease acquisition/release, duplicate task rejection, abort/retry semantics, reconciliation of stale leases.
- [ ] **Unit tests** for: routing resolution logic, label parser, task state machine transitions, lease TTL calculations, worktree path derivation.
- [ ] All existing tests continue to pass (272+ tests).
