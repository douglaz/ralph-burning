# Review: SUGGESTIONS

## Required Changes
1. **Claimed-task cleanup on pre-dispatch failures** [daemon_loop.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/daemon_loop.rs#L202) [daemon_loop.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/daemon_loop.rs#L214)
Current: After a task is claimed and its lease/worktree is created, `ensure_project(...)` and `mark_active(...)` still use `?` and can exit the daemon early without failing the task or releasing the lease. I reproduced this with an existing `standard` project plus a routed `docs_change` task on the same `project_id`: `daemon start --single-iteration` exited non-zero, left the task in `claimed`, and left both the lease JSON and writer lock on disk.
Expected: Any failure after claim must transition the task to `failed` and release/remove the lease and worktree before returning.
Reference: Loop 11 acceptance criteria for “Task dispatch” and “Worktree lifecycle”.

2. **`daemon abort` does not meet the active-task cleanup contract** [daemon.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/cli/daemon.rs#L132) [20260312225704-impl-notes.md](/root/new-ralph-burning/ralph-burning-rewrite/20260312225704-impl-notes.md#L9)
Current: The CLI only performs immediate lease/worktree cleanup when the original state was `claimed`. For `active` tasks it only marks the task `aborted` and returns; the implementation notes explicitly call this out as a spec deviation.
Expected: `daemon abort <task-id>` must cancel the in-flight execution and release/remove the lease and worktree for both `claimed` and `active` tasks.
Reference: Loop 11 acceptance criterion for `daemon abort <task-id>`.

3. **Polling cadence is not honored while work exists** [daemon_loop.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/daemon_loop.rs#L101)
Current: `poll_interval` is only slept when `process_cycle()` returns `false`. If pending tasks exist, the loop immediately starts another cycle, so backlog is drained back-to-back instead of polling at the configured interval.
Expected: `daemon start` should poll at the configured interval and process only one task per poll cycle, even when more pending tasks are queued.
Reference: Loop 11 acceptance criterion for `daemon start`.

## Recommended Improvements
1. Add CLI/integration coverage for aborting an `active` task and for post-claim failure cleanup. `nix develop -c cargo test` is green, so these spec gaps are currently escaping through missing coverage.
