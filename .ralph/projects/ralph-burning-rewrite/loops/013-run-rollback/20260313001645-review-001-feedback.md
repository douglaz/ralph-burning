# Review: SUGGESTIONS

## Required Changes
1. **Rollback branch retention**: Re-executing stages after a rollback must not reuse the same durable payload/artifact filenames.
   - Current: Stage history IDs are derived from `run_id + stage_id + cycle + attempt` in [engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs:1796), `run resume` intentionally keeps the original `run_id` and recreates the cursor with the same cycle and `attempt = 1` in [engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs:2478) and [engine.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs:2546), and the filesystem adapter writes those IDs to fixed `history/payloads/<id>.json` / `history/artifacts/<id>.json` paths in [fs.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/adapters/fs.rs:783). After a rollback/resume, the rerun targets the same file paths as the abandoned branch, so those records cannot remain on disk as distinct durable history.
   - Expected: Rolled-back branch records must stay on disk while only the visible branch is surfaced. That requires unique durable IDs per post-rollback execution branch or another storage scheme that preserves abandoned payload/artifact files instead of reusing the same canonical filenames.
   - Reference: Feature spec acceptance criterion: rolled-back payload/artifact records "remain on disk but are excluded from `run history` and `run status` queries by the journal's rollback event acting as a logical boundary."

2. **Rollback snapshot/journal consistency**: Logical rollback currently commits `run.json` before the journal boundary that makes history queries consistent with it.
   - Current: `perform_rollback()` writes the restored paused snapshot first in [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs:542), then appends `rollback_performed` afterward in [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs:555). If that append fails, `run status` reads the rolled-back snapshot while `run history` still derives from the pre-rollback journal branch in [service.rs](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/project_run_record/service.rs:397), leaving canonical state inconsistent.
   - Expected: A rollback must not expose a rolled-back snapshot without the corresponding `rollback_performed` journal event. Either persist the journal boundary before exposing the new snapshot, or add compensation so append failure restores the previous snapshot/state.
   - Reference: Feature spec acceptance criteria requiring `rollback_performed` to be appended and requiring `run history` / `run status` to reflect post-rollback state; prompt section 11 states `journal.ndjson` is the authoritative event source.
