# Feature: Run Rollback

## Description
Implement the `ralph-burning run rollback --to <target> [--hard]` command, which allows a user to rewind a project's run state to a previously-completed durable stage boundary. A *logical* (soft) rollback resets the stage cursor and run snapshot to the rollback point without touching the repository working tree. A *hard* rollback additionally performs a repository reset (via the worktree adapter) to the git state recorded at the rollback point. The core invariant "logical rollback must happen before repository reset" governs the ordering for hard rollback. Rollback points are created automatically at each durable stage boundary during run execution and persisted under `projects/<project-id>/rollback/`. The feature also adds automatic rollback-point creation to the workflow engine's stage-completion path so that every successful stage transition produces a recoverable checkpoint.

## Acceptance Criteria
- [ ] The workflow engine creates a `RollbackPoint` after every successful stage completion and persists it under `projects/<project-id>/rollback/<rollback-id>.json` via the filesystem adapter.
- [ ] Each rollback point captures the rollback ID (UUID), timestamp, completed stage ID, cycle number, and a snapshot of the run state at that boundary.
- [ ] `run rollback --to <stage>` (soft/logical) validates that the target stage has a persisted rollback point for the active project, resets `run.json` to the captured snapshot, appends a `rollback_performed` journal event, updates `rollback_point_meta` (increments `rollback_count`, sets `last_rollback_id`), and sets run status to `paused` so the user must explicitly `run resume` to continue.
- [ ] `run rollback --to <stage> --hard` performs the logical rollback first, then invokes the worktree adapter to reset the repository to the git commit SHA recorded in the rollback point. If the git reset fails, the logical rollback is still durable (invariant: logical before physical).
- [ ] Rollback is only allowed when the run is in `failed` or `paused` status. Attempting rollback on `running`, `not_started`, or `completed` runs fails with an explicit error message.
- [ ] The `--to` target must name a stage that belongs to the project's flow preset; specifying an invalid stage name or a stage with no rollback point fails with a descriptive error.
- [ ] Rollback to a stage discards payload and artifact records for all stages after the target (they remain on disk but are excluded from `run history` and `run status` queries by the journal's rollback event acting as a logical boundary).
- [ ] `run history` and `run status` correctly reflect post-rollback state, showing the rollback event and only pre-rollback-point history.
- [ ] Hard rollback records the git commit SHA in the rollback point; the `RollbackPoint` model gains an optional `git_sha: Option<String>` field populated by the engine when a stage completes.
- [ ] Rollback count and last rollback ID are monotonic across multiple rollbacks within the same run.
- [ ] A new Gherkin feature file `tests/conformance/features/run_rollback.feature` defines at least 8 scenarios covering: soft rollback success, hard rollback success, rollback from wrong status, rollback to invalid stage, rollback to stage without checkpoint, multiple sequential rollbacks, rollback then resume continues from target, and hard rollback with git-reset failure preserving logical state.
- [ ] Unit tests cover rollback domain validation (status check, stage membership, rollback-point lookup) and snapshot restoration logic.
- [ ] Adapter contract tests verify rollback-point persistence round-trip (write, list, read by stage).
