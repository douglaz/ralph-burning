## Required Changes
1. **Start and stage-entry failure boundaries**: `run start` still has an uncovered failure path that can leave canonical state inconsistent.
Current: [`engine.rs`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L188) writes the initial running snapshot, then appends `run_started` at [`engine.rs`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L190), then appends `stage_entered` at [`engine.rs`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L202), and only after that writes the stage-entry snapshot/status update at [`engine.rs`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L215). Those journal/snapshot writes are not wrapped in rollback or `run_failed` handling, so a failure after `stage_entered` but before stage commit can bubble out as a raw error and leave the run in `running` state instead of a persisted failed boundary.
Expected: failures in the `run_started` / `stage_entered` boundary need the same durability treatment as stage commit. If the failure is after `stage_entered`, the run must persist `failed` state at that stage boundary; if it is before `stage_entered`, the stage must not become visible.
Reference: acceptance criteria “On successful start...” and “Failure invariants are explicit...”

2. **Required CLI integration coverage**: the public test surface still does not match the spec’s required CLI coverage.
Current: [`cli.rs`](/root/new-ralph-burning/ralph-burning-rewrite/tests/cli.rs#L2471) verifies zero mutation on unsupported workspace version, not on an actual preflight backend failure. The only mid-stage failure coverage is unit-level in [`workflow_engine_test.rs`](/root/new-ralph-burning/ralph-burning-rewrite/tests/unit/workflow_engine_test.rs#L473) and [`workflow_engine_test.rs`](/root/new-ralph-burning/ralph-burning-rewrite/tests/unit/workflow_engine_test.rs#L539). The Gherkin scenario exists at [`run_start_standard.feature`](/root/new-ralph-burning/ralph-burning-rewrite/tests/conformance/features/run_start_standard.feature#L171), but there is still no CLI integration test for that path.
Expected: add CLI integration coverage for a real preflight failure with no mutation and for mid-stage failure without partial durable-history exposure. If the hardcoded stub backend prevents that, the binary/test harness needs a test-only injection seam.
Reference: acceptance criterion “Add CLI integration and Gherkin coverage for...”
