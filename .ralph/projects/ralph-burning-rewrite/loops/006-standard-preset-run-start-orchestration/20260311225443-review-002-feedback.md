## Required Changes
1. **Atomic stage-commit durability**: snapshot-write failure still leaks a committed stage into durable history.
   - Current: [`engine.rs`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L338) appends `stage_completed` before the next snapshot write, and on snapshot-write failure at [`engine.rs`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L388) it only removes the payload/artifact pair before calling `fail_run`. The comment at [`engine.rs`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/workflow_composition/engine.rs#L374) explicitly acknowledges that the `stage_completed` journal entry remains. The regression test at [`workflow_engine_test.rs`](/root/new-ralph-burning/ralph-burning-rewrite/tests/unit/workflow_engine_test.rs#L528) checks file rollback but does not assert journal rollback.
   - Expected: A stage must not become durably visible until payload, artifact, `stage_completed`, and the next `run.json` snapshot are all committed together. If snapshot or journal persistence fails, there must be no visible payload/artifact pair and no `stage_completed` event at all.
   - Reference: Loop 6 acceptance criteria “A stage is successful only after...” and “Failure invariants are explicit...”
