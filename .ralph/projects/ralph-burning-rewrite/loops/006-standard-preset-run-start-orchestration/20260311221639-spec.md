# Feature: Standard Preset Run Start Orchestration

## Description
Loops 1 through 5 are already complete in the approved specs and reflected in git checkpoints through the current loop-5 planning head: the rewrite has workspace/bootstrap commands, fixed flow presets, stage contracts and deterministic renderers, effective config plus active-project resolution, canonical project/run records and query commands, and the agent-execution facade with stub backend support. The next missing roadmap slice is step 6: implement the first real workflow engine behind `ralph-burning run start`.

This feature should add the straight-through `standard` preset execution path by wiring `workflow_composition`, `project_run_record`, and `agent_execution` together. It should execute enabled standard stages in canonical order, persist validated payloads and rendered artifacts at durable stage boundaries, and update `run.json` plus `journal.ndjson` as the source of truth. It should not yet add retry/remediation, resume, amendment-queue behavior, or non-standard flow execution.

## Acceptance Criteria
- [ ] `ralph-burning run start` validates workspace version, active project selection, canonical project record integrity, and canonical run snapshot integrity before any mutation.
- [ ] `run start` is allowed only when the project snapshot is `not_started` with no active run; running, paused, failed, or completed snapshots fail explicitly and never create a duplicate run.
- [ ] This slice supports only projects with flow `standard`; `quick_dev`, `docs_change`, and `ci_improvement` fail with an explicit unsupported-for-now error and leave all canonical state unchanged.
- [ ] The standard stage plan is derived from the existing flow definition in canonical order and honors `prompt_review.enabled`: when enabled, the run starts at `prompt_review`; when disabled, it starts at `planning` and writes no `prompt_review` history.
- [ ] Before persisting `run_started`, the engine resolves stage-to-role/backend/model targets for every stage it will execute and performs capability plus availability preflight. If any preflight check fails, `run.json`, `journal.ndjson`, payloads, artifacts, sessions, and the active-project pointer remain unchanged.
- [ ] On successful start, the engine creates exactly one new run ID and persists `run.json` with `status=running`, `active_run.stage_cursor` at the first executable stage, `cycle=1`, `attempt=1`, and `completion_round=1`, then appends `run_started` and first `stage_entered` journal events.
- [ ] Stage-to-role mapping is deterministic and centralized: `prompt_review`/`planning` use `Planner`, `implementation` uses `Implementer`, `qa`/`acceptance_qa` use `QaValidator`, `review` uses `Reviewer`, and `completion_panel`/`final_review` use `CompletionJudge`.
- [ ] Every stage invocation goes through `AgentExecutionService`; raw backend output is persisted only under `runtime/backend/`, and operational runtime logs are persisted only under `runtime/logs/`.
- [ ] A stage is successful only after the validated payload JSON, rendered Markdown artifact, `stage_completed` journal event, and updated `run.json` / next-stage cursor are committed as one logical durable boundary. If payload write succeeds but artifact, journal, or snapshot persistence fails, the stage remains uncommitted: no paired history record is visible, no `stage_completed` event exists, and the cursor stays at the prior durable boundary.
- [ ] On straight-through success with the stub backend, `run start` executes all enabled standard stages in order and ends with `run.json` status `completed`, `active_run=null`, `completion_rounds=1`, monotonic cycle/attempt values, and a terminal `run_completed` journal event persisted before any best-effort cleanup.
- [ ] `run history` and `run tail` after a successful run show one payload/artifact pair per completed stage in canonical stage order and never include raw backend output or runtime logs unless `run tail --logs` is used.
- [ ] This slice performs no retries. Any transport failure, schema-validation failure, domain-validation failure, timeout, cancellation, or non-passing QA/review outcome halts the run immediately and executes no later stages.
- [ ] Failure invariants are explicit: if failure happens before `stage_entered`, that stage produces no journal or durable-history records; if failure happens after `stage_entered` but before successful commit, that stage may retain raw diagnostics only, but it has no payload/artifact pair, no `stage_completed`, and the run persists a failed state at that stage boundary.
- [ ] Non-passing outcomes in `qa`, `review`, `completion_panel`, `acceptance_qa`, or `final_review` are terminal for this slice: the run is marked failed, no amendment-queue items are created, no completion round advances, and no remediation or resume logic is attempted.
- [ ] Runtime-log append failures are cosmetic only: they must not fabricate, remove, or reorder durable history, and they must not turn an already committed stage into a failed durable-history state.
- [ ] Add unit/service tests for stage-plan selection, preflight short-circuiting, happy-path completion, and atomic stage-commit failure handling.
- [ ] Add CLI integration and Gherkin coverage for: successful standard `run start`, prompt-review-disabled start, active-run rejection, unsupported-flow rejection, preflight failure with no mutation, and mid-stage failure without partial durable-history exposure.
- [ ] `cargo test` and `cargo clippy --all-targets --all-features -- -D warnings` pass.
