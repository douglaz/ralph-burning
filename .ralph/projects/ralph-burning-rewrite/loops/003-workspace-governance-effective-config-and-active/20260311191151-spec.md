# Feature: Workspace Governance Effective Config and Active Project Resolution

## Description
Implement roadmap slice 3: layered workspace configuration with `config show/get/set/edit` CLI commands, and active-project resolution via the `.ralph-burning/active-project` pointer file. The existing `workspace_governance` context currently handles workspace init and config loading but has no support for effective configuration (workspace-level defaults, runtime overrides), mutable config updates, or tracking which project is currently selected. This slice fills that gap so that downstream work (project/run canonical store in slice 4, agent execution in slice 5) can resolve effective settings and know which project commands target without explicit `--id` flags.

The config surface covers keys implied by the master prompt: `prompt_review.enabled` (controls whether the optional `prompt_review` stage runs in the `standard` flow), default backend family, default model overrides, and validation strictness. The implementation stores configuration in `workspace.toml` using a structured TOML schema, resolves effective values through a defaults-then-override chain, and enforces fail-fast on unsupported workspace versions (invariant already partially present, now wired through all config-reading paths).

## Acceptance Criteria
- [ ] `workspace.toml` schema is extended with an optional `[settings]` table supporting at minimum these keys: `prompt_review.enabled` (bool, default `true`), `default_flow` (string, must be a valid `FlowPreset`, default `"standard"`), `default_backend` (string, optional), `default_model` (string, optional). Unknown keys in `[settings]` are preserved on round-trip (forward-compatible editing).
- [ ] An `EffectiveConfig` struct in `workspace_governance` resolves the full configuration by merging persisted `workspace.toml` settings with compiled defaults. Every supported key has a typed accessor with a documented default value. `EffectiveConfig::load(base_dir)` is the single entry point for all config reads.
- [ ] `ralph-burning config show` loads the workspace, checks the version (fail-fast on mismatch), and prints all effective settings as a human-readable TOML-like table showing each key, its effective value, and its source (`default` or `workspace.toml`).
- [ ] `ralph-burning config get <key>` prints the effective value for a single key. Exits non-zero with a clear error for unknown keys.
- [ ] `ralph-burning config set <key> <value>` validates the key name and value type, writes the updated value into `workspace.toml` under `[settings]`, and prints a confirmation. Exits non-zero for invalid keys or values (e.g., setting `default_flow` to an unknown preset).
- [ ] `ralph-burning config edit` opens `workspace.toml` in `$EDITOR` (or `$VISUAL`, falling back to `vi`). After the editor exits, the file is re-validated; if validation fails, the user is warned on stderr and the command exits non-zero (but the file is not reverted — the user must fix manually).
- [ ] `active-project` is a plain-text file in `.ralph-burning/` containing a single project ID (trimmed, no newline normalization issues). When absent, commands that require an active project fail with an explicit message naming `project select`.
- [ ] `ralph-burning project select <id>` validates that the project directory `.ralph-burning/projects/<id>/` exists and contains `project.toml`, writes the ID to `active-project`, and prints a confirmation. Exits non-zero if the project does not exist.
- [ ] A `resolve_active_project(base_dir) -> AppResult<ProjectId>` function in `workspace_governance` reads and validates `active-project`, returning a typed `ProjectId` or a descriptive error. This function is the single resolution point for all commands that operate on the current project.
- [ ] All config and active-project commands check workspace version first and fail fast on unsupported versions (wiring the existing `ensure_supported_workspace_version` through the CLI path).
- [ ] The filesystem adapter (`adapters/fs.rs`) gains methods for atomic config writes (write-to-temp-then-rename) and active-project file read/write, keeping side effects behind the adapter boundary.
- [ ] Unit tests cover: `EffectiveConfig` default resolution, override merging, round-trip serialization of extended `workspace.toml`, unknown-key preservation, invalid key/value rejection, active-project read/write/missing cases.
- [ ] Integration tests cover: `config show` output format, `config get` known and unknown keys, `config set` valid and invalid mutations, `project select` with existing and non-existing project directories.
- [ ] Gherkin scenarios added to `tests/conformance/features/workspace_config.feature`: show effective config, get known key, get unknown key, set valid key, set invalid value, set invalid key, edit with valid result, edit with invalid result.
- [ ] Gherkin scenarios added to `tests/conformance/features/active_project.feature`: select existing project, select non-existing project, resolve when no active project set, resolve after valid select.
- [ ] `cargo test` and `cargo clippy --all-targets --all-features -- -D warnings` pass after all changes.
