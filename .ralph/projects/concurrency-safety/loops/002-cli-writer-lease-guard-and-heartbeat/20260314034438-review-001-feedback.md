## Required Changes
1. **Stale CLI lease reconciliation**: the core stale-lock recovery behavior from `prompt.md` is still missing, so `daemon reconcile` cannot recover a crashed CLI-held writer lock.
   Current: [`LeaseService::reconcile()`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/lease_service.rs#L257) still iterates `store.list_leases()` only, [`FsDaemonStore::list_leases()`](/root/new-ralph-burning/ralph-burning-rewrite/src/adapters/fs.rs#L1109) explicitly filters out `LeaseRecord::CliWriter`, and [`handle_reconcile()`](/root/new-ralph-burning/ralph-burning-rewrite/src/cli/daemon.rs#L187) still calls that unchanged worktree-only path. The new coverage in [`automation_runtime_test.rs`](/root/new-ralph-burning/ralph-burning-rewrite/tests/unit/automation_runtime_test.rs#L3534) and [`cli.rs`](/root/new-ralph-burning/ralph-burning-rewrite/tests/cli.rs#L4314) only exercises guard lifecycle and post-run cleanup, not stale CLI reconcile or end-to-end recovery.
   Expected: `daemon reconcile` must scan stale CLI lease records, remove the CLI lease file plus writer lock without requiring any task/worktree, count them in `stale_leases` and `released_leases`, keep `failed_tasks == 0`, and add the required reconcile and CLI/conformance tests proving a stale CLI lock is cleaned and `run start`/`run resume` succeeds afterward.
   Reference: `prompt.md` sections `Scope 1. CLI writer-lock recovery`, `Reconcile Behavior`, `Acceptance Criteria`, and `Tests`.

2. **Heartbeat shutdown ordering**: the guard does not actually stop the heartbeat task before cleanup, which violates the heartbeat shutdown requirement and can race with lease deletion.
   Current: the heartbeat task does synchronous read/write work in [`heartbeat_tick()`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/cli_writer_lease.rs#L105) after each sleep in [`CliWriterLeaseGuard::acquire()`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/cli_writer_lease.rs#L72), while [`Drop`](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/cli_writer_lease.rs#L120) only calls `notify_one()`, `abort()`, and then immediately deletes the lease and lock. If a tick is already inside the synchronous file I/O, `abort()` does not wait for it to finish, so it can still rewrite the lease after drop begins.
   Expected: shutdown must be deterministic before cleanup starts, either by awaiting task completion from an async-owned finalizer or by redesigning heartbeat/cleanup so a tick cannot run once drop/close has started.
   Reference: `prompt.md` section `CLI Lease Behavior` and the feature-spec acceptance criterion `Heartbeat shutdown is guard-owned`.
