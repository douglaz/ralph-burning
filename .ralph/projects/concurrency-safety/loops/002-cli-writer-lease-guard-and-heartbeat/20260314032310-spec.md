# Feature: CLI Writer Lease Guard and Heartbeat

## Description
Loop 1 and the latest planning checkpoint completed only the data/storage foundation: `CliWriterLease`, `LeaseRecord`, mixed lease-record store APIs, and backward-compatible serde are already in place, while `run start`/`run resume` still use the bare `"cli"` writer lock and `daemon reconcile` still ignores CLI leases. The next missing feature is a dedicated CLI writer-lease guard that wraps the existing project writer lock with a durable CLI lease record and periodic heartbeats, so long-running CLI runs stay reclaim-safe and leave a reconcile-visible stale lease after a crash. No daemon-loop behavior change belongs in this loop; the existing no-`set_current_dir` coverage remains a regression guard.

## Acceptance Criteria
- [ ] Add a dedicated CLI writer-lease helper/guard that uses the existing `CliWriterLease` and `LeaseRecord` storage path; it must not call `LeaseService::acquire()` and must not create daemon tasks, daemon journal entries, or worktrees.
- [ ] `run start` and `run resume` acquire the existing project writer lock through this guard before any run-state mutation and keep the current active-lock behavior, including surfacing `ProjectWriterLockHeld` for live contention.
- [ ] On successful acquisition, the guard generates one `lease_id`, writes that `lease_id` into the existing project writer-lock file, and persists a matching `CliWriterLease` record for the same project.
- [ ] If writer-lock acquisition fails, no CLI lease record is written, no heartbeat task is spawned, and the project snapshot/journal remain untouched.
- [ ] If writer-lock acquisition succeeds but CLI lease-record creation fails, the helper releases the writer lock before returning; invariant: the failed acquisition leaves behind neither a CLI lease record nor a writer lock.
- [ ] The guard owns a heartbeat task with TTL `300` seconds and cadence `30` seconds for the full lifetime of the CLI command; heartbeat ticks update only `last_heartbeat` on the existing CLI lease record.
- [ ] Heartbeat shutdown is guard-owned: on drop, the guard cancels heartbeat updates and waits for the heartbeat task to stop before cleanup begins.
- [ ] Heartbeat failure must be conservative: a failed heartbeat tick leaves the current CLI lease record and writer lock intact for later recovery, rather than partially cleaning either resource.
- [ ] Drop cleanup attempts CLI lease-record deletion and writer-lock release independently; invariant: failure in one cleanup sub-step must not skip the other, and normal exit, error unwind, and panic unwind all use the same RAII cleanup path.
- [ ] Focused unit coverage proves the guard creates a reconcile-visible `CliWriterLease`, heartbeat updates advance `last_heartbeat`, and drop cleanup removes both the lease record and writer lock on normal completion and injected run errors.
- [ ] Existing CLI writer-lock tests are updated to assert no CLI lease file remains after successful or failed `run start`/`run resume`, while preserving current contention assertions and user-visible behavior.
- [ ] No `std::env::set_current_dir` call is introduced into daemon-loop code; the existing daemon CWD-safety structural/runtime tests remain unchanged and green.
