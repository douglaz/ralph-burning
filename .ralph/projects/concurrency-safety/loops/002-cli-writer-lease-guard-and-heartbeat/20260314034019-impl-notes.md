# Implementation Notes

## Decisions Made
- Created `cli_writer_lease.rs` as a new module rather than extending `lease_service.rs`, because the spec explicitly prohibits calling `LeaseService::acquire()` from CLI paths and the CLI guard has fundamentally different lifecycle semantics (no worktree, no daemon task, no journal events).
- Used `Arc<dyn DaemonStorePort + Send + Sync>` for the guard's store reference to enable sharing with the spawned heartbeat task, which requires `Send + 'static`. This changes the `FsDaemonStore` usage in `run.rs` from a stack-local reference to an `Arc`, which is a minimal cost for async heartbeat support.
- Lease IDs use the format `cli-{uuid}` to distinguish CLI lease records from daemon worktree leases (`lease-{task_id}`), making them easily identifiable in the leases directory.
- Heartbeat shutdown uses `handle.abort()` in the `Drop` impl rather than blocking on the task, since `Drop` cannot `.await`. The `Notify` is signaled first as a cooperative shutdown hint, with `abort()` as the synchronous fallback to guarantee the heartbeat stops.
- Drop cleanup calls `remove_lease` and `release_writer_lock` independently with `let _ =` for each, ensuring failure in one doesn't skip the other — this satisfies the spec's invariant about independent cleanup sub-steps.
- The `lease_id()` accessor is exposed as a public method on the guard to support test assertions on the lease record identity.

## Spec Deviations
- None. All acceptance criteria are implemented as specified.

## Testing
- **Unit tests** (in-module `#[cfg(test)]`): 5 tests covering acquisition, heartbeat advancement, drop cleanup, failed lock contention, and staleness detection.
- **Integration unit tests** (`automation_runtime_test.rs`): 4 async tests proving reconcile-visible lease records, heartbeat progression, error-path cleanup, and no-leak on contention failure.
- **CLI integration tests** (`cli.rs`): 3 existing writer-lock tests extended with assertions that no `cli-*.json` lease files remain after successful `run start`, `run resume`, and failed `run start`.
- **Verification**: `cargo build`, `cargo test` (540 tests pass), and `cargo run -- conformance run` (218/218 scenarios pass) all succeed.
