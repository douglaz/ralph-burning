# Implementation Response (Iteration 2)

## Changes Made
1. **Atomic rename-to-staging writer-lock release** (`src/adapters/fs.rs`): Replaced the open→read→inode-check→drop-fd→`remove_file(path)` sequence with a four-phase approach that eliminates the post-check/pre-unlink TOCTOU gap:
   - **Phase 1 (read-only):** Open via fd, read contents, check owner. On mismatch, return immediately with no filesystem mutation.
   - **Phase 2 (atomic rename):** `fs::rename(path, staging)` atomically moves the lock to a staging path (`writer-<project>.lock.releasing`). No TOCTOU — rename is a single atomic syscall.
   - **Phase 3 (inode verification):** Compare `fstat(fd)` against `stat(staging)`. If inodes differ, a concurrent writer replaced the lock between phases 1 and 2; restore the replacement via `hard_link(staging, path)` and return `OwnerMismatch { actual_owner: "(replaced)" }`.
   - **Phase 4 (staging delete):** `remove_file(staging)`. The staging path is exclusively ours, so the delete operates on the exact file instance we verified. Zero gap between the final identity check and the unlink.

2. **Updated test hooks** (`src/adapters/fs.rs`): Replaced `post_read_hook` with `pre_rename_hook` (between content match and rename) and `post_verify_hook` (between final inode check and staging delete).

3. **Updated TOCTOU regression** (`release_writer_lock_detects_replaced_file_via_inode`): Uses `pre_rename_hook` to swap the lock before rename. Inode comparison detects the swap, restores via `hard_link`, returns `OwnerMismatch`. Asserts replacement survives.

4. **New post-check regression** (`release_writer_lock_post_check_replacement_survives`): Uses `post_verify_hook` to create a new lock after the final identity check. Asserts release returns `Released` (staging deleted) and the new lock at the canonical path is untouched.
