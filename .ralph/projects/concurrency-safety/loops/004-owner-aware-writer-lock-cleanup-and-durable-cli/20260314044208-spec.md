# Feature: Owner-Aware Writer-Lock Cleanup and Durable CLI Lease Teardown

## Description
Loop 1 already completed the CLI lease model/storage layer, and Loop 2 already completed the CLI lease guard, heartbeat, stale-CLI reconcile flow, and the no-`set_current_dir` regression coverage. Git checkpoint history then advanced through loop 3 completion, but final review explicitly restarted planning because two accepted amendments remain unimplemented in the current tree: shared writer-lock release still blindly unlinks `writer-<project>.lock`, and both CLI cleanup paths delete the durable CLI lease before proving the lock was released. This feature hardens the shared writer-lock teardown path so cleanup is ownership-checked and keeps the CLI lease durable until lock release succeeds, without changing heartbeat policy, task/worktree modeling, or daemon CWD behavior.

## Acceptance Criteria
- [ ] The shared writer-lock release path becomes owner-aware: it must take the expected `lease_id` (or equivalent owner token), read the current `writer-<project>.lock` contents, and remove the file only when the contents match.
- [ ] Lock-file absence and lock-owner mismatch are reported distinctly enough for callers and tests to treat both as cleanup failures; neither case may delete the current lock file.
- [ ] Every release of the shared project writer lock passes the same `lease_id` that acquired it, including CLI acquisition rollback, CLI guard drop, stale CLI reconcile, daemon worktree acquisition rollback, and `LeaseService::release`.
- [ ] `CliWriterLeaseGuard::drop` stops heartbeat activity first, then attempts owner-aware writer-lock release, and only deletes the CLI lease record after lock removal succeeds.
- [ ] Failure invariant for CLI guard cleanup: if writer-lock release returns already-absent, owner-mismatch, or I/O error, the current writer-lock file must remain untouched and the CLI lease record must remain on disk for later reconcile visibility.
- [ ] `daemon reconcile` validates the CLI lease `project_id` before any cleanup side effects, then performs owner-aware writer-lock release before lease deletion. Invalid project IDs, absent locks, owner mismatches, and I/O failures must increment `stale_leases`, must not increment `released_leases`, and must leave `failed_tasks` unchanged.
- [ ] Failure invariant for stale CLI reconcile: if lock release fails, only the CLI lease record remains durable and no task/worktree cleanup is attempted; if lock release succeeds but lease deletion fails, the writer lock stays released, `released_leases` does not increment, and the only leftover durable artifact may be the lease record.
- [ ] A normal stale CLI cleanup with a matching owner token still removes both the writer lock and CLI lease record, and a subsequent `run start` or `run resume` can reacquire the project writer lock.
- [ ] Existing live-lock contention behavior is preserved: healthy CLI and daemon writers still contend on the same project-level writer lock and still surface `ProjectWriterLockHeld` for active conflicts.
- [ ] `src/contexts/automation_runtime/daemon_loop.rs` remains free of `set_current_dir` call sites, and the existing daemon CWD-safety tests stay unchanged and green.
- [ ] Unit coverage proves: owner-matched release succeeds; owner-mismatch does not delete a replaced lock; CLI guard cleanup leaves the lease durable when lock release fails; stale CLI reconcile reports cleanup failure without deleting the lease on missing or mismatched lock ownership.
