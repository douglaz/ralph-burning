# Review: SUGGESTIONS

## Required Changes
1. **Writer-lock release still has a shared-staging race**: the new compare-and-delete path can still delete a newer lock because every releaser uses the same temporary path.
Current: [src/adapters/fs.rs#L1276](/root/new-ralph-burning/ralph-burning-rewrite/src/adapters/fs.rs#L1276) renames every lock to the fixed `writer-<project>.lock.releasing` path, and [src/adapters/fs.rs#L1342](/root/new-ralph-burning/ralph-burning-rewrite/src/adapters/fs.rs#L1342) assumes that staging path is “exclusively ours”. It is not. If releaser A moves its validated lock to staging, writer B can acquire a new canonical lock, then B’s own release can rename that new lock onto the same staging path before A reaches [src/adapters/fs.rs#L1351](/root/new-ralph-burning/ralph-burning-rewrite/src/adapters/fs.rs#L1351). A then unlinks B’s file without ever validating B’s contents, and callers in [cli_writer_lease.rs#L160](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/cli_writer_lease.rs#L160) and [lease_service.rs#L478](/root/new-ralph-burning/ralph-burning-rewrite/src/contexts/automation_runtime/lease_service.rs#L478) will treat that as a successful `Released`.
Expected: make the staging target unique per release attempt, or otherwise guarantee the final delete can only touch the exact inode that was validated. Add a regression for the post-verify window where a second releaser reuses the staging slot.
Reference: prompt/design requirement that the shared writer-lock cleanup “remove the file only when the contents match”, plus the acceptance criteria preserving replaced-lock safety and active writer contention semantics.
