# Implementation Response (Iteration 3)

## Changes Made
1. **Post-rename verification failure recovery** (`src/adapters/fs.rs`): Both `file.metadata()` (fstat on the open fd) and `fs::metadata(&staging)` failure paths now attempt `fs::rename(&staging, &path)` to restore canonical lock visibility before returning the error. Previously, these paths returned `Err` immediately, leaving the canonical `writer-<project>.lock` path gone and the lock stranded at the staging path. This satisfies the spec invariant that on I/O error the current writer-lock file must remain untouched (or at minimum, recoverable at the canonical path).

2. **AlreadyExists-only staging deletion on restore failure** (`src/adapters/fs.rs`): The `hard_link(&staging, &path)` error branch now distinguishes `AlreadyExists` (a newer writer already replaced the canonical lock — safe to delete the superseded staging copy) from other I/O errors (leave the staging file intact for later recovery). Previously, every `hard_link` failure deleted the staging copy, which could drop the only surviving lock artifact on an unrelated I/O error (e.g., ENOSPC, EIO).

3. **New `post_rename_hook` test hook** (`src/adapters/fs.rs`): Added a `post_rename_hook` to `release_lock_testing` that fires after the atomic rename-to-staging succeeds but before Phase 3 inode verification. This allows tests to manipulate the staging file to exercise verification failure paths.

4. **Post-rename verification failure regression** (`src/adapters/fs.rs` tests): Added `release_writer_lock_post_rename_verification_failure_preserves_lock` which uses `post_rename_hook` to delete the staging file and place a replacement lock at the canonical path. Asserts the function returns an error (metadata verification failed) and the replacement lock at the canonical path survives untouched.
