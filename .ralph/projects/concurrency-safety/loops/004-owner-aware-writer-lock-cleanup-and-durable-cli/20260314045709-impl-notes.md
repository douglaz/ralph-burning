# Implementation Notes

## Decisions Made
- Introduced a new `WriterLockReleaseOutcome` enum (with `Released`, `AlreadyAbsent`, `OwnerMismatch`) rather than extending the existing `ResourceCleanupOutcome`, since ownership semantics are specific to writer locks and don't apply to lease file deletion via `remove_lease`.
- Owner mismatch is treated as a cleanup failure in both `Idempotent` and `Strict` release modes — a mismatched lock is never a safe condition to silently accept.
- Added `writer_lock_owner_mismatch: bool` field to `ReleaseResult` so that worktree-lease reconcile can report mismatch distinctly from absence/I/O errors.
- The `OwnerMismatch` variant carries `actual_owner: String` so error reporting can show what's actually in the lock file.
- The `fs.rs` implementation uses read-then-compare-then-delete (not atomic CAS), which matches the existing lock model (O_EXCL create for acquisition). The read-delete race is handled by treating `NotFound` on delete as `AlreadyAbsent`.
