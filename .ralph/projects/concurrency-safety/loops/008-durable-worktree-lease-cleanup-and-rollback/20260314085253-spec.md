# Feature: Durable Worktree Lease Cleanup and Rollback Failure Propagation

## Description
Git checkpoints and completed loop artifacts show loops 1, 2, 4, and 6 are already implemented: mixed `LeaseRecord` storage, CLI writer-lease guard plus heartbeat, stale CLI reconcile, owner-aware writer-lock release, fail-closed lock-restore logic, explicit `CliWriterLeaseGuard::close()`, and the existing no-`set_current_dir` daemon guard. The remaining open work is the Round 3 hardening gap: `LeaseService::release()` still deletes a worktree lease before proving the writer lock was released, and both `CliWriterLeaseGuard::acquire()` and `LeaseService::acquire()` still discard rollback writer-lock release failures. The next feature should close that shared failure-mode hole so writer-lock release failure never hides durable recovery state and never surfaces as only the original acquisition error.

## Acceptance Criteria
- [ ] `LeaseService::release()` performs cleanup in this order: worktree removal, owner-aware writer-lock release, then worktree lease-file deletion.
- [ ] If worktree removal fails, `LeaseService::release()` must not attempt writer-lock release or lease-file deletion, and it must return `resources_released == false`.
- [ ] If writer-lock release returns `AlreadyAbsent`, `OwnerMismatch`, or I/O error after successful worktree removal, `LeaseService::release()` must not delete the worktree lease file, must return `resources_released == false`, and must preserve enough detail for callers to report the specific lock-release failure.
- [ ] If writer-lock release succeeds but worktree lease-file deletion fails, the writer lock stays released, the lease file remains durable, `resources_released == false`, and no compensating lock recreation is attempted.
- [ ] Partial worktree cleanup must remain fail-closed for callers: `LeaseReleased` journal entries and task lease-reference clearing must happen only when worktree removal, writer-lock release, and lease-file deletion all positively succeed.
- [ ] `CliWriterLeaseGuard::acquire()` must stop discarding rollback writer-lock release failures when CLI lease persistence fails. If rollback release succeeds, acquisition leaves behind neither CLI lease file nor writer lock. If rollback release fails, acquisition returns a combined error that preserves the lease-write failure and explicitly states that the writer lock may still be held.
- [ ] `LeaseService::acquire()` must stop discarding rollback cleanup failures. When worktree creation or worktree lease persistence fails after lock acquisition, rollback must attempt every applicable cleanup step and return a combined error that preserves the triggering failure plus any rollback failures.
- [ ] Any acquisition rollback path that fails to release the writer lock must explicitly report that the project writer lock may still be held, even if other rollback steps succeeded.
- [ ] Active contention behavior remains unchanged: live CLI or daemon writers still return `ProjectWriterLockHeld`, and the new combined-error path is used only for non-contention rollback failures.
- [ ] Add unit coverage for a worktree release where writer-lock release fails after worktree removal and assert the worktree lease file still exists and `resources_released == false`.
- [ ] Add unit coverage for CLI acquisition where lease-record persistence fails and rollback lock release also fails, and assert the returned error includes both failure causes and the "writer lock may still be held" warning.
- [ ] Add unit coverage for worktree acquisition where worktree creation or lease persistence fails and rollback lock release fails, and assert the returned error includes both the acquisition failure and rollback failure details.
- [ ] `src/contexts/automation_runtime/daemon_loop.rs` remains unchanged with respect to CWD handling and still contains no `set_current_dir` call sites.
