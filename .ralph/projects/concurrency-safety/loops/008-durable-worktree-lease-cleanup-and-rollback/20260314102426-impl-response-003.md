# Implementation Response (Iteration 3)

## Changes Made
1. **Fail-closed lease-file preservation on partial worktree cleanup** (`lease_service.rs:308-323`): Phase 3 lease-file deletion now requires `writer_lock_released && !worktree_already_absent` — when worktree was only `AlreadyAbsent`, the lease file stays durable on disk. This prevents a subsequent `cleanup_aborted_task()` from finding no lease via `find_lease_for_task()` and incorrectly clearing `task.lease_id`.

2. **Updated `acquire()` doc comment** (`cli_writer_lease.rs:123-129`): Now describes the `AcquisitionRollbackFailed` path instead of saying the writer lock is always released before returning.

3. **New regression test** `release_with_absent_worktree_preserves_lease_file_for_subsequent_lookups`: Covers the exact daemon handoff scenario — asserts lease file exists on disk, `find_lease_for_task()` still discovers it, and `task.lease_id` remains set after partial release.

4. **Strengthened 2 existing tests**: Both `release_idempotent_mode_missing_worktree_returns_partial` and `abort_cleanup_with_missing_worktree_returns_partial_in_idempotent_mode` now also assert lease file durability and `find_lease_for_task()` discoverability.
