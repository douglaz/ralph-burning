# Implementation Response (Iteration 3)

## Changes Made
1. **Fail-closed lease-file preservation on partial worktree cleanup** (`lease_service.rs:308-323`): Phase 3 lease-file deletion is now gated on `writer_lock_released && !worktree_already_absent` instead of just `writer_lock_released`. When the worktree was only `AlreadyAbsent`, the lease file remains durable on disk even though the writer lock was released. This prevents a subsequent `cleanup_aborted_task()` call from finding no lease via `find_lease_for_task()` and incorrectly clearing `task.lease_id` — closing the fail-closed gap identified in AC 5.

2. **Updated `acquire()` doc comment** (`cli_writer_lease.rs:123-129`): The doc comment now describes the `AcquisitionRollbackFailed` path: when rollback succeeds only the original error is returned, and when rollback fails the error preserves both the trigger and a warning that the writer lock may still be held.

3. **New regression test** `release_with_absent_worktree_preserves_lease_file_for_subsequent_lookups` (`automation_runtime_test.rs`): Sets up a task with lease and writer lock but no worktree, calls `release()` with `Idempotent` mode, and asserts: `resources_released == false`, lease file still exists on disk, `find_lease_for_task()` still discovers the lease, and `task.lease_id` remains set. This directly covers the daemon handoff scenario where a deleted lease file would have caused `cleanup_aborted_task()` to clear `lease_id`.

4. **Strengthened existing tests** (`automation_runtime_test.rs`): Both `release_idempotent_mode_missing_worktree_returns_partial` and `abort_cleanup_with_missing_worktree_returns_partial_in_idempotent_mode` now additionally assert lease file durability on disk and (for the abort test) that `find_lease_for_task()` still returns the lease.
