# Implementation Notes

## Decisions Made
- **Release cleanup reorder**: Changed `LeaseService::release()` from worktree → lease-file → writer-lock to worktree → writer-lock → lease-file. Lease-file deletion is now gated on writer-lock release returning `Released`; any other outcome (AlreadyAbsent, OwnerMismatch, I/O error) preserves the lease file for operator recovery.
- **Worktree removal errors captured, not propagated**: `release()` now returns `Ok(ReleaseResult { resources_released: false, worktree_error: Some(...) })` instead of `Err(...)` when worktree removal fails. This keeps the result contract uniform and lets callers use the `worktree_error` field for detailed reporting.
- **Writer-lock AlreadyAbsent is always a failure for resources_released**: Even in `Idempotent` mode, writer-lock AlreadyAbsent now makes `resources_released=false`. CLI abort tests were updated to create writer locks in their fixtures to match this stricter contract.
- **AcquisitionRollbackFailed error**: New typed error variant preserving both the original trigger failure and rollback failure details (including "writer lock may still be held" warning). Replaces the silent `let _ =` pattern in both `LeaseService::acquire()` and `CliWriterLeaseGuard::acquire()`.
- **Rollback attempts every applicable step**: In `LeaseService::acquire()`, when lease persistence fails, rollback attempts both worktree removal and writer-lock release, aggregating all failures into a single combined error.

## Spec Deviations
- The "both substeps absent/error" reconcile tests were updated to reflect the new cleanup ordering where lease-file deletion is skipped when writer-lock release fails. Previously these tests asserted both lease-file and writer-lock failures; now only writer-lock failure is reported since lease-file deletion is never attempted.
- CLI integration abort tests needed writer lock fixtures added because the stricter `resources_released` contract no longer treats AlreadyAbsent writer locks as success in Idempotent mode. This is consistent with AC #3.
