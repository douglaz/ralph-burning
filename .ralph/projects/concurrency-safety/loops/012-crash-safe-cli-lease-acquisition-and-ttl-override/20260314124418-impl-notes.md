# Implementation Notes

## Decisions Made
- **Reordered `CliWriterLeaseGuard::acquire()` to persist-before-lock.** The durable CLI lease record is now written _before_ acquiring the writer lock (Step 1: `write_lease_record`, Step 2: `acquire_writer_lock`). This guarantees the crash-safety invariant: a stranded writer lock always has a matching reconcile-visible lease record. On contention, the prewritten lease is cleaned up via `remove_lease` instead of the old `release_writer_lock` rollback path.
- **Introduced `saturating_heartbeat_deadline()` as the shared TTL helper.** Rather than exposing only a `u64`-to-`i64` conversion, the helper encapsulates both `chrono::Duration::try_seconds()` (which rejects values that overflow chrono's internal `TimeDelta` representation) and `checked_add_signed()` (which rejects durations that overflow `DateTime`). On either overflow, `DateTime::MAX_UTC` is returned, meaning the lease is never stale. This is used by both `lease_heartbeat_deadline()` (the per-lease-type `is_stale_at()`) and both reconcile TTL override paths.
- **Kept `saturating_ttl_seconds()` public.** The `u64`-to-`i64` saturating cast helper remains available for callers that need just the raw i64 conversion, though reconcile now uses the higher-level `saturating_heartbeat_deadline()` directly.
- **Replaced `CliAcquireRollbackFailStore` mock and test.** The old mock simulated `write_lease_record` failure + `release_writer_lock` rollback failure, which no longer applies with the reordered acquire. Replaced with two new mocks: `CliAcquireOrderTrackingStore` (proves ordering) and `CliContentionCleanupFailStore` (proves contention + `remove_lease` failure reports both causes).

## Spec Deviations
- The `saturating_ttl_seconds()` function signature returns `i64` as specified, but callers must still use `Duration::try_seconds()` (not `Duration::seconds()`) because chrono 0.4.44's `TimeDelta::seconds()` panics for values that exceed its internal bounds even below `i64::MAX`. The `saturating_heartbeat_deadline()` helper handles this transparently.
- The spec mentions "equivalent to the existing default lease deadline logic" for the saturating conversion. The implementation goes further: the existing `lease_heartbeat_deadline()` now also uses `try_seconds` + `checked_add_signed` via the shared helper, making it consistent with the override path and safe against extreme `ttl_seconds` values on the lease struct itself.

## Testing
- **`cli_acquire_persists_lease_before_writer_lock`** — Unit test using `CliAcquireOrderTrackingStore` that records `write_lease_record` and `acquire_writer_lock` calls, asserting lease persistence is the first operation.
- **`failed_lock_acquisition_leaves_no_lease_record`** — Existing inline test in `cli_writer_lease.rs` that now validates the new flow: prewritten lease is rolled back on contention, leaving no residual lease record.
- **`cli_contention_cleanup_failure_reports_both_causes`** — Unit test using `CliContentionCleanupFailStore` that fails both lock acquisition and `remove_lease`, asserting the returned `AcquisitionRollbackFailed` error preserves both the contention cause and the cleanup failure detail, and the competing writer's lock is unchanged.
- **`reconcile_oversized_ttl_override_does_not_reclaim_fresh_worktree_or_cli_lease`** — Unit test using `u64::MAX` as `ttl_override_seconds` with both a fresh worktree lease and a fresh CLI lease, asserting `stale_leases == 0`, `released_leases == 0`, `failed_tasks == 0`.
- **`cli_daemon_reconcile_oversized_ttl_does_not_reclaim_fresh_cli_lease`** — CLI integration test that runs `daemon reconcile --ttl-seconds 18446744073709551615` and proves a fresh CLI-held writer lock is not reclaimed.
- All 454 unit/integration tests pass, all 218 conformance scenarios pass, `daemon_loop.rs` contains no `set_current_dir` call sites.
