# Feature: Crash-Safe CLI Lease Acquisition and TTL Override Saturation

## Description
Loops 1, 2, 4, 6, 8, and 10 already delivered the explicit `CliWriterLease`/`LeaseRecord` model, the CLI heartbeat guard and explicit `close()`, owner-aware writer-lock release, durable worktree cleanup ordering, stale-CLI prune-on-missing-lock behavior, the `noexec`-safe CLI fixture, and the existing daemon CWD regression guard. Git checkpoint history after loop 10 only records loop 11 completion artifacts, and the live code still contains two round-5 gaps: `CliWriterLeaseGuard::acquire()` takes the writer lock before persisting the CLI lease, and `LeaseService::reconcile()` casts `ttl_override_seconds` directly to `i64` in both stale checks. The next feature should close those last safety holes by making CLI acquisition crash-safe and by saturating reconcile TTL overrides so oversized values cannot reclaim healthy leases.

## Acceptance Criteria
- [ ] `CliWriterLeaseGuard::acquire()` guarantees that a reconcile-visible CLI lease record exists before a CLI-held `writer-<project>.lock` can be stranded. A crash after lease persistence but before `acquire()` returns must never leave a CLI writer lock with no matching CLI lease record.
- [ ] The CLI path still uses the dedicated CLI lease flow only; it must not call `LeaseService::acquire()`, create daemon tasks, or create worktrees.
- [ ] The same `lease_id` is used for both the durable `CliWriterLease` record and the writer-lock file contents.
- [ ] On normal active contention, if writer-lock acquisition fails and prewritten-lease cleanup succeeds, `run start` and `run resume` still return `ProjectWriterLockHeld` and leave behind no new CLI lease record.
- [ ] Failed-contention cleanup is strict: if writer-lock acquisition fails and deleting the prewritten CLI lease returns `AlreadyAbsent` or any I/O error, the command returns a combined acquisition/cleanup error. State invariant: the competing writer’s lock file must remain untouched, no run-state mutation may occur, and the only leftover durable artifact may be the unused CLI lease record.
- [ ] If a post-lock acquisition step fails after the writer lock is acquired, cleanup must still use owner-aware writer-lock release against the same `lease_id` and must never orphan a writer lock without a durable CLI lease record.
- [ ] `LeaseService::reconcile()` uses a saturating TTL conversion for `ttl_override_seconds`, equivalent to the existing default lease deadline logic. Values above `i64::MAX` are treated as `i64::MAX` seconds rather than wrapping negative.
- [ ] Oversized TTL override invariant: `daemon reconcile --ttl-seconds` with any `u64` value greater than `i64::MAX` must not mark a fresh worktree lease stale, must not fail an active daemon task, and must not release a healthy CLI writer lock.
- [ ] Add focused unit coverage proving CLI lease persistence happens before writer-lock acquisition is attempted, and that contention rollback removes the prewritten lease when cleanup succeeds.
- [ ] Add focused unit coverage for contention plus prewritten-lease cleanup failure, asserting the returned error preserves both failure causes and the live competing writer lock content is unchanged.
- [ ] Add reconcile coverage for oversized TTL override with both a fresh worktree lease and a fresh CLI lease, asserting `stale_leases == 0`, `released_leases == 0`, and `failed_tasks == 0`.
- [ ] `src/contexts/automation_runtime/daemon_loop.rs` remains free of `set_current_dir` call sites and the existing CWD-safety tests remain unchanged.
