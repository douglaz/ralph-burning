# Feature: Final CLI Lease Shutdown and Fail-Closed Lock Restore

## Description
Loops 1, 2, and 4 already delivered the lease schema/storage layer, the CLI lease guard with heartbeat, stale CLI reconcile, owner-aware writer-lock cleanup, durable stale-lease handling, and the daemon CWD regression guard. The remaining missing work is the final hardening pass from round 2 of final review: `FsDaemonStore::release_writer_lock()` still uses `rename(&staging, &path)` in verification-error recovery branches, and `run start` / `run resume` still rely on implicit `Drop`, so a successful run can hide CLI lease teardown failures. This feature makes writer-lock recovery fail closed and adds an explicit fallible `CliWriterLeaseGuard::close()` path for normal CLI success exits. This should be the last feature before completion.

## Acceptance Criteria
- [ ] `FsDaemonStore::release_writer_lock()` no longer uses `rename(&staging, &path)` in any verification-error recovery path after the lock has been moved to staging.
- [ ] Verification-failure invariant: if canonical `writer-<project>.lock` has been recreated after staging, release returns an error or mismatch outcome and leaves that canonical lock untouched; the staged old lock may remain durable, but it must not replace or delete the canonical lock.
- [ ] Verification-failure invariant when canonical is still absent: restore may only succeed through a non-clobbering strategy that fails cleanly on `AlreadyExists`; if restore cannot prove exclusivity, return an error and leave the staged artifact durable.
- [ ] `CliWriterLeaseGuard` gains an explicit fallible shutdown API for normal completion (`close()` or equivalent) that stops heartbeat activity, performs owner-aware writer-lock release, and deletes the CLI lease record only after lock release positively succeeds.
- [ ] `Drop` remains best-effort cleanup for unwind/error paths only and becomes idempotent after a successful explicit close.
- [ ] `run start` and `run resume` call the explicit close path before printing success or returning `Ok(())`.
- [ ] Close failure invariant `writer_lock_absent`: return an error, keep the CLI lease record on disk, and do not recreate or silently clear any lock artifact.
- [ ] Close failure invariant `writer_lock_owner_mismatch` or writer-lock I/O error: return an error, keep the CLI lease record on disk, and do not delete or overwrite the current canonical or staged lock.
- [ ] Close failure invariant `lease_file_delete` after successful lock release: return an error, keep the writer lock released, keep the CLI lease record durable, and do not attempt to recreate the writer lock.
- [ ] If close fails after the run engine has already completed, the command exits non-zero and does not roll back snapshot, journal, or artifact state already committed by the run.
- [ ] Focused regression coverage proves both remaining gaps: verification-error recovery cannot clobber a newly reacquired canonical lock, and a successful `run start` / `run resume` surfaces explicit close failure instead of silently succeeding.
- [ ] Existing stale CLI reconcile accounting, live lock contention, and the no-`set_current_dir` daemon-loop guard remain unchanged.
