# Implementation Notes

## Decisions Made
- Replaced `fs::rename(&staging, &path)` in both verification-error recovery branches (`fd_meta` failure and `staged_meta` failure) with `fs::hard_link(&staging, &path)` followed by staging cleanup. `hard_link` fails with `AlreadyExists` when the canonical lock was recreated, making recovery fail-closed instead of clobbering. This matches the pattern already used by the inode-mismatch recovery branch.
- Added `close()` as a consuming method (`pub fn close(mut self) -> AppResult<()>`) that takes ownership of the guard, ensuring it can only be called once. The shared logic lives in `close_inner(&mut self)` which is also called by `Drop`. Idempotency is achieved by checking the `closed` atomic flag.
- Used `GuardCloseFailed { step, details }` error variant to distinguish close sub-step failures (`writer_lock_absent`, `writer_lock_owner_mismatch`, `writer_lock_io_error`, `lease_file_delete`) so CLI callers and tests can match on the specific failure mode.
- Added a runtime test seam (`RALPH_BURNING_TEST_DELETE_LOCK_BEFORE_CLOSE` env var) in `handle_start` only (not `handle_resume`) to exercise close-failure at the CLI level. This follows the existing env-var-based test seam pattern used for backend failure injection.
- The `close()` method on success calls `remove_lease` only after lock release positively returns `Released`. All other outcomes (absent, mismatch, I/O error) preserve the lease record durable for later reconcile visibility.

## Spec Deviations
- The `RALPH_BURNING_TEST_DELETE_LOCK_BEFORE_CLOSE` seam is only wired in `handle_start`, not `handle_resume`, since both paths use the identical guard and close logic — one test is sufficient for the regression without duplicating the seam.
- The verification-error recovery `AlreadyExists` branch is difficult to reach through the full `release_writer_lock` flow because it requires `fstat` or `stat` to fail while the files still exist. The regression test instead exercises the non-clobbering invariant through the inode-mismatch path (using pre-rename + post-rename hooks to create a scenario where canonical is recreated between rename and verification), which uses the same `hard_link` mechanism and proves the AlreadyExists guard.

### New tests added:
1. **`verification_error_recovery_does_not_clobber_reacquired_canonical`** (fs.rs internal) — proves that when canonical is recreated between rename-to-staging and inode verification, `hard_link` fails with `AlreadyExists` and the new canonical lock is preserved.
2. **`cli_guard_close_succeeds_and_drop_is_noop`** — normal close succeeds, both lock and lease cleaned, subsequent drop is idempotent.
3. **`cli_guard_close_fails_when_lock_absent`** — close returns `writer_lock_absent` error, lease record stays durable.
4. **`cli_guard_close_fails_when_lock_owner_mismatch`** — close returns `writer_lock_owner_mismatch` error, lease stays durable, usurper's lock untouched.
5. **`cli_guard_close_lease_delete_failure_keeps_lock_released`** — when lease file delete fails after successful lock release, close returns `lease_file_delete` error, but lock stays released.
6. **`cli_run_start_close_failure_exits_nonzero`** (CLI integration) — a successful run with injected close failure exits non-zero and preserves the CLI lease record.
