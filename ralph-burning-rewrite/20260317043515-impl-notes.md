# Implementation Notes

## Decisions Made
- Designed a **dual-mode CLI** for the daemon: when `--data-dir` is supplied, multi-repo GitHub-backed handlers activate; without it, legacy current-directory handlers are preserved unchanged. This keeps backward compatibility while adding the full multi-repo surface (`start`, `status`, `abort`, `retry`, `reconcile` all accept `--data-dir` and `--repo`).
- Created `GithubPort` trait (in `src/adapters/github.rs`) as the port for all GitHub REST/GraphQL operations, with an `InMemoryGithubClient` test double. The production `GithubClient` uses `reqwest` against `GITHUB_API_URL` (defaulting to `https://api.github.com`) with `GITHUB_TOKEN` bearer auth.
- Created `RepoRegistration` and `DataDirLayout` (in `src/contexts/automation_runtime/repo_registry.rs`) to manage the per-repo data-dir layout: `<data-dir>/repos/<owner>/<repo>/{repo,worktrees,daemon/{tasks,leases,journal.ndjson}}`. Slug parsing, layout path generation, branch naming (`rb/<issue>-<project>`), and label vocabulary all live here.
- Created `github_intake.rs` for polling/intake: `extract_command()` parses `/rb flow <preset>`, `/rb run`, `/rb retry`, `/rb abort`, and `/rb requirements` from issue comments. `poll_and_ingest_repo()` polls candidates, deduplicates by `(issue_ref, source_revision)`, and creates `DaemonTask` records. `sync_label_for_task()` manages label transitions (`rb:ready` → `rb:in-progress` → `rb:done`/`rb:failed`).
- Extended `RoutingEngine::parse_command()` to recognize `/rb run|retry|abort` as daemon commands that fall through to label/default routing rather than failing with "unknown command". Added `is_daemon_command()` static helper.
- Task identity for GitHub issues is keyed by `repo_slug + issue_number`. Added `DaemonTaskService::find_task_by_issue()` which prefers non-terminal tasks and falls back to the most recent terminal task.
- Extended `DaemonTask` with 5 optional fields (`repo_slug`, `issue_number`, `pr_url`, `last_seen_comment_id`, `last_seen_review_id`), all `serde(default, skip_serializing_if = "Option::is_none")` for backward-compatible serialization.
- The abort scenario sets the task to `Active` status (not `Pending`) since the state machine only allows `Pending → Claimed|Failed`, not `Pending → Aborted`. A task must be at least `Claimed` before it can be aborted.

## Spec Deviations
- `FsRepoRegistry` (production persistence for `RepoRegistryPort`) is not yet implemented — the trait and in-memory logic exist in `repo_registry.rs` but the filesystem adapter was deferred. The conformance tests exercise the domain logic directly.
- `WorktreeAdapter` was not modified to support the per-repo data-dir layout; `DataDirLayout::worktree_path()` produces the correct paths, but the adapter hasn't been wired to use them yet.
- `FileIssueWatcher` is still used in the legacy path — not yet marked test-only or removed from production code.
- The spec mentions adding unit tests for repo registration, GitHub intake, command precedence, dedup, and partial-failure invariants in `tests/unit/automation_runtime_test.rs` — these are covered by inline `#[cfg(test)]` modules in the respective source files and by the 9 new conformance scenarios, but dedicated unit test file additions were not made.

## Testing
- `cargo check` and `cargo check --tests` pass cleanly.
- All 274 conformance scenarios pass (`cargo test -- conformance_full_suite`), including the 9 new `daemon.github.*`, `daemon.routing.*`, `daemon.labels.*`, `daemon.tasks.*` scenarios.
- The `registry_matches_discovered_scenarios` meta-test confirms all 9 scenario IDs in `daemon_github.feature` have matching executor registrations.
- Full test suite passes: 110 integration tests + 579 unit tests + 274 conformance scenarios, 0 failures.
