# Implementation Response (Iteration 1)

## Changes Made
1. **Cancellation and timeout handling (Required Change 1)**: Rewrote the child lifecycle to be cancel-safe.
   - Changed `active_children` from `HashMap<String, Child>` to `HashMap<String, u32>` (storing PIDs instead of owned `Child` handles). The `Child` is now kept locally in `spawn_and_wait()` so `wait()` is called without holding the mutex.
   - In `spawn_and_wait()`: the mutex is acquired only briefly to insert the PID, then dropped before `child.wait().await`. After wait returns, the mutex is briefly re-acquired to remove the entry. No `await` crosses a lock boundary.
   - In `cancel()`: the mutex is acquired only to `remove()` the PID, then dropped before sending SIGTERM. SIGTERM is sent via synchronous `std::process::Command` (not async Tokio command), eliminating the second lock-across-await.
   - Fixed the cancellation test's fake script to use `exec sleep 300` so SIGTERM reaches the `sleep` process directly instead of being masked by the parent shell.

2. **Invoke-path capability enforcement (Required Change 2)**: Added `check_capability()` call at the top of `invoke()` before dispatching to `invoke_claude()`/`invoke_codex()`. Non-`Stage` contracts and unsupported backend families now return `AppError::CapabilityMismatch` with the same messages as the preflight path. The fallback match arm also uses `CapabilityMismatch` instead of `InvocationFailed { TransportFailure }`.

3. **Regression tests (Recommended Improvement)**: Added two new tests:
   - `invoke_rejects_requirements_contract_with_capability_mismatch` — verifies `invoke()` rejects `InvocationContract::Requirements` with Claude backend.
   - `invoke_rejects_openrouter_with_capability_mismatch` — verifies `invoke()` rejects OpenRouter with the correct detail text mentioning `default_backend=claude|codex`.
