## Known Issues

Live backend smoke was executed on 2026-03-19 against all three backends. All three
rows FAIL during `project bootstrap` (quick requirements phase), before any project
or run state is created. The harness correctly isolates workspace state and records
evidence. Specific bugs found and fixes applied:

1. **Claude (row 1)**: Claude CLI returns null `structured_output` and empty `result`
   for the `requirements:project_seed` contract despite `--json-schema` being provided.
   **Root cause correction** (iteration 5): requirements stages use
   `SessionPolicy::NewSession` (`service.rs:1893`), NOT `--resume`. The prior diagnosis
   blaming session-resume + `--json-schema` interaction was incorrect. The `--json-schema`
   flag is correctly passed on every fresh invocation. The failure is a Claude CLI-side
   issue with the `project_seed` schema. **Fix applied**: Claude decoder now provides
   detailed diagnostics (contract label, stdout length, session policy) instead of the
   generic "invalid Claude result JSON" error, enabling faster root-cause identification.
   Requires re-run to determine if this is an intermittent CLI issue or a persistent
   schema-size problem.

2. **Codex (row 2)**: OpenAI strict-mode schema validation rejects `RequirementsReviewPayload`
   because `follow_ups` is listed in `properties` but not in `required`. **Fix applied**:
   `inject_additional_properties_false()` has been replaced by `enforce_strict_mode_schema()`
   in `process_backend.rs`, which now ensures ALL property keys from `properties` are
   included in the `required` array. This handles schemas generated by `schemars` for types
   with `#[serde(default)]` fields (e.g. `follow_ups` on `RequirementsReviewPayload`,
   `RevisionFeedback`). Requires re-run to verify PASS.

3. **OpenRouter (row 3)**: Transient `HTTP 502: Provider returned error` from the upstream
   model provider during the `requirements:requirements_review` contract. **Fix applied**:
   `OpenRouterBackendAdapter::request_body()` now applies `enforce_strict_mode_schema()`
   to the JSON schema before sending with `strict: true`, fixing the same `required` array
   issue that would have blocked OpenRouter even after the 502 was resolved. The adapter
   previously passed the raw `schemars` schema with `strict: true` without enforcing the
   required-completeness constraint. Requires re-run with a stable upstream to verify PASS.

The harness now configures single-backend workspaces (all roles overridden to the backend
under test) to avoid mixed-backend panel failures during smoke.
