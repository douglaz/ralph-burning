# Implementation Notes

## Decisions Made
- Replaced the hardcoded requirements history IDs in `service.rs` with round-aware formats: question-set IDs use the round being generated (`run.question_round + 1`), while draft/review IDs use an effective round of `max(run.question_round, 1)` so quick mode and the empty-question fast path still produce `-1`.
- Updated the committed question boundary bookkeeping to store the computed question round in `run.question_round`, include that round in the awaiting-answers status summary, and restore the previous `question_round` and `latest_question_set_id` if the post-write journal append fails.
- Added an end-to-end regression test that drives `draft()` through `answer()` and asserts a round-2 run persists `draft-2` and `review-2` history files instead of reusing `-1`.
