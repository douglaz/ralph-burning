# Implementation Response (Iteration 2)

## Changes Made
1. **Atomic stage-commit durability fix** (`src/contexts/workflow_composition/engine.rs`):
   - Reordered the stage commit sequence from payload/artifact → journal → snapshot to **payload/artifact → snapshot → journal**. Now the `stage_completed` journal event is appended only after both the payload/artifact pair and the updated snapshot cursor are durably written. If the snapshot write fails, `stage_completed` is never appended, eliminating the journal leak identified in the review.
   - On snapshot failure: the payload/artifact pair is rolled back, the snapshot is restored to its pre-commit state via `fail_run` (which overwrites with failed status), and no `stage_completed` event exists.
   - On journal failure (after successful snapshot): the payload/artifact pair is rolled back, and `fail_run` overwrites the snapshot to failed state, so no partially-committed stage is visible.

2. **Unit test journal-leak assertions** (`tests/unit/workflow_engine_test.rs`):
   - Renamed `snapshot_failure_after_stage_completed_rolls_back_and_fails_run` to `snapshot_failure_during_stage_commit_rolls_back_without_journal_leak` to reflect the reordered commit sequence.
   - Added explicit assertion that no `stage_completed` events exist in the journal after snapshot failure — this directly verifies the spec's "no paired history record is visible" invariant.
   - Added matching `stage_completed` absence assertion to `journal_failure_after_payload_rolls_back_and_fails_run`.
   - Fixed pre-existing clippy `io_other_error` warnings in the `FailingJournalStore` and `FailingSnapshotWriteStore` test doubles.

3. **CLI integration tests** (`tests/cli.rs`):
   - Added `run_start_with_prompt_review_disabled_produces_seven_stages`: sets `prompt_review.enabled=false` via `config set`, runs `run start`, and verifies 7 payloads, no `prompt_review` stage in journal, and completed status.
   - Added `run_start_preflight_failure_leaves_state_unchanged`: corrupts workspace version to 999 to trigger pre-engine validation failure, then asserts byte-identical `run.json`, byte-identical `journal.ndjson`, and zero payloads — verifying no state mutation on pre-engine failure.

4. **Gherkin conformance scenarios** (`tests/conformance/features/run_start_standard.feature`):
   - Added SC-START-018: Prompt-review-disabled run start completes with 7 stages, no prompt_review events in journal, first stage is planning.
   - Added SC-START-019: Preflight failure leaves all canonical state byte-identical to pre-run state, no payloads/artifacts/sessions created.
   - Added SC-START-020: Mid-stage failure exposes no partial durable history — no payload/artifact files, no `stage_completed` event, journal ends with `run_failed`.

## Could Not Address
- CLI-level mid-stage failure test: the binary hardcodes `StubBackendAdapter::default()` which always succeeds, so injecting a persistence failure during stage commit is not possible without adding test-only hooks to the binary. This path is fully covered by unit tests (`journal_failure_after_payload_rolls_back_and_fails_run` and `snapshot_failure_during_stage_commit_rolls_back_without_journal_leak`) and the Gherkin scenario SC-START-020 defines the expected behavior for future integration when non-stub backends are available.
