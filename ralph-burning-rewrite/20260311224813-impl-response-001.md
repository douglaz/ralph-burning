# Implementation Response (Iteration 1)

## Changes Made
1. **Durable start/stage boundaries (Required Change 1)**: Reordered the run-start boundary so `run.json` with `Running` status is persisted _before_ `run_started` and `stage_entered` journal events (`engine.rs:172-193`). For stage commits, journal-append or snapshot-write failures after a successful payload/artifact write now: (a) roll back the payload/artifact pair via `remove_payload_artifact_pair`, (b) call `fail_run` to persist `run_failed` journal event and failed snapshot, ensuring no partial durable history is visible (`engine.rs:350-408`). Added `remove_payload_artifact_pair` to `PayloadArtifactWritePort` trait (`service.rs:104-113`) and implemented it in `FsPayloadArtifactWriteStore` (`fs.rs:684-703`).

2. **Final review routing (Required Change 2)**: Changed `BackendRole::for_stage` so `StageId::FinalReview` maps to `CompletionJudge` instead of `Reviewer` (`domain.rs:251`). Updated the `role_mapping_is_deterministic` test assertion to expect `CompletionJudge` for `FinalReview` (`workflow_engine_test.rs:92`).

3. **Failing-port tests (Recommended Improvement)**: Added `FailingJournalStore` and `FailingSnapshotWriteStore` test fakes that delegate to the real FS stores but inject failures on the Nth call. Added two new tests:
   - `journal_failure_after_payload_rolls_back_and_fails_run` — verifies journal-append failure after payload/artifact write triggers rollback and failed state
   - `snapshot_failure_after_stage_completed_rolls_back_and_fails_run` — verifies snapshot-write failure after `stage_completed` journal triggers rollback and failed state
