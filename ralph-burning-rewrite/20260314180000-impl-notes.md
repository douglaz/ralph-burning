# Implementation Notes

## Decisions Made
- Used file-based approach for fake CLI test scripts: envelope/payload JSON is written to files and `cat`-ed by fake scripts, avoiding shell escaping issues with embedded JSON in heredocs
- Used `tokio::process::Command::new("kill")` for SIGTERM delivery instead of `libc::kill` because `#![forbid(unsafe_code)]` is set project-wide; non-zero kill exit (process already gone) is treated as Ok since the spec says "already exited races" should succeed
- Took stdout/stderr handles from the child before registering it in `active_children`, then spawned async tasks to read them concurrently with `child.wait()` — this avoids the `wait_with_output()` ownership issue while keeping the child registered for cancel
- Added `fs` and `io-util` features to tokio for `tokio::fs` (Codex temp file I/O) and `AsyncReadExt`/`AsyncWriteExt` (stdin/stdout piping)
- Session metadata follows backend rules exactly: Claude prefers envelope `session_id` with fallback to prior session when resuming; Codex sets session_id only when `exec resume` was used
- Cancel removes child from `active_children` immediately after signal attempt, regardless of kill outcome, preventing stale entries

## Spec Deviations
- The spec's acceptance criterion says "the adapter must not add a second timeout wrapper" — this is satisfied by design; `invoke()` has no timeout logic, relying entirely on `AgentExecutionService`'s existing `tokio::time::timeout` wrapper
- The spec mentions spawn failures should leave `active_children` unchanged — implemented: if `Command::spawn()` fails, the child is never inserted into the map
- The capability mismatch detail text for unsupported families was enhanced to say "self-hosted workflow runs require default_backend=claude or default_backend=codex" per the acceptance criteria
