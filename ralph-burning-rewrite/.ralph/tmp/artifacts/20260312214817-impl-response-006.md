# Implementation Response (Iteration 6)

## Changes Made

1. **`requirements show` no longer reports stale pending questions after the answer boundary** (Required Change 1)
   - Refined the `pending_question_count` derivation in `show()` (`service.rs` lines 318-340) so that for `Failed` runs, pending questions are only surfaced when the latest durable boundary is still the committed question set.
   - If `latest_draft_id` is set (draft committed past question boundary), `pending_question_count` returns `None`.
   - If the journal contains an `AnswersSubmitted` event (answers durably submitted even though the run later failed), `pending_question_count` returns `None`.
   - Only when neither condition holds does `show()` surface the stored `pending_question_count` for failed runs.
   - Added two regression tests: `show_does_not_report_pending_questions_after_answer_boundary` (simulates `AnswersSubmitted` + `RunFailed` journal state at `question_round: 2`) and `show_does_not_report_pending_questions_when_draft_committed` (simulates `latest_draft_id` set on a failed run).

2. **Seed rollback now persists terminal state before cleanup** (Required Change 2)
   - Reordered both seed-failure branches (`service.rs` lines 681-688 and 700-704) so that `fail_run()` is called **before** `remove_seed_pair()`.
   - This ensures that if the process crashes between the terminal state transition and seed file cleanup, canonical state is already `Failed` rather than appearing non-terminal with missing seed files.
   - Added `seed_rollback_persists_failed_state_before_cleanup` regression test verifying the completed run's seed file and history layout.

3. **Gherkin scenarios** — RD-024 (stale pending questions after answer boundary) and RD-025 (seed rollback terminal-transition ordering).
