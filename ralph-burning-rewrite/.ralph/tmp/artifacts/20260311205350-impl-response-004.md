# Implementation Response (Iteration 4)

## Changes Made
1. **Prompt reference durability** — Changed `create_project` in `service.rs` to persist `"prompt.md"` as the canonical `prompt_reference` instead of forwarding the source `prompt_path`. The durable project record is now self-contained and independent of the original source file location. Added a CLI test (`project_create_records_canonical_prompt_reference_not_source_path`) that creates with a non-standard source filename and verifies `project.toml` records `prompt_reference = "prompt.md"` with no trace of the source path.

2. **Delete failure addressability** — Strengthened the restore-failure branch in `FsProjectStore::delete_project` in `fs.rs`. When `remove_dir_all` fails and the rename-back also fails (e.g., due to partial removal leaving inconsistent state), the code now falls back to a recursive copy (`copy_dir_recursive`) to restore the project at its canonical path before returning the error. If even the copy fallback fails, a compound `CorruptRecord` error is returned that clearly describes both the removal and restore failures. This ensures the spec guarantee that a failed delete leaves the project addressable.

3. **Filesystem adapter contract tests** — Added `tests/unit/adapter_contract_test.rs` with 30 dedicated adapter contract tests covering:
   - `FsProjectStore`: round-trip create/read, exists, list (sorted), delete, missing `project.toml` corruption on read/exists/list, malformed TOML corruption, no leftover staging dirs, canonical prompt reference persistence
   - `FsJournalStore`: round-trip read, append-and-read, missing file corruption, corrupt JSON
   - `FsRunSnapshotStore`: round-trip, missing file corruption, corrupt JSON, semantic inconsistency
   - `FsArtifactStore`: empty dirs, payload round-trip, artifact round-trip, corrupt payload/artifact JSON
   - `FsRuntimeLogStore`: empty, NDJSON read, malformed line skipping
   - `FsActiveProjectStore`: read none, round-trip write/read, clear, clear-when-absent
