# Implementation Response (Iteration 4)

## Changes Made

1. **Requirements-run storage contract** (Required Change 1)
   - Updated `FsRequirementsStore::create_run_dir()` in `src/adapters/fs.rs` to create empty `answers.toml` and `answers.json` files for every requirements run at directory creation time. Quick-mode runs that skip question generation and draft-mode runs with empty questions now satisfy the spec's canonical file layout requirement. The `answers.toml` is initialized with a comment header and `answers.json` with an empty answers list. Both use `if !exists` guards so they don't overwrite real data when `create_run_dir` is invoked for runs that later populate these files through the normal question/answer flow.

2. **Canonical question-boundary failure state** (Required Change 2)
   - **Payload/artifact persistence failure** (service.rs ~line 182): now appends a `RunFailed` journal event after setting `run.json` to failed, so the terminal transition is durably journaled.
   - **Answers.toml write failure** (service.rs ~line 227): now appends a `RunFailed` journal event after setting `run.json` to failed.
   - **`pending_question_count` recording**: moved `let question_count = qs.questions.len() as u32; run.pending_question_count = Some(question_count);` to immediately after the question-set boundary commit and before branching into empty-question, awaiting-answers, or template-write failure paths. This ensures `pending_question_count` is always recorded in canonical state even for failed runs at the question boundary.
   - **`show()` updated**: `pending_question_count` is now exposed for both `AwaitingAnswers` and `Failed` runs (was previously only `AwaitingAnswers`), so `requirements show` can report the pending question count for failed runs at the committed question boundary.

3. **Staged artifact cleanup on rename failure** (Recommended Improvement 2)
   - Both `write_payload_artifact_pair_atomic()` implementations (project store and requirements store) in `src/adapters/fs.rs` now remove the orphaned `artifact_staging` file when the final artifact rename fails, preventing leaked temporary files in `runtime/temp/`.
