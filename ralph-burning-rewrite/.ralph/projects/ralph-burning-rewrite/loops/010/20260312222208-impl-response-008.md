# Implementation Response (Iteration 8)

## Changes Made

1. **Requirements journal commit handling — fail_run() best-effort journal** (Required Change 1, prerequisite)
   - Changed `fail_run()` in `service.rs` to use `let _` for its own journal append instead of `?`, matching the workflow engine's `fail_run()` at `engine.rs:1755`. This ensures that when `fail_run()` is called because a journal append failed, the recovery itself does not fail on the same broken journal.

2. **Requirements journal commit handling — boundary recovery at RunCreated** (Required Change 1)
   - Both `draft()` and `quick()` now wrap the `RunCreated` journal append in `if let Err(e)` and call `fail_run()` to transition the run to Failed state before returning the error.

3. **Requirements journal commit handling — boundary recovery at QuestionsGenerated** (Required Change 1)
   - If the `QuestionsGenerated` journal append fails after `write_run()` has advanced the question-set boundary, the service now: (a) rolls back the question payload/artifact pair via `remove_payload_artifact_pair`, (b) clears `latest_question_set_id` and `question_round` so `run.json` does not reference deleted history, (c) calls `fail_run()`.

4. **Requirements journal commit handling — boundary recovery at AnswersSubmitted** (Required Change 1)
   - If the `AnswersSubmitted` journal append fails after `write_run()` has advanced past the question boundary, the service calls `fail_run()` to transition to Failed.

5. **Requirements journal commit handling — boundary recovery at DraftGenerated** (Required Change 1)
   - If the `DraftGenerated` journal append fails, the service rolls back the draft payload/artifact pair, clears `latest_draft_id` and `recommended_flow`, and calls `fail_run()`.

6. **Requirements journal commit handling — boundary recovery at ReviewCompleted** (Required Change 1)
   - If the `ReviewCompleted` journal append fails, the service rolls back the review payload/artifact pair, clears `latest_review_id`, and calls `fail_run()`. Draft history survives.

7. **Requirements journal commit handling — boundary recovery at SeedGenerated** (Required Change 1)
   - If the `SeedGenerated` journal append fails, the service rolls back the seed payload/artifact pair, clears `latest_seed_id`, calls `fail_run()` (terminal-transition ordering: Failed state persisted before cleanup), then removes seed files. Draft and review history survive.

8. **Requirements journal commit handling — RunCompleted best-effort** (Required Change 1)
   - The `RunCompleted` journal append is now best-effort (`let _`). When all durable state (run.json, seed files, seed history) is already committed as Completed, a missing RunCompleted journal event is non-ideal but does not affect state consistency.

9. **Fault-injection tests** (Recommended Improvement 1)
   - `run_created_journal_failure_persists_failed_state_in_draft` — verifies draft run transitions to Failed on RunCreated journal failure.
   - `run_created_journal_failure_persists_failed_state_in_quick` — same for quick mode.
   - `questions_generated_journal_failure_rolls_back_and_fails_run` — verifies payload/artifact rollback and boundary marker clearing.
   - `draft_generated_journal_failure_rolls_back_and_fails_run` — verifies draft payload rollback.
   - `review_completed_journal_failure_rolls_back_and_fails_run` — verifies review payload rollback with draft surviving.
   - `seed_generated_journal_failure_rolls_back_and_fails_run` — verifies seed payload/artifact rollback, seed file removal, draft+review survival.
   - `run_completed_journal_failure_preserves_completed_state` — verifies run remains Completed (best-effort), journal has SeedGenerated but not RunCompleted.

10. **Gherkin scenarios** — RD-028 through RD-033.
