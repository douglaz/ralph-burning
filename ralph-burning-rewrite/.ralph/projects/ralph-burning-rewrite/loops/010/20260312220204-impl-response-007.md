# Implementation Response (Iteration 7)

## Changes Made

1. **Answers template round-trip safety** (Required Change 1)
   - **Domain validation**: Added TOML bare-key constraint to question ID validation in `contracts.rs`. Question IDs must now consist only of ASCII alphanumeric characters, underscores, and hyphens (`[A-Za-z0-9_-]`). IDs like `team name` (space) or `api.version` (dot) are rejected at the domain validation boundary with a clear error mentioning "TOML bare keys".
   - **Template escaping**: Rewrote `generate_answers_template()` in `service.rs` to properly handle special characters:
     - Multi-line prompts and rationale are split across comment lines, each prefixed with `#`.
     - Default values are escaped for TOML basic strings via a new `toml_escape_basic_string()` helper that handles `\`, `"`, newlines, carriage returns, tabs, backspace, and form feed.

2. **Regression tests** (Recommended Improvement 1)
   - `question_set_with_space_in_id_fails_domain_validation` — verifies IDs with spaces are rejected.
   - `question_set_with_dot_in_id_fails_domain_validation` — verifies IDs with dots are rejected.
   - `question_set_with_valid_bare_key_ids_passes_validation` — verifies alphanumeric/underscore/hyphen IDs pass.
   - `draft_with_special_chars_in_prompts_and_defaults_produces_valid_toml` — end-to-end service test that generates a question set with quotes, newlines, and backslashes in prompts/defaults, then asserts the generated `answers.toml` parses as valid TOML and the default values round-trip correctly.

3. **Gherkin scenarios** — RD-026 (TOML bare-key ID validation) and RD-027 (template round-trip with special characters).

All **296 unit tests** and **75 CLI tests** pass. `cargo check` clean.
