# Implementation Notes

## Decisions Made
- **Three payload families cover all 16 stages.** Rather than 16 individual payload types, stages are grouped into Planning (4 stages), Execution (5 stages), and Validation (7 stages) families. Each family shares a typed payload struct. This matches the spec's grouping of required fields by family and avoids type explosion while still enforcing all required fields per category.
- **Schema validation via serde deserialization.** JSON Schema generation uses `schemars` (0.8) for external consumers and documentation. Actual schema validation is implemented via `serde_json::from_value` deserialization, which enforces the same structural requirements as the generated schema. This avoids an additional `jsonschema` validator dependency while satisfying the "JSON Schema generation/validation" requirement.
- **Contract evaluation returns `Result<EvalSuccess, ContractError>`.** `EvalSuccess` carries both the `ValidatedBundle` (payload + artifact) and an optional `outcome_failure` field. This models the distinction between hard failures (schema/domain/render — no bundle returned) and workflow-level outcome failures (QA/review non-passing — bundle IS returned, but with `QaReviewOutcomeFailure` flag).
- **`ReviewOutcome::is_passing()` treats only `Approved` as passing.** `ConditionallyApproved`, `RequestChanges`, and `Rejected` all map to `QaReviewOutcomeFailure` per the spec's "not cleanly approved" language.
- **Semantic validation rules are intentionally minimal.** Planning requires non-empty problem_framing, at least one work item with positive order. Execution requires non-empty change_summary, at least one step with positive order. Validation requires follow_up_or_amendments when outcome is non-passing. These enforce the spec's required-field contracts without over-constraining domain logic that belongs to later slices.
- **`ContractError` placed in `shared/error.rs`** alongside `AppError`, with a `failure_class()` method mapping each variant to its `FailureClass`. Render errors map to `DomainValidationFailure` since they indicate an internal contract consistency issue rather than a distinct failure class.
- **`StageId::ALL` and `StageId::display_name()` added** to support registry completeness verification and human-readable Markdown headings respectively. No existing enum variants or flow orderings were modified.

## Spec Deviations
- **JSON Schema validation is via serde, not a dedicated JSON Schema validator.** The spec says "JSON Schema generation/validation." We generate schemas via `schemars` and validate structurally via serde deserialization. A dedicated `jsonschema` crate was considered but adds complexity for equivalent validation coverage. The generated schemas are available via `StageContract::json_schema()` for any consumer that needs standalone schema validation.
- **No separate `RenderFailure` in `FailureClass` enum.** The spec's failure-class enum has six members and doesn't include a render-specific class. Render errors (which should not occur if schema+semantic pass) are mapped to `DomainValidationFailure` in `ContractError::failure_class()`.

## Testing
- **48 unit tests pass** (up from 12 in loop 1), covering:
  - Registry completeness: every `StageId` and every stage in every built-in flow resolves to a contract
  - Family mapping correctness: planning/execution/validation stages map to their expected families
  - Schema failure short-circuits: invalid JSON produces `SchemaValidation` error, never reaches semantic validation or rendering
  - Domain failure short-circuits: schema-valid but semantically invalid payloads produce `DomainValidation` error, never reach rendering
  - Successful evaluation: each family produces a `ValidatedBundle` with non-empty artifact
  - QA/review outcome failure: non-passing outcomes return bundle WITH `QaReviewOutcomeFailure`, not schema/domain errors
  - JSON Schema generation: every stage produces a non-trivial schema
  - Deterministic rendering: same payload rendered twice produces byte-identical output
  - Artifact structure: section headings, content markers, display names all verified
  - Conformance scenario IDs (SC-EVAL-001 through SC-EVAL-007): explicit tests mapping to Gherkin scenarios
- **`cargo clippy --all-targets --all-features -- -D warnings` passes clean**
- **9 Gherkin conformance scenario IDs** added in `tests/conformance/features/stage_contracts.feature` (SC-EVAL-001 through SC-EVAL-009)
