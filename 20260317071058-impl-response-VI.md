# Implementation Response (Iteration 6)

## Changes Made
1. **RC1 — GitHub-only production intake**: `daemon start` without `--data-dir` now always returns an error in normal CLI usage. The legacy single-repo path (with `FileIssueWatcher`) is gated behind `RALPH_BURNING_TEST_LEGACY_DAEMON=1` and `--single-iteration` — it is unreachable from any normal CLI invocation. The env-var gate ensures file-based intake remains available as test infrastructure without being part of the shipped production CLI surface. Integration tests and conformance scenarios set this env var automatically.

2. **RC2 — Explicit `/rb retry` transition handling**: Both `Failed` and `Aborted` tasks are now retryable end to end. `TaskStatus::can_transition_to` now permits `Aborted -> Pending` (model.rs), and `DaemonTaskService::retry_task` accepts both `Failed` and `Aborted` statuses (task_service.rs). The GitHub intake `handle_explicit_command` already handled both statuses, so the inconsistency between intake and task service is resolved. A `/rb retry` on an aborted issue now successfully transitions to Pending and reconciles the label to `rb:ready`, instead of quarantining the repo.

3. **RI1 — Conformance coverage**: Added `daemon.tasks.retry_aborted_issue` scenario that creates an `Aborted` task, retries it via `DaemonTaskService::retry_task`, and verifies the transition to `Pending` with incremented attempt count. Added `daemon.tasks.start_requires_data_dir` scenario that verifies the production CLI rejects `daemon start` without `--data-dir`. Both scenarios and their feature file entries are registered.
