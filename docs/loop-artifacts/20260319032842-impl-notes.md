# Implementation Notes

## Decisions Made
- The diagnostics service (`BackendDiagnosticsService`) is constructed directly from `&EffectiveConfig` rather than going through the composition layer, since it is purely read-only and does not need backend adapters, session stores, or output stores. This avoids unnecessary coupling.
- `backend check` validates config-level enablement and panel resolution only. It does not perform live transport-level availability checks (e.g., pinging OpenRouter API or checking process executable permission) because those would introduce side effects that violate the strictly read-only requirement.
- Panel probe supports three synthetic targets (`completion_panel`, `final_review_panel`, `prompt_review_panel`) in addition to all 10 singular `BackendPolicyRole` values. These are parsed as string matches rather than extending the role enum, keeping the domain model clean.
- CLI override flags (`--backend`, `--planner-backend`, etc.) are accepted on `check`, `show-effective`, and `probe` subcommands, reusing the same `CliBackendOverrides` struct and `BackendSelection::from_backend_name` parsing as `run start`/`run resume`.
- The `BackendPolicyService::backend_enabled_public()` method was added as a thin public wrapper around the existing private `backend_enabled()` method rather than making the private method public, preserving the existing API surface.
- `backend list` reports `compile_only: true` for the `stub` family to distinguish compile-time-only backends from production backends.

## Spec Deviations
- `workspace_governance/config.rs` did not need new helpers — the existing `ConfigEntry`/`ConfigValueSource` system with `EffectiveConfig::get()` was sufficient for source-precedence reporting. The spec listed this file for modification but the existing infrastructure was adequate.
- `composition/agent_execution_builder.rs` was not modified — the diagnostics service is constructed directly from `EffectiveConfig` at the CLI handler level, which is simpler and avoids adding builder complexity for a read-only service.
- `backend check` does not distinguish between flow presets for readiness evaluation. Since backend enablement is flow-independent (panel configuration is global, not per-flow), the flow parameter is accepted but does not affect the check logic. This matches the actual resolution semantics.

## Testing
- **14 unit tests** in `backend_diagnostics_test.rs` covering: list (2), check (3), show-effective (3), probe (5), aggregation (1)
- **3 new unit tests** in `backend_policy_test.rs` covering: prompt review panel resolution, `backend_enabled_public` API, optional panel member omission semantics
- **12 CLI integration tests** in `cli.rs` covering: text and JSON output for all 4 subcommands, non-zero exit on check failure, CLI override simulation
- **5 conformance scenarios** in `backend_operations.feature` with matching executors: list, check, show-effective, completion panel probe, final review panel probe
- **Registry consistency**: `conformance_spec_test::registry_matches_discovered_scenarios` passes, confirming all 5 new scenarios are registered
- All tests pass with `cargo test --features test-stub`
