# Implementation Response (Iteration 8)

## Changes Made
1. **Shared batch journal atomicity (Required Change 1)**: `stage_amendment_batch` now tracks successful journal appends via an `appended_count` counter. If a later append fails after earlier lines are already on disk, the function recognizes that the journal has orphaned `amendment_queued` entries that cannot be un-appended. Instead of returning a plain journal I/O error (which implies clean rollback), it returns `AppError::CorruptRecord` describing the partial-journal state (e.g. "batch journal append failed after 1 of 2 events: ..."). The snapshot and amendment files are still rolled back, and any rollback failures are included in the composite error message.

2. **Rollback failure handling (Required Change 2)**: Both `add_manual_amendment` and `stage_amendment_batch` no longer swallow rollback failures with `let _ =` after journal append errors. Each rollback operation (snapshot restore, file cleanup) is now checked. If either fails, a `CorruptRecord` error is returned that includes the original journal error, the snapshot restore result, and the file cleanup result — matching the composite-error pattern already used in `execute_rollback` at service.rs:658-668. When rollback fully succeeds and no partial journal state exists, the plain journal error is still returned as before.

3. **Failure-injection test coverage (Recommended Improvement)**: Added 2 new test fixtures and 4 new unit tests:
   - `FailAfterNAppendsJournalStore` — journal read succeeds, first N appends succeed, then fails. Enables testing mid-batch journal failure with partial appends.
   - `FailingRollbackSnapshotStore` — first write succeeds (canonical commit), subsequent writes fail (rollback attempt). Enables testing rollback failure after journal error.
   - `stage_amendment_batch_surfaces_partial_journal_as_corrupt_record` — verifies `CorruptRecord` with "batch journal append failed after 1 of 2 events" when the second journal append fails in a 2-amendment batch.
   - `add_manual_amendment_returns_corrupt_record_when_rollback_fails` — verifies `CorruptRecord` when snapshot restore fails after journal error.
   - `stage_amendment_batch_returns_corrupt_record_when_rollback_fails` — verifies `CorruptRecord` when snapshot restore fails during batch staging.
   - `add_manual_amendment_returns_corrupt_record_when_file_rollback_fails` — verifies `CorruptRecord` when file cleanup fails (via `FailingRemoveAmendmentQueue`) after journal error.

4. **Docs updated**: `amendments.md` failure safety section updated to describe partial-journal and rollback-failure error reporting for both `add` and `stage_amendment_batch`. `slice-3.md` updated with iteration 8 changes.
