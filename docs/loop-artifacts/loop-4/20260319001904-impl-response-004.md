# Implementation Response (Iteration 4)

## Changes Made
1. **Amendment commit transactionality (Required Change 1)**: Reordered `add_manual_amendment` and `stage_amendment_batch` so the canonical snapshot is committed BEFORE journal events are written. This eliminates the failure path where a snapshot write error leaves orphaned, visible journal entries without matching canonical state. Journal writes are now best-effort after canonical commit — if a journal append fails, the amendment is still visible in `run.json` and functions correctly. For `add_manual_amendment`: file → snapshot → journal (best-effort). For `stage_amendment_batch`: files → snapshot → journal events (best-effort). Both paths roll back amendment files if the snapshot write fails.

2. **Remove failure on disk deletion (Required Change 2, sub-item 1)**: `remove_amendment` now deletes the durable amendment file first, then updates the canonical snapshot. If file deletion fails, no mutation is visible — the snapshot is never touched. If the snapshot write fails after a successful file deletion, the amendment file is restored via `write_amendment` so disk and snapshot stay consistent. This ensures completion blocking (which checks both snapshot and disk at `engine.rs:5134`/`5144`) works correctly regardless of which operation fails.

3. **Clear partial-failure invariants (Required Change 2, sub-item 2)**: `clear_amendments` now deletes files first, tracking removed vs remaining, then updates the snapshot. Three failure paths are handled:
   - All files deleted, snapshot write fails → all files are restored, error propagated.
   - Partial files deleted, snapshot repair write fails → `AmendmentClearPartial` is still returned with exact removed/remaining IDs (the repair write is best-effort).
   - Partial files deleted, snapshot repair write succeeds → `AmendmentClearPartial` returned normally.
   This ensures the caller always gets the exact removed and remaining IDs, satisfying the spec's partial-failure contract.

4. **Partial-clear conformance proof (Required Change 3)**: `parity_slice3_clear_partial_failure` now requires BOTH the removed ID AND the remaining ID to be present in stderr as a complete pairing. Changed from `if !has_removed_id && !has_remaining_id` (passes if just one side present) to `if !ordering_a && !ordering_b` where each ordering requires both `removed: {id}` AND `remaining: {id}` to be present. This deterministically proves the exact-ID contract.

5. **New unit test**: Added `remove_amendment_fails_when_file_deletion_fails` — uses `FailingRemoveAmendmentQueue` to verify that when file deletion fails, `remove_amendment` returns an error and the snapshot is never written.

6. **Docs updated (Recommended Improvement)**: `amendments.md` failure safety section rewritten to accurately describe the actual write ordering for each operation (file → snapshot → journal for add; file → snapshot for remove/clear) and the exact rollback behavior on failure.
