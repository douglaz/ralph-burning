# Implementation Response (Iteration 1)

## Changes Made
1. **Flow-scoped diagnostics** (Required Change 1): `check_backends` now accepts a `FlowPreset`, resolves the `FlowDefinition`, and scopes all checks to the flow's actual stages. Only policy roles present in the flow's stage list are checked. Completion panel is only checked if `StageId::CompletionPanel` is in the flow. Final review is only checked if `StageId::FinalReview` is in the flow and enabled. Prompt review is only checked if `StageId::PromptReview` is in the flow and enabled. `probe` now validates that the requested panel target exists in the given flow before resolving, returning a clear error like `"flow 'docs_change' does not include stage 'completion_panel'"` when it does not. New helper `roles_for_flow()` and `validate_flow_has_stage()` implement the scoping. Files: `diagnostics.rs`, `policy.rs`.

2. **`show-effective` contract** (Required Change 2): The `EffectiveBackendView` DTO now reports `default_session_policy` (renamed from the misleading singular `session_policy`), and each `RoleEffectiveView` includes a `session_policy` field reflecting the runtime's actual per-role behavior: `"new_session"` for panel roles (completer, final_reviewer, prompt_reviewer, prompt_validator, arbiter) and `"reuse_if_allowed"` for main stage roles (planner, implementer, reviewer, qa, acceptance_qa). This matches the `SessionPolicy::NewSession` / `SessionPolicy::ReuseIfAllowed` usage in `engine.rs`, `completion.rs`, `prompt_review.rs`, and `final_review.rs`. Source precedence for `PromptReviewer` and `Arbiter` roles now resolves via `prompt_review.refiner_backend` and `final_review.arbiter_backend` config keys respectively, instead of falling through to `"default"`. Files: `diagnostics.rs`, `backend.rs`.

3. **Panel probe and failure detail** (Required Change 3): `probe_final_review_panel` now serializes the arbiter as a dedicated `arbiter` field on `PanelProbeView`, exposing backend family, model ID, and required status in both text and JSON output. Panel check failures now report the exact backend family from the error (e.g. `"openrouter"` instead of `"mixed"`) and the precise config field path (e.g. `"completion.backends"` instead of `"completion_panel config"`). New `classify_panel_error_structured()` extracts structural detail from `BackendUnavailable` errors. Files: `diagnostics.rs`, `backend.rs`.

4. **Readiness validation** (Required Change 4): Added `check_backends_with_availability()` async method that first runs config resolution checks, then collects unique resolved backend targets and calls `adapter.check_availability()` for each, using the same `AgentExecutionPort` trait the runtime uses. Failures are reported with `BackendCheckFailureKind::AvailabilityFailure`. The CLI `backend check` handler now builds the production adapter and calls the availability-aware path. Falls back to config-only checking if the adapter cannot be built. Files: `diagnostics.rs`, `backend.rs`.
