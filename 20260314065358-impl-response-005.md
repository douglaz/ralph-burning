# Implementation Response (Iteration 5)

## Changes Made
1. **Post-rename verification/restore failures now surface explicit I/O errors** (`src/adapters/fs.rs`): Both `file.metadata()` (fstat) and `fs::metadata(&staging)` failure recovery paths now check whether `fs::rename(&staging, &path)` succeeds. Previously, the restore result was silently discarded with `let _ = fs::rename(...)`. Now, if the rename-back fails, the function returns an `AppError::Io` with a compound message identifying both the original verification failure and the restore failure. This satisfies the spec invariant that post-rename recovery must either restore canonical lock visibility or surface an explicit I/O error.

2. **Inode-mismatch restore failure returns I/O error instead of OwnerMismatch** (`src/adapters/fs.rs`): When `hard_link(&staging, &path)` fails with a non-`AlreadyExists` error (e.g., ENOSPC, EIO), the function now returns `Err(AppError::Io(...))` instead of `Ok(OwnerMismatch { actual_owner: "(replaced)" })`. This prevents a stranded staging artifact from masquerading as a normal ownership conflict, ensuring callers (CLI guard drop at `cli_writer_lease.rs:161` and reconcile at `lease_service.rs:480`) correctly treat it as a cleanup failure that preserves the CLI lease for later recovery.

3. **Conformance harness binary resolution hardened** (`src/contexts/conformance_spec/scenarios.rs`): Added `RALPH_BURNING_CLI_PATH` env var override to `binary_path()`. When set, the conformance runner uses the specified path instead of `std::env::current_exe()`. This allows the integration test to pin sub-spawns to a stable binary path.

4. **Conformance full-suite test isolation** (`tests/cli.rs`): `conformance_full_suite_passes` now hard-links the CLI binary to a stable temp path before spawning the conformance suite, passing `RALPH_BURNING_CLI_PATH` to the child. The hard link pins the inode so nested sub-spawns remain stable even if cargo relinks the original binary during parallel `cargo test` execution. Falls back to `std::fs::copy` if hard_link fails (e.g., cross-device).

5. **New regression test: rename-back failure surfaces compound error** (`src/adapters/fs.rs` tests): Added `release_writer_lock_rename_back_failure_surfaces_io_error` which uses `post_rename_hook` to delete the staging file and write a replacement at the canonical path. When `fs::metadata(&staging)` fails and the rename-back also fails (ENOENT on deleted staging), the test asserts: (a) the result is `Err`, (b) the error message mentions "could not restore lock", and (c) the replacement lock at the canonical path survives untouched.
