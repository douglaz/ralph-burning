# Requirements Draft

## Problem Summary

After the bead selector transitions a bead to Claimed state (9ni.8.2), there is no automated bridge that claims the bead in br, generates the bead-backed prompt, creates the Ralph task/project, and transitions the controller to Running. Currently these are separate manual CLI steps (br update, project create-from-bead, workflow start). This bead implements the atomic claim-and-create pipeline: on Claimed state entry, call br update --status=in_progress, generate the prompt via the canonical prompt contract, invoke create_project_from_bead_context with task_source metadata linking milestone/bead, update milestone state with the claim and linked task_id, and transition the controller from Claimed to Running. If br claim fails, the controller must transition to NeedsOperator without creating a task.

## Goals

- Implement an automated claim handler that fires when the controller enters Claimed state with an active_bead_id, executing the full claim-to-running pipeline
- Call br update <bead_id> --status=in_progress as the first mutation, before any Ralph task/project creation
- Generate the bead-backed prompt using the canonical prompt contract (bead_execution_prompt/1) with full milestone and bead context from br show
- Create a Ralph project/task from the generated prompt via create_project_from_bead_context, populating TaskSource with milestone_id, bead_id, plan_hash, and plan_version
- Record the claim and linked task_id in milestone state: update MilestoneControllerRecord.active_task_id, write journal event, and call record_bead_start for lineage tracking
- Transition the controller from Claimed to Running via sync_controller_task_running after successful task creation
- On br claim failure (bead already in_progress, tool error, timeout), transition to NeedsOperator with diagnostic reason and do NOT create the Ralph task
- On task creation failure after successful br claim, record partial state (bead claimed but no task) so recovery is possible, and transition to NeedsOperator

## Non-Goals

- Bead selection logic — covered by 9ni.8.2, which produces the Claimed state with active_bead_id
- Task execution and workflow orchestration — covered by 9ni.8.4, which runs the quick_dev engine on the created task
- Prompt contract design or generation logic — covered by 9ni.6.1/6.2, reused here via BeadProjectContext
- Success reconciliation and bead closing — covered by 9ni.8.5.1, which handles post-completion cleanup
- Modifying the br adapter or BrCommand interface — update_bead_status already exists in br_process.rs
- Parallel bead execution — this bead enforces the one-at-a-time invariant

## Constraints

- Must use the existing BrAdapter.update_bead_status(id, 'in_progress') method — no new br commands
- Must use the existing create_project_from_bead_context service function for task creation
- Must use the existing sync_controller_task_running function for the Claimed→Running transition
- Claiming and task creation must be as atomic as possible: br claim → prompt generation → task creation → milestone state update → controller transition, with no intermediate states visible to other consumers
- The claim handler must be callable from both the CLI run path and the automation_runtime daemon loop (success_reconciliation continuation)
- Must not break the existing resume logic in MilestoneControllerResumePort which handles Claimed state with task_id present vs absent
- br sync --flush-only must be called after br update to persist the status change durably
- All mutations must use structured tracing with milestone_id, bead_id, and task_id fields for observability

## Acceptance Criteria

- br update <bead_id> --status=in_progress is called explicitly BEFORE any Ralph task/project creation attempt
- A Ralph project/task is created from the bead-backed prompt with TaskSource containing correct milestone_id, bead_id, plan_hash, and plan_version
- The generated prompt passes validate_canonical_prompt_shape and contains the bead_execution_prompt/1 contract marker
- MilestoneControllerRecord.active_task_id is set to the created project's ID after successful task creation
- A MilestoneControllerTransitionEvent for Claimed→Running is written to controller-journal.ndjson with bead_id and task_id
- record_bead_start is called with the correct milestone_id, bead_id, project_id, and run_id for lineage tracking
- If br update fails, the controller transitions to NeedsOperator with a reason containing the br error, and no Ralph task is created
- If task creation fails after br claim succeeds, the controller transitions to NeedsOperator with a reason indicating partial claim state (bead claimed, task creation failed)
- The claimed bead no longer appears in br ready output after successful claim
- Unit tests cover: successful claim-to-running pipeline, br claim failure → NeedsOperator, task creation failure after claim → NeedsOperator with partial state
- Integration test proves the full pipeline from Claimed state entry through Running state with correct milestone journal entries

## Risks and Open Questions

- Race condition: if the process crashes between br claim and task creation, the bead is marked in_progress in br but no Ralph task exists. The NeedsOperator transition handles this, but should the resume handler also detect this orphan state and retry task creation?
- br update --status=in_progress is not idempotent if the bead is already in_progress from a prior partial claim. Should the claim handler treat 'already in_progress' as success (idempotent retry) or as an error requiring operator intervention?
- The existing complete_next_milestone_bead_selection in run.rs transitions to Claimed but does not set active_task_id (it's None). The claim handler needs a clear integration point — should it be called inline after selection returns Claimed, or should it be a separate function triggered by detecting Claimed state with no task_id?
- Prompt generation requires br show to fetch full bead detail plus milestone plan loading. If br show fails after br update succeeds, this is another partial state. Should br show be called before br update to front-load the failure point?
- The flow preset for the created project needs to come from somewhere — the milestone plan, the bead labels, or a default. Current handle_create_from_bead takes it from CLI args. What determines the flow for automated claiming?
